# Phase 1 — UI Shell and Structural Layout

## Goal
Build the static machine console shell first.

## Required Deliverables
- header bar
- telemetry band
- core machine view region
- bottom narrative/control band
- right advisory panel

## Constraints
- fixed machine-centered layout
- clean outer shell
- no particles outside machine
- exact labels only
- no blank metrics
- no pseudo-text
- no animation required yet except minimal UI polish if needed

## Output expectation
A clean, readable, high-trust interface scaffold.

---

# Phase 2 — Mocked Storyboard Playback

## Goal
Support deterministic switching across the 5 canonical states:

1. CLEAN START
2. EARLY CAPTURE
3. RISING LOAD
4. THRESHOLD STATE
5. BACKFLUSH + AFTERMATH

## Required Deliverables
- frame preset state objects
- frame selector controls
- state-driven visual differences
- frame title + system status updates
- stage fouling updates
- telemetry updates
- advisory updates

## Constraints
- same machine geometry across frames
- same camera/layout across frames
- only machine state changes
- keep detergent realism subtle and internal

## Output expectation
A working storyboard console that feels like one machine evolving, not multiple different screens.

---

# Phase 3 — Hydrion-to-UI Binding Adapter

## Goal
Replace the assumption of permanent mock-only state.

## Required Deliverables
- explicit raw truth-state type
- explicit display-state type
- mapping function:
  - raw Hydrion truth -> display state
- support for two state modes:
  - mock
  - sim

## Constraints
- no hidden derivation logic
- document provisional derived values clearly
- preserve all labels and layout
- do not redesign the machine view

## Output expectation
A clean adapter layer that allows mocked states now and real Hydrion state later.

---

# Phase 4 — Playback Controller

## Goal
Turn the storyboard/live view into an actual simulation console.

## Required Deliverables
- mode support:
  - storyboard
  - live
- play / pause
- previous / next
- frame selection
- mode label
- state/frame label
- minimal playback strip

## Recommended
- simple speed control
- clean timeline strip

## Constraints
- playback controls must feel instrument-like
- not a generic video player
- machine remains visually dominant
- playback is subordinate to machine truth

## Output expectation
A console that lets users navigate state and time intentionally.

---

# Phase 5 — Animation Layering

## Goal
Make the machine feel alive without breaking truth.

## Required Deliverables
Add grounded motion for:
- flow intensity / coherence
- pressure burden glow
- fouling density progression
- particle transport
- detergent haze / bubbles
- backflush pulse
- bypass path readiness / activation

## Constraints
- no decorative lightning
- no excessive sci-fi energy effects
- flow must feel like detergent-bearing water
- pressure must feel like hydraulic strain
- fouling must feel material
- backflush must be partial, not magical
- all motion must map to state

## Output expectation
A living machine interface with physically meaningful motion.

# Code Architecture Requirements

## Preferred structure
- `HydrosMachineViewV1.tsx`
- `components/`
  - telemetry components
  - machine components
  - advisory components
  - playback controls
- `state/`
  - mock frame presets
  - truth-state type
  - display-state type
  - mapping helpers
  - playback controller hook

## Important
Keep logic separated from presentation:
- presentation components should not contain hidden machine logic
- state adaptation should be explicit
- playback controller should be isolated
