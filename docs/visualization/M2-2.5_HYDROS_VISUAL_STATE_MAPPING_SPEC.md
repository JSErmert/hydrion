# HydrOS Visual State Mapping Specification

## Purpose

Define how HydrOS / Hydrion simulation state maps into visual behavior.

This document is the bridge between:

- Hydrion physics state
- HydrOS Machine View
- HydrOS Animation Spec
- React implementation

It answers:

> “When the machine state changes, exactly what should the user see?”

This is not a design mood board.

This is:
> the authoritative state-to-visual contract for HydrOS M2

---

# Core Principle

Every major visual change in HydrOS must be traceable to:

- a real state variable
- or a governed derived variable

No major visual behavior should exist without a state basis.

---

# State Layers

HydrOS Machine View uses three classes of state:

## 1. Truth State
Direct outputs from Hydrion.

Examples:
- `q_processed_lmin`
- `q_in_lmin`
- `q_bypass_lmin`
- `dp_total_pa`
- `dp_stage1_pa`
- `dp_stage2_pa`
- `dp_stage3_pa`
- `fouling_frac_s1`
- `fouling_frac_s2`
- `fouling_frac_s3`
- `recoverable_s1`
- `recoverable_s2`
- `recoverable_s3`
- `irreversible_s1`
- `irreversible_s2`
- `irreversible_s3`
- `bf_active`
- `bf_pulse_idx`
- `bf_cooldown_remaining`
- `bypass_active`
- `maintenance_required`

## 2. Sensor State
Measured / inferred values shown as user-facing telemetry.

Examples:
- filtered pressure reading
- filtered flow reading
- optical load proxy
- sensor lag / noise variants

## 3. Derived Visual State
Computed from truth or sensor state for display logic.

Examples:
- pressure burden class
- fiber load class
- system status string
- next action
- advisory text
- flow visual intensity
- fouling texture density
- bypass readiness

---

# Mapping Philosophy

Visuals must map according to these rules:

## Rule 1
Truth drives machine behavior.

## Rule 2
Sensor state drives telemetry presentation when applicable.

## Rule 3
Derived state is allowed only if:
- formula is explicit
- meaning is stable
- it does not hide underlying truth

## Rule 4
Visual severity must not exceed model severity.

Example:
A placeholder electrostatic model must not appear more visually “important” than a grounded hydraulic stress state.

# 1. Flow Path Mapping

## Primary Driver
- `q_processed_lmin`

## Supporting Drivers
- `q_in_lmin`
- `q_bypass_lmin`
- `bf_active`
- detergent / particle proxy density

## Visual Outputs
- flow thickness
- flow speed
- flow brightness
- turbulence intensity
- mainline continuity

## Mapping

### Low flow regime
Condition:
- `q_processed_lmin <= 5`

Visual:
- thin stream
- slower motion
- reduced brightness
- more visible internal chamber detail

### Nominal flow regime
Condition:
- `5 < q_processed_lmin < 15`

Visual:
- balanced stream width
- stable coherent motion
- medium brightness
- minimal turbulence

### Peak flow regime
Condition:
- `q_processed_lmin >= 15`

Visual:
- thicker stream
- higher velocity
- stronger brightness
- more visible turbulence / agitation near restrictions

## Bypass interaction
If `q_bypass_lmin > 0`:
- reduce main processed stream width proportionally
- branch flow path becomes visible
- processed stream remains primary unless bypass dominates

## Backflush interaction
If `bf_active == 1`:
- local reverse pulse overlays affected region
- forward flow may reduce visually if model indicates it
- reverse motion remains localized, not full-scene inversion

# 2. Stage Fouling Mapping

## Primary Drivers
- `fouling_frac_s1`
- `fouling_frac_s2`
- `fouling_frac_s3`

## Supporting Drivers
- `recoverable_s1/s2/s3`
- `irreversible_s1/s2/s3`

## Visual Outputs
- texture density
- opacity / clog coverage
- material type appearance
- stage burden emphasis

---

## Stage 1 Visual Mapping

### Meaning
Coarse stage:
- catches large fibers
- early entanglement
- less pressure-sensitive per unit load

### Visual language
- chunky fibers
- loose clusters
- sparse obstruction strands

### Mapping
- `0.00–0.15` → very light entanglement
- `0.15–0.35` → visible snag clusters
- `0.35–0.60` → moderate cluttered fiber capture
- `0.60–1.00` → dense but still porous blockage mass

---

## Stage 2 Visual Mapping

### Meaning
Intermediate stage:
- mixed capture behavior
- denser distributed material
- transitional resistance region

### Visual language
- mixed fiber + fine matter
- more even distribution
- denser texture than Stage 1

### Mapping
- `0.00–0.15` → faint distributed capture
- `0.15–0.35` → visible distributed haze
- `0.35–0.60` → moderate packed texture
- `0.60–1.00` → heavy distributed fouling layer

---

## Stage 3 Visual Mapping

### Meaning
Fine pleated stage:
- dominant capture zone
- strongest pressure sensitivity
- cake-like surface layer

### Visual language
- dense surface film
- granular wet cake
- strong opacity rise
- bottleneck appearance

### Mapping
- `0.00–0.10` → barely visible haze
- `0.10–0.30` → thin coating
- `0.30–0.60` → clearly visible cake growth
- `0.60–0.85` → dense surface burden
- `0.85–1.00` → saturated bottleneck state

# 3. Pressure Overlay Mapping

## Primary Drivers
- `dp_total_pa`
- `dp_stage1_pa`
- `dp_stage2_pa`
- `dp_stage3_pa`

## Derived Variables
- `pressure_kpa = dp_total_pa / 1000`
- stage burden ranking

## Visual Outputs
- glow intensity
- local hotspot location
- contour density
- structural tension feel

---

## Total Pressure Classes

### Low pressure
Condition:
- `< 20 kPa`

Visual:
- cool blue / neutral tone
- faint internal glow
- minimal strain

### Moderate pressure
Condition:
- `20–40 kPa`

Visual:
- slight amber warming
- visible stage hotspot
- stronger casing emphasis

### High pressure
Condition:
- `40–60 kPa`

Visual:
- concentrated orange burden
- strongest hotspot near dominant stage
- visible tension / compression feel

### Severe pressure
Condition:
- `> 60 kPa`

Visual:
- red-orange stress concentration
- strong bottleneck emphasis
- caution posture in UI

---

## Stage hotspot mapping

Determine dominant pressure stage:
- `max(dp_stage1_pa, dp_stage2_pa, dp_stage3_pa)`

Visual:
- dominant stage gets strongest localized glow
- supporting stages receive weaker gradients
- hotspot should never be global if stage-level data exists

# 4. Particle Transport Mapping

## Primary Drivers
- particle density proxy
- stage capture logic
- `q_processed_lmin`
- fouling state
- detergent haze / bubble proxy

## Visual Outputs
- particle count shown
- particle speed
- particle capture density
- outlet cleanliness

## M2 Particle Classes

### Fibers
Visual:
- thin thread-like particles
- dominant class
- higher stage interaction

### Fragments
Visual:
- compact dots / flecks
- less numerous
- lower clogging signature

## Mapping by state

### Low particle load
Visual:
- sparse particles
- easy-to-follow motion
- little accumulation

### Moderate particle load
Visual:
- steady transport
- visible capture in Stage 1 / 2
- growing stage density

### High particle load
Visual:
- dense incoming transport
- strong stage interaction
- visibly reduced outlet particle count once stages load

---

## Capture logic visual rules

Particles may only:
1. continue downstream
2. stop and become part of stage burden
3. reverse or detach during backflush if applicable

Particles must never disappear without one of those outcomes being implied.

## Primary Drivers
- derived detergent haze parameter
- bubble density proxy
- particle density
- flow regime

## Visual Outputs
- fluid haze
- microbubble density
- slight opacity increase
- turbulence richness

## Mapping

### Low detergent presence
Visual:
- mostly clear blue fluid
- faint haze
- minimal bubbles

### Moderate detergent presence
Visual:
- slightly milky / translucent fluid
- visible but controlled bubbles
- richer flow texture

### High detergent presence
Visual:
- stronger haze
- more microbubble presence

# 6. Backflush Mapping

## Primary Drivers
- `bf_active`
- `bf_pulse_idx`
- `bf_cooldown_remaining`
- stage recoverable fouling fractions

## Visual Outputs
- reverse flow pulse
- pulse intensity
- clearing behavior
- cooldown posture

## Mapping

### Not active
Condition:
- `bf_active == 0`

Visual:
- backflush path dormant
- no reverse motion
- possible standby highlight only if recommendation is active

### Active pulse
Condition:
- `bf_active == 1`

Visual:
- strong reverse motion in path
- bright pulse wavefront
- local stage clearing begins
- pressure burden reduces partially

### Cooldown
Condition:
- `bf_active == 0 && bf_cooldown_remaining > 0`

Visual:
- no active pulse
- backflush path subdued
- machine in post-event calm state
- pulse meter shows cooldown / completion state

## Important rule
Backflush must reduce fouling visually, but never fully reset unless the model truly resets.

# 7. Bypass Mapping

## Primary Drivers
- `bypass_active`
- `q_bypass_lmin`
- pressure threshold proximity

## Derived Variable
- `bypass_ready`

Suggested `bypass_ready` logic:
- true if pressure is approaching threshold and `bypass_active == 0`

## Visual Outputs
- bypass label state
- branch path illumination
- flow split visibility

## Mapping

### OFF
Condition:
- `bypass_active == 0 && bypass_ready == false`

Visual:
- branch path dark
- no split flow

### READY
Condition:
- `bypass_active == 0 && bypass_ready == true`

Visual:
- subtle branch glow
- advisory readiness
- no active split yet

### ACTIVE
Condition:
- `bypass_active == 1`

Visual:
- branch stream visible
- mainline flow reduced proportionally
- strong but controlled protection posture

# 8. Telemetry Text Mapping

Telemetry must use exact formatting.

## FLOW RATE
Source:
- `q_processed_lmin`

Format:
- `13.5 L/min`

## PRESSURE
Source:
- `dp_total_pa / 1000`

Format:
- `45.2 kPa`

## BYPASS
Source:
- `bypass_active`
- optional `bypass_ready`

Format:
- `OFF`
- `READY`
- `ACTIVE`

## STAGE FOULING
Source:
- `fouling_frac_s1/s2/s3`

Format:
- integer percentage
- example: `42% FOULING`

## EFFICIENCY
Source:
- derived v1 metric

Format:
- `82%`

## PULSE
Sources:
- `bf_active`
- `bf_pulse_idx`
- cooldown logic

Format examples:
- `0%`
- `READY`
- `PULSE 01`
- `PULSE 01 · 42%`
- `COOLDOWN`

## Rule
No telemetry field may be blank.
No telemetry field may show decorative placeholder dashes.

# 9. System Status Mapping

## Purpose
Convert multi-variable machine state into one dominant status string.

## Recommended precedence order

### 1. BACKFLUSH ACTIVE
Condition:
- `bf_active == 1`

### 2. BYPASS ACTIVE
Condition:
- `bypass_active == 1`

### 3. MAINTENANCE REQUIRED
Condition:
- `maintenance_required == 1`

### 4. RISING LOAD
Condition:
- moderate/high pressure or strong fouling buildup without maintenance trigger

### 5. STABILIZED / MONITORING
Condition:
- recent recovery state with residual burden

### 6. NORMAL OPERATION
Condition:
- no higher severity conditions active

## Rule
Only one major system status string appears at once.

# 10. Fiber Load Mapping

## Purpose
Provide a compact qualitative burden summary.

## Inputs
Can derive from:
- particle density proxy
- fouling progression
- stage capture burden

## Output Classes
- `LOW`
- `MODERATE`
- `HIGH`
- `CRITICAL`

## Suggested logic
Low:
- clean / sparse particle state

Moderate:
- visible capture building

High:
- major stage loading in progress

Critical:
- maintenance threshold or equivalent high burden state

# 11. Advisory Mapping

## Purpose
Turn state into concise action language.

## Outputs
Examples:
- `CONTINUE RUN`
- `MONITOR STAGE 01`
- `MONITOR STAGE 03`
- `PREPARE MAINTENANCE`
- `INITIATE BACKFLUSH`
- `BYPASS READY`
- `MONITOR NEXT CYCLE`

## Rules
- max 2–3 short actions
- imperative phrasing
- no speculation
- must follow current machine state

# 12. Calibration Trust Mapping

## Purpose
Expose realism honesty.

## Input Class
Each subsystem or feature must be tagged:
- `GROUNDED`
- `PROXY`
- `PLACEHOLDER`

## Visual Outputs
- stable confidence chips
- subtle semantic coloring
- low-priority motion only

## Rule
Calibration trust is visible but never competes with machine truth.

# 13. Playback Mode Mapping

- real-time values
- subtle transitions
- no frame jumps

- deterministic state presets

### Replay Mode
