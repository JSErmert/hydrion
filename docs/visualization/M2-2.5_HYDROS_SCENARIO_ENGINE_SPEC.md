# HydrOS Scenario Engine Specification

## Purpose

Define how HydrOS creates, controls, and executes **simulation scenarios**.

This system governs:

- input conditions
- disturbance patterns
- test environments
- repeatable experiments
- calibration conditions

It is the foundation for:

- RL training
- validation testing
- research experimentation
- eventual City of HydrOS integration

---

# Core Principle

A scenario is:

> a structured, reproducible, parameterized environment definition

NOT:
- a random simulation run
- a UI preset
- a one-off configuration

---

# Scenario Engine Responsibilities

The scenario engine must:

1. define system inputs over time
2. define disturbances
3. define operating regimes
4. control initial conditions
5. support reproducibility via seeds
6. generate meaningful test conditions
7. integrate cleanly with Hydrion

---

# Scenario Structure

Each scenario must define:

```ts
type HydrosScenario = {
  id: string;
  name: string;
  description: string;

  initialState: InitialStateConfig;

  flowProfile: FlowProfile;
  particleProfile: ParticleProfile;

  disturbances: DisturbanceEvent[];

  durationSec: number;
  dt: number;

  seed?: number;
};
1. Initial State Configuration

Defines system start condition.

type InitialStateConfig = {
  fouling_s1: number;
  fouling_s2: number;
  fouling_s3: number;

  storage_fill: number;

  pressure_bias?: number;
  flow_bias?: number;
};
Use Cases
clean start
partially fouled system
near-failure system
post-maintenance system
2. Flow Profile

Defines how water enters the system.

type FlowProfile = {
  type: "constant" | "ramp" | "burst" | "realistic";

  baseFlowLmin: number;

  variability?: number;
  burstAmplitude?: number;
  burstFrequency?: number;

  transientEvents?: Array<{
    time: number;
    duration: number;
    flow: number;
  }>;
};
Required Behavior
must support:
5 L/min (low)
12–15 L/min (nominal)
20 L/min (peak)
must support bursty patterns (washing machine reality)
3. Particle Profile

Defines what enters the system.

type ParticleProfile = {
  type: "fiber_dominant" | "mixed" | "heavy_load";

  density: number; // normalized [0–1]

  sizeDistribution: number[]; // placeholder for PSD bins

  variability?: number;
};
Notes
fiber-dominant is default for laundry
must later support PSD realism (M3+)
4. Disturbance Events

Defines abnormal conditions.

type DisturbanceEvent = {
  type:
    | "flow_spike"
    | "particle_spike"
    | "sensor_noise"
    | "partial_blockage"
    | "foam_event";

  time: number;
  duration: number;

  intensity: number;
};
Supported Disturbances
Flow Spike
sudden jump to peak flow
Particle Spike
increased fiber density
Partial Blockage
sudden rise in fouling or resistance
Foam Event
affects sensor realism (future)
Sensor Noise
modifies sensor layer only
5. Scenario Categories (Required)

HydrOS must support canonical scenario classes:

A. Baseline Operation
nominal flow
low disturbance
clean or moderate fouling
B. High Load Scenario
high particle density
fast clogging
C. Stress Scenario
repeated spikes
high fouling
frequent backflush demand
D. Failure Scenario
near saturation
bypass triggering
minimal recovery
E. Calibration Scenario
controlled input
no randomness
deterministic output
6. Scenario Execution Model

Scenario engine must:

step Hydrion with:
current flow input
particle load
disturbances
update state each timestep
expose:
truth_state
sensor_state
event markers
7. Scenario → Playback Integration

The scenario engine must integrate with:

Machine View
state updates drive visuals
Playback Controller
timeline reflects scenario time
Event Replay
disturbance events become replay anchors
8. Scenario → RL Integration

Scenario engine must support:

deterministic seeds
repeatable runs
consistent initial conditions
consistent reward evaluation
9. Scenario Definition Storage

Scenarios should be defined as:

id: baseline_nominal

initialState:
  fouling_s1: 0.05
  fouling_s2: 0.02
  fouling_s3: 0.01

flowProfile:
  type: realistic
  baseFlowLmin: 13.5
  variability: 0.2

particleProfile:
  type: fiber_dominant
  density: 0.3

disturbances:
  - type: flow_spike
    time: 120
    duration: 10
    intensity: 1.5

durationSec: 300
dt: 0.1
seed: 42
10. Scenario Engine Architecture

Recommended separation:

Scenario Definition

YAML / JSON

Scenario Runtime

Applies time-based changes

Scenario Adapter

Feeds Hydrion

11. Minimal v1 Implementation

Must include:

static scenarios (no editor yet)
flow profile support
particle density support
disturbance injection
deterministic execution
12. Future Expansion

Later phases should add:

scenario editor UI
scenario comparison engine
multi-run batch system
statistical output layer
City of HydrOS integration
13. Failure Conditions

Scenario engine fails if:

runs are not reproducible
disturbances are unclear
flow is unrealistic
particle behavior is inconsistent
scenarios are not distinguishable
RL cannot learn meaningful behavior
Final Definition

HydrOS Scenario Engine is:

the structured generator of machine reality

It defines:

what the system experiences
how it is stressed
how it evolves

Without it, HydrOS is a simulation.

With it, HydrOS becomes:

a controlled experimental system.
