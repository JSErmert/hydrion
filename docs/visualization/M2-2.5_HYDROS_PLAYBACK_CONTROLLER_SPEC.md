# HydrOS Playback Controller Specification

## Purpose

Define how users control time, state progression, and mode switching inside HydrOS Machine View.

This specification governs the playback layer for:

- storyboard playback
- live simulation mode
- event replay
- future comparison mode

It is the bridge between:

- Hydrion state evolution
- Machine View rendering
- user interaction with simulation time

This is not a generic media player.

It is:
> a machine-state playback and inspection system

---

# Core Principle

HydrOS playback must help the user answer:

- What state is the machine in now?
- How did it get there?
- What changed between two moments?
- What event caused the visible transition?
- What should happen next?

Playback control must feel like:
- technical instrumentation
- event inspection
- simulation control

It must NOT feel like:
- entertainment media playback
- a casual timeline scrubber
- a video editor
- a game replay bar

---

# Supported Playback Modes

HydrOS M2–2.5 supports four conceptual modes:

## 1. Storyboard Mode
Discrete hand-authored frame states.

## 2. Live Simulation Mode
Continuous state updates from Hydrion.

## 3. Event Replay Mode
Focused replay of important transitions.

## 4. Comparison Mode (Future)
Side-by-side or differential runs.

---

# Mode 1 — Storyboard Mode

## Purpose
Allow deterministic navigation through the 5 locked visual narrative states.

## Required Storyboard Frames

1. `CLEAN START`
2. `EARLY CAPTURE`
3. `RISING LOAD`
4. `THRESHOLD STATE`
5. `BACKFLUSH + AFTERMATH`

These are canonical visualization states.
They are not arbitrary screenshots.

## Required Controls

- previous frame
- next frame
- direct frame selection
- auto-play through storyboard
- pause storyboard playback

## Expected Use
- concept review
- design review
- UI validation
- communication with stakeholders
- fast narrative walkthroughs

## Timing Behavior
Storyboard playback should not feel like video playback.
Instead:
- each frame transition should feel deliberate
- playback pacing should be slow enough to inspect
- transitions may animate between frames if mapping allows

Suggested default:
- 1.5–3.0 s between frame advances in auto-play mode

---

# Mode 2 — Live Simulation Mode

## Purpose
Display real-time or pseudo-real-time Hydrion state updates.

## Required Behavior
- machine state changes continuously
- telemetry updates from live state
- animations interpolate appropriately
- no frame boundaries shown by default

## Required Controls
- pause state updates visually
- resume live playback
- snapshot current state
- inspect current values

## Optional Controls
- slow down live state visualization
- speed up non-physical playback for analysis
- single-step if simulation transport supports it

## Expected Use
- debugging
- validation
- simulation observation
- live control-loop interpretation

## Critical Rule
Pause in live mode should pause UI playback/visual progression, not necessarily mutate or stop the underlying simulator unless explicitly wired to do so.

---

# Mode 3 — Event Replay Mode

## Purpose
Replay meaningful machine transitions as inspectable narratives.

## Supported Event Types

- threshold crossing
- backflush burst
- backflush sequence
- bypass activation
- recovery window
- major fouling rise

## Required Behavior
- isolate event interval
- replay from a short pre-event buffer
- show event duration clearly
- allow repeated review

## Required Controls
- replay event
- step through event states
- scrub within event window
- reset to event start

## Expected Use
- analyzing maintenance behavior
- demonstrating causality
- debugging dynamics
- validating whether visuals match state logic

## Important Rule
Event replay is not an animation gimmick.
It is a diagnostic tool.

---

# Mode 4 — Comparison Mode (Future)

## Purpose
Compare two runs or two conditions side by side.

## Future Comparison Types
- calibrated vs proxy
- before vs after backflush
- electrostatics OFF vs ON
- scenario A vs scenario B
- truth vs sensor-biased view

## M2 Requirement
Do not implement full comparison mode yet unless it falls out cleanly from architecture.
Only structure playback so it will be possible later.

---

# Playback Controller UI Regions

The playback controller should be a **machine console control strip**, not a generic media footer.

It should integrate beneath or near the machine in a way that feels instrument-like.

## Required Elements

### 1. Mode Indicator
Show current mode:
- `STORYBOARD`
- `LIVE`
- `REPLAY`

### 2. Current State / Frame Label
Examples:
- `FRAME 03 · RISING LOAD`
- `LIVE · 00:28:47`
- `REPLAY · PULSE 01`

### 3. Core Controls
Must support:
- play
- pause
- previous
- next

### 4. Timeline / Scrubber
Behavior varies by mode:
- discrete frames in Storyboard
- continuous time in Live
- bounded event window in Replay

### 5. Speed Control
Optional but recommended.
Examples:
- `0.5x`
- `1.0x`
- `2.0x`

### 6. Snapshot / Reset
Optional in v1, but useful if implementation remains clean.

# Playback State Model

HydrOS playback should use an explicit controller state.

## Recommended type

```ts
type PlaybackMode = "storyboard" | "live" | "replay";

type StoryboardFrameId =
  | "clean_start"
  | "early_capture"
  | "rising_load"
  | "threshold_state"
  | "backflush_aftermath";

type ReplayEventType =
  | "threshold_crossing"
  | "backflush_burst"
  | "backflush_sequence"
  | "bypass_activation"
  | "recovery_window";

type PlaybackControllerState = {
  mode: PlaybackMode;
  isPlaying: boolean;
  speed: number;
  currentTimeSec: number;
  currentFrameIndex?: number;
  selectedReplayEvent?: ReplayEventType | null;
  replayStartSec?: number | null;
  replayEndSec?: number | null;
};
Storyboard Playback Mapping
Inputs
canonical frame presets
frame order index
Outputs
selected display state
displayed frame label
progress position
transition target
Rules
frame order is fixed
direct frame selection is allowed
frame transitions may interpolate visually, but state identity must remain clear
Suggested Default Order

0 → clean_start
1 → early_capture
2 → rising_load
3 → threshold_state
4 → backflush_aftermath

Live Playback Mapping
Inputs
incoming Hydrion truth state stream
optional sensor state stream
current wall-clock or sim-clock time
Outputs
current display state
live run time label
motion interpolation target
Rules
live data is authoritative
display may smooth selected values
meaningful abrupt events must remain visible
Smoothing Guidance
Good candidates for smoothing
flow label transitions
pressure label transitions
fouling texture density
burden glow intensity
Avoid oversmoothing
backflush onset
bypass activation
maintenance threshold crossing
pulse indexing
Replay Window Mapping
Inputs
stored state history
event markers
replay interval bounds
Outputs
replay playback state
event-specific timeline bounds
current event position
Requirements

The controller must be able to:

jump to event start
scrub event window
restart replay
stop replay and return to prior mode
Rule

Replay is only possible if historical states or event windows are stored.
If not yet implemented, the architecture must still leave room for it.

Display Labels

Playback labels must be explicit.

Storyboard examples
FRAME 01 · CLEAN START
FRAME 04 · THRESHOLD STATE
Live examples
LIVE · 00:28:47
Replay examples
REPLAY · BACKFLUSH BURST
REPLAY · THRESHOLD CROSSING
Rule

Labels must always reflect actual playback mode and position.

Timeline Behavior by Mode
Storyboard
discrete segments
5 fixed positions
snapping behavior preferred
Live
continuous bar or continuous run indicator
optional expanding run duration
no snapping
Replay
bounded local window
short focused range
event start and end visually marked
User Interaction Rules
Play
starts progression in current mode
Pause
freezes playback progression in current mode
does not necessarily stop underlying simulator unless explicitly connected
Previous / Next
valid in storyboard mode
optional in replay mode if stepping supported
not primary in live mode
Scrub
storyboard: snap to frame
replay: continuous within event window
live: only if historical buffer exists
Mode Switch

Switching modes should:

be explicit
preserve machine stability
not silently discard important context
Recommended v1 Implementation Scope

Implement now:

Required
mode toggle: storyboard vs live
play / pause
previous / next (storyboard)
frame selection (storyboard)
current mode label
current state/frame label
Recommended
speed control
basic scrubber
timeline strip
Future
replay event system
buffered live scrubbing
comparison mode
Playback + Machine View Contract

Playback controller must never dominate the machine.

The hierarchy is:

machine core
playback context
telemetry / advisory

Playback controls should feel:

integrated
minimal
instrument-like

Not:

oversized
flashy
consumer-video-like
Animation Coordination Rules

Playback must coordinate with animation, not fight it.

Storyboard transitions
can animate between states
should preserve continuity
must not blur state identity
Live mode
animation follows incoming state
interpolation should remain physically plausible
Replay mode
can emphasize event progression
may slow time if useful
must preserve causality
Failure Conditions

Playback design fails if:

mode is unclear
frame / time position is unclear
play/pause changes feel arbitrary
timeline behaves differently without explanation
machine truth is obscured by controller complexity
playback feels like entertainment UI rather than instrument control
Recommended React Architecture
Suggested separation
1. Playback Controller Hook

Example:

useHydrosPlaybackController()

Handles:

mode
play/pause
frame index
speed
timeline position
2. State Source Adapter

Chooses:

storyboard preset
live Hydrion state
replay slice
3. Machine View Renderer

Pure visual layer.

Rule

Do not bury playback logic inside presentation components.

Final Definition

HydrOS playback control is:

the temporal instrumentation layer of the simulation console

It must let users understand not only what the machine is,
but what it is doing over time and why.

If the machine can be seen but not navigated, playback failed.
If playback exists but feels generic or detached from the machine, playback failed.
The goal is:

disciplined control of machine-state time
