# HydrOS Animation Specification

## Purpose

This document defines how HydrOS should animate as a living machine.

It is not decorative motion design.

It is:
- physics-driven motion language
- state-driven visual behavior
- a machine-interpretation system for users

Every animation must help the viewer understand:
- flow
- pressure
- fouling
- backflush
- bypass
- system confidence

---

# Core Principle

HydrOS animation must feel:

- physically motivated
- elegant
- controlled
- cinematic
- high-trust
- non-gimmicky

The system should never feel like a gaming UI or a generic dashboard.

It should feel like:
> a premium industrial machine interface with visible internal truth

---

# Global Motion Rules

## Rule 1 — No meaningless animation
If it does not map to:
- a state variable
- a derived variable
- a machine event

it should not move.

## Rule 2 — Motion should clarify, not distract
Animation should make the machine easier to understand in 1–3 seconds.

## Rule 3 — Timing must match system behavior
Fast things:
- pressure spikes
- pulse bursts
- bypass activation

Slow things:
- fouling buildup
- recovery decay
- calibration status changes

## Rule 4 — Motion hierarchy
Priority order of visual importance:

1. backflush / bypass events
2. live flow behavior
3. pressure burden
4. fouling accumulation
5. sensor overlays
6. calibration trust shifts

## Rule 5 — Visual uncertainty must remain honest
Proxy or placeholder systems should not be given stronger or more dramatic motion language than grounded systems.

# Animation Layers

## Layer 1 — Baseline Idle State

### Purpose
Show that the system is alive, even when stable.

### Behavior
- faint flow shimmer in inlet/outlet channels
- subtle machine glow pulse
- tiny ambient particle drift within active stages
- extremely low-frequency UI luminance breathing

### Timing
- 3.5–6.0 second loops
- low amplitude only

### Emotional effect
- system feels active
- never feels dead or static
- never feels flashy

---

## Layer 2 — Flow Animation

## Purpose
Show throughput and direction immediately.

## Visual Encoding
- horizontal directional stream
- particle streaks or luminous vectors
- thickness scales with flow rate
- speed scales with velocity
- brightness scales with processed throughput

## Behavior by State

### Low flow
- narrow stream
- slower movement
- quieter brightness
- more visible internal detail

### Nominal flow
- smooth continuous stream
- balanced brightness
- elegant directional motion

### High / peak flow
- thicker stream
- higher velocity
- slight turbulence shimmer
- stronger edge glow

### Near overload
- increased agitation
- local instability at pressure-sensitive zones
- branch readiness visible near bypass path

## Timing
- fully continuous
- updated in real time with flow state
- smoothing window should be short enough to feel live, but not noisy

## Constraint
Flow animation must never imply more processed flow than the model actually outputs.

---

## Layer 3 — Pressure Animation

## Purpose
Make invisible hydraulic burden visible.

## Visual Encoding
Use one or more of:
- local color warming
- intensity bloom
- contour glow
- pressure pulse tracing along casing

## Pressure Behavior

### Clean / low pressure
- cool blue / neutral steel tone
- no visible stress concentration
- subtle structural glow only

### Moderate pressure
- slight warming near stage boundaries
- more visible compression lines
- local intensity rise around most burdened stage

### High pressure
- orange-white edge brightening
- structural tension effect
- stage casing glow intensifies
- pressure field visibly concentrates at bottleneck

### Post-backflush
- rapid drop in local burden glow
- residual stress remains if fouling not fully cleared

## Timing
- pressure transitions should lag flow slightly less than fouling
- spike response: 150–350 ms
- decay response: 300–800 ms
- high-pressure lingering glow: optional 0.3–0.6 s tail

## Constraint
Pressure animation must feel tense, not explosive.

---

## Layer 4 — Fouling Animation

## Purpose
Show accumulation, distribution, and maintenance need.

## Core Rule
Fouling is the slowest but most narratively important animation layer.

It should feel like:
> the machine gradually carrying more burden

## Stage-Specific Visual Language

### Stage 1 — Coarse
Visuals:
- larger fiber clusters
- loose tangles
- intermittent snag structures
- sparse but obvious obstruction

Behavior:
- appears first
- grows visibly early
- may not dominate pressure visually at first

### Stage 2 — Medium
Visuals:
- denser mixed matter
- intermediate particulate/fiber texture
- more even distribution

Behavior:
- transitional accumulation
- bridges early-stage and fine-stage behavior

### Stage 3 — Fine Pleated
Visuals:
- dense coating
- fine granular or film-like buildup
- cake layer spreading over media surface
- local pore-darkening

Behavior:
- grows into system bottleneck
- strongest pressure-coupled fouling expression
- should feel like the stage that eventually “chokes” the machine

## Temporal Behavior

### Early run
- Stage 1 visibly catches first debris
- Stage 3 still relatively open but beginning to haze

### Mid run
- Stage 2 and 3 accumulation becomes more apparent
- Stage 3 begins visually dominating burden

### Late run
- Stage 3 visibly dense and stressed
- pressure glow and fouling density align
- maintenance threshold becomes visually intuitive

## Recovery Behavior
Backflush should:
- reduce visible fouling partially
- clear outer layers first
- leave residual burden
- never make the stage look brand new unless system is explicitly reset

## Timing
- accumulation should feel gradual
- updates every 0.5–2.0 s visually, even if underlying model updates faster
- nonlinear acceleration should become noticeable later in the run

## Constraint
Fouling must not “pop in.”
It must accumulate.

---

## Layer 5 — Backflush Animation

## Purpose
Backflush must feel like a discrete machine event.

## Core Character
- sharp
- bright
- directional
- brief
- purposeful

## Visual Elements
- reverse-flow pulse through backflush path
- bright pressure burst
- short clearing effect inside stages
- local turbulence disruption
- pulse indexing if multiple pulses are shown

## Pulse Sequence
Current doctrine:
- 3 pulses
- 0.4 s active
- 0.25 s interpulse
- cooldown after burst

## Visual Sequence

### Burst start
- short arming flash
- backflush path lights up
- machine burden pauses visually for a beat

### Active pulse
- reverse directional motion
- strong brightness spike
- local debris displacement
- sudden reduction in outer fouling layer density

### Interpulse
- pulse glow decays
- reverse motion stops
- residual turbulence remains briefly

### Next pulse
- repeat with slightly reduced clearing yield if diminishing returns are visible

### Post-burst
- system settles
- pressure burden drops partially
- residual fouling remains visible
- cooldown indicator activates

## Timing
- backflush pulse must feel faster than any other event in the system
- onset: 80–120 ms
- active pulse: match real modeled duration
- visual fade tail: 120–250 ms

## Constraint
Backflush should never look like a total cleanse unless the model actually says so.

---

## Layer 6 — Bypass Animation

## Purpose
Show overload protection as a structural event.

## Character
Bypass should feel:
- rare
- sudden
- serious
- protective

## Visual Encoding
- alternate branch path activates
- bright diverted flow line
- distinct color from processed flow
- processed channel visibly reduces relative intensity

## Activation Behavior
- bypass path “opens”
- branch stream appears
- outlet split becomes visually obvious
- system status shifts immediately

## Deactivation Behavior
- bypass fades more slowly than it activates
- hysteresis should be visually implied
- branch should not flicker on/off rapidly

## Timing
- activation: 120–220 ms
- deactivation: 300–700 ms

## Constraint
Bypass should be dramatic but not catastrophic-looking.

---

## Layer 7 — Timeline / Event Narrative Animation

## Purpose
Represent machine history as a narrative strip.

## Events to show
- cycle start
- flow stabilization
- fouling rise
- threshold crossing
- pulse events
- bypass readiness
- bypass activation
- maintenance advisory

## Motion Style
- event markers slide or light into place
- active event is highlighted
- past events remain visible but dimmed
- current phase indicator should track system progress smoothly

## Timing
- event appearance should feel deliberate, not instant
- 200–350 ms for event reveal
- current marker motion should be continuous

## Constraint
Timeline should feel like a machine log, not a social media feed or playback scrubber by default.

---

## Layer 8 — Calibration Trust Animation

## Purpose
Communicate realism honesty.

## States
- grounded
- proxy
- placeholder

## Visual Behavior
- grounded: stable illumination
- proxy: softer pulsing uncertainty band
- placeholder: lower-confidence subdued or amber-toned presence

## Timing
- trust state should change rarely
- transitions should be slow and explicit
- 400–800 ms fade

## Constraint
Calibration status must not compete visually with physical events.

---

## Layer 9 — Truth vs Sensor Overlay Animation

## Purpose
Show difference between actual state and measured state.

## Behavior
- truth layer remains crisp and primary
- sensor layer may show:
  - slight lag
  - mild smoothing
  - modest jitter
  - optical uncertainty

## Pressure / Flow
- small delay acceptable
- mostly faithful

## Optical
- more noise
- more uncertainty
- occasional fouling-related distortion

## Timing
- sensor animation should always feel derived from truth, never independent

## Constraint
User must never confuse sensor artifact with actual machine state.

# Visual Mood References

HydrOS motion should feel like a blend of:

- precision instrumentation
- industrial machine diagnostics
- premium operating interface
- cinematic internal cutaway

It should NOT feel like:
- gaming HUD
- flashy sci-fi for its own sake
- crypto dashboard
- neon overload
- consumer gimmick UI

|---|---|---|
| Backflush burst | Highest | discrete machine event |
| Bypass activation | Highest | protective structural change |
| Flow state | High | always-on machine behavior |
| Pressure burden | High | key invisible variable |
| Fouling growth | High | maintenance narrative |
| Sensor noise | Medium | secondary interpretation |
| Calibration trust | Low-Medium | important but not dominant |

# Machine View v1 Minimum Animation Set

For the first implemented version, HydrOS only needs:

1. continuous flow motion
2. pressure glow behavior
3. gradual fouling buildup
4. backflush burst animation
5. bypass branch activation
6. event timeline highlighting

Everything else can be layered later.

# Frame-Based Storyboard Candidate

## Frame 1 — Clean Start
- nominal cool flow
- low pressure glow
- light Stage 1 capture only

## Frame 2 — Rising Load
- Stage 1 debris visible
- Stage 2 moderate accumulation
- Stage 3 haze begins
- pressure warming starts

## Frame 3 — Threshold Near
- Stage 3 dense coating
- pressure concentrated
- maintenance advisory appears

## Frame 4 — Backflush Pulse
- reverse burst active
- bright backflush path
- local clearing in stages

## Frame 5 — Post-Pulse Recovery
- pressure partially reduced
- residual fouling remains
- cooldown state visible

## Frame 6 — Overload / Bypass Ready
- processed path stressed
- bypass branch glowing in standby or active mode

# Final Definition

HydrOS animation is:

> the visual choreography of physical truth

If motion is beautiful but not grounded, it fails.
If motion is grounded but unreadable, it must be redesigned.
The goal is not spectacle.
The goal is intuitive machine truth.
