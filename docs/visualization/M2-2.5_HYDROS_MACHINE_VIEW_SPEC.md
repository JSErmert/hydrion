# HydrOS Machine View Specification

## Purpose

Define how HydrOS physics is visually represented.

This is NOT UI design.

This is:
> a visual mapping of simulation truth into human intuition

---

# Core Principle

Every visual element must map to:
- a real state variable
- or a derived physical quantity

No decorative elements allowed.

---

# Visual Zones

## 1. Machine Core View (Centerpiece)

This is the main system.

Must show:

- inlet → flow entry
- Stage 1 (coarse)
- Stage 2 (medium)
- Stage 3 (fine pleated)
- outlet

Also include:
- backflush path
- bypass path

---

## 2. Flow Representation

Flow must be:

- directional (left → right)
- animated
- velocity-scaled

### Visual Encoding

- thickness = flow rate
- speed = velocity
- brightness = throughput

---

## 3. Pressure Representation

Pressure is invisible physically → must be visualized

### Encoding options

- color gradient (cool → hot)
- glow intensity
- contour lines

### Behavior

- increases with fouling
- spikes near Stage 3
- drops after backflush

---

## 4. Fouling Representation

Each stage must show:

- buildup over time
- density variation

### Encoding

Stage 1:
- chunky / fiber clusters

Stage 2:
- mixed texture

Stage 3:
- dense surface coating (cake)

### Behavior

- gradual accumulation
- nonlinear acceleration
- partial clearing after backflush

---

## 5. Backflush Visualization

Backflush must feel like an EVENT

### Behavior

- reverse flow pulse
- visible burst
- short duration
- repeated sequence

### Encoding

- reverse direction animation
- bright pulse flash
- temporary clearing effect

---

## 6. Bypass Visualization

Must only activate under overload

### Encoding

- branch flow path
- distinct color
- partial diversion

### Behavior

- activates suddenly
- deactivates with hysteresis

---

## 7. Timeline Layer

Must show:

- flow events
- fouling rise
- pulse events
- threshold crossing

This is NOT a chart.

It is:
> a narrative strip of system behavior

---

## 8. Calibration Trust Layer

Every major component must show:

- grounded
- proxy
- placeholder

### Encoding

- green = grounded
- yellow = proxy
- red/orange = placeholder

This prevents fake realism.

---

## 9. Sensor Overlay

Must distinguish:

- truth vs measurement

### Encoding

- slight delay
- noise
- smoothing

---

# Interaction Model

User should be able to:

- pause system
- scrub timeline
- isolate stages
- trigger backflush manually
- compare before/after

---

# Design Tone

HydrOS should feel:

- cinematic
- precise
- industrial
- minimal
- high-trust

NOT:
- playful
- cluttered
- dashboard-like
- generic SaaS

---

# Critical Constraints

1. No visual without physics mapping
2. No smoothing that hides real behavior
3. No fake signals
4. Must reflect calibration uncertainty

---

# Future Expansion (M2+)

- particle visualization
- electrostatic field overlay
- agent recommendations (City of HydrOS)
- predictive alerts

---

# Final Definition

Machine View is:

> The visual language of HydrOS reality

If it looks right but is not grounded → it is wrong  
If it is grounded but unclear → it must be redesigned
