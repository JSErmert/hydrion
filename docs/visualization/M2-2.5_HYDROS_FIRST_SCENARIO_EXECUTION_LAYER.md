# HydrOS First Scenario Execution Layer

## Purpose

Define the first executable scenario runtime layer for HydrOS.

This layer is the initial practical implementation of the Scenario Engine.
It is responsible for taking a structured scenario definition and producing
time-evolving machine state that can drive:

- Hydrion execution
- Machine View playback
- event markers
- validation runs
- future RL scenarios

This is the first version of:
> scenario definition → scenario runtime → machine state stream

---

# Core Principle

A scenario must become:

> a deterministic sequence of machine inputs over time

This layer must not yet attempt to solve:
- full scenario editor UX
- batch study orchestration
- complex stochastic scheduling
- full comparison engine

Instead, it must provide the minimum clean runtime needed to execute
research-grade scenarios in a structured way.

---

# Responsibilities

The first scenario execution layer must:

1. load a scenario definition
2. initialize a scenario runtime state
3. compute current inputs at each timestep
4. apply disturbances when active
5. feed those inputs into Hydrion
6. emit per-step execution records
7. mark major scenario events for playback and replay

---

# High-Level Architecture

## 1. Scenario Definition
Static structured description of:
- initial state
- flow profile
- particle profile
- disturbances
- duration
- dt
- seed

## 2. Scenario Runtime
Tracks:
- current time
- active disturbances
- current input conditions
- event markers

## 3. Hydrion Adapter
Converts scenario runtime conditions into:
- machine inputs
- environment configuration adjustments
- step execution calls

## 4. Output Stream
Produces:
- truth state history
- sensor state history
- event markers
- metadata

Scenario Execution Loop

At a high level, the first execution loop should work like this:

load scenario
initialize env
apply initial state
for each timestep:
    compute flow input
    compute particle input
    activate/deactivate disturbances
    apply scenario conditions to env
    step Hydrion
    record truth/sensor state
    record event markers if needed
return execution history
Phase 1 Scope

The first implementation only needs to support:

Required
deterministic static scenario loading
time-based flow calculation
time-based particle density calculation
disturbance activation / deactivation
scenario stepping against Hydrion
output history recording
Not required yet
graphical scenario editor
scenario comparison UI
advanced event analytics
probabilistic disturbance generation
multi-run orchestration
Input Computation Rules
1. Flow Input Computation

At each timestep, compute:

function computeFlowAtTime(
  profile: FlowProfile,
  t: number,
  seed?: number
): number
Behavior by profile type
constant

Return:

baseFlowLmin
ramp

Return:

gradual increase or decrease from base toward target
v1 may encode target in transientEvents or a simple extension field
burst

Return:

base flow plus periodic burst structure
may use:
burstAmplitude
burstFrequency
realistic

Return:

base flow plus bounded variability
plus transient event overlays if provided
Constraint

The result must remain physically bounded for HydrOS:

no negative flow
no impossible spikes
nominally within doctrine unless testing overload intentionally
2. Particle Density Computation

At each timestep, compute:

function computeParticleDensityAtTime(
  profile: ParticleProfile,
  t: number,
  seed?: number
): number
Behavior by profile type
fiber_dominant
steady moderate density
low-to-moderate variability
default HydrOS laundry case
mixed
moderate density
more balanced behavior
heavy_load
high density
suitable for stress scenarios
Constraint

Keep normalized density in:

[0, 1]
Disturbance Activation Model

Each timestep, determine which disturbances are active.

function getActiveDisturbances(
  disturbances: DisturbanceEvent[],
  t: number
): DisturbanceEvent[]

A disturbance is active if:

time <= t < time + duration

When disturbance state changes:

emit event markers
Disturbance Effects (v1)

The first execution layer should support simple, explicit disturbance transforms.

flow_spike

Effect:

add positive offset to current flow
clamp to scenario-safe maximum if needed
particle_spike

Effect:

add positive offset to current particle density
clamp to [0,1]
partial_blockage

Effect:

inject additional burden factor
may temporarily bias stage resistance or fouling seed
v1 can implement as additional fouling offset before stepping
sensor_noise

Effect:

no truth-state change
mark for future sensor layer handling
may just emit marker in v1
foam_event

Effect:

no truth-state change initially
mark for future optical/sensor realism handling
may just emit marker in v1

Recommended strategy

The adapter should:

set any controllable environment input channels
inject particle density proxy where supported
apply disturbance-side bias carefully
avoid mutating unrelated system state
Important

If Hydrion does not yet expose direct setters for some inputs, create a clearly named

Event Marker Rules

The scenario execution layer must emit event markers whenever meaningful transitions occur.

Required markers
scenario_start

At t = 0

disturbance_start

When any disturbance activates

disturbance_end

When any disturbance deactivates

scenario_end

At final timestep

Conditional markers

Emit if observed in Hydrion truth state:

threshold_crossing

When maintenance_required changes from 0 -> 1

backflush_start

When bf_active changes from 0 -> 1

backflush_end

When backflush burst completes or activity returns inactive after a burst

bypass_start

When bypass_active changes from 0 -> 1

bypass_end

When bypass_active changes from 1 -> 0

These markers are essential for:

playback controller
event replay
timeline visualization

Minimal Python-Oriented Runtime Skeleton

If implemented in Python first, the first clean version can look like this:

class ScenarioRunner:
    def __init__(self, env):
        self.env = env

    def run(self, scenario: dict) -> dict:
        dt = scenario["dt"]
        duration = scenario["durationSec"]
        total_steps = int(duration / dt)

        self.env.reset(seed=scenario.get("seed"))
        apply_initial_state_to_hydrion(self.env, scenario["initialState"])

        history = {
            "scenarioId": scenario["id"],
            "dt": dt,
            "steps": [],
            "eventMarkers": [],
        }

        history["eventMarkers"].append({
            "time": 0.0,
            "type": "scenario_start",
            "label": "Scenario Start",
        })

        prev_flags = {
            "maintenance_required": 0,
            "bf_active": 0,
            "bypass_active": 0,
        }

        for step_idx in range(total_steps):
            t = step_idx * dt

            flow = compute_flow_at_time(scenario["flowProfile"], t, scenario.get("seed"))
            particle_density = compute_particle_density_at_time(
                scenario["particleProfile"], t, scenario.get("seed")
            )
            active_disturbances = get_active_disturbances(scenario["disturbances"], t)

            inputs = {
                "flowLmin": flow,
                "particleDensity": particle_density,
                "activeDisturbances": active_disturbances,
            }

            apply_scenario_inputs_to_hydrion(self.env, inputs)

            obs, reward, done, info = self.env.step(self._build_action_for_runtime())

            truth_state = dict(self.env.truth_state)
            sensor_state = dict(getattr(self.env, "sensor_state", {}))

            history["steps"].append({
                "t": t,
                "stepIndex": step_idx,
                "scenarioInputs": inputs,
                "truthState": truth_state,
                "sensorState": sensor_state,
                "reward": reward,
                "done": done,
                "info": info,
            })

            history["eventMarkers"].extend(
                detect_runtime_markers(t, prev_flags, truth_state)
            )

            prev_flags["maintenance_required"] = int(truth_state.get("maintenance_required", 0))
            prev_flags["bf_active"] = int(truth_state.get("bf_active", 0))
            prev_flags["bypass_active"] = int(truth_state.get("bypass_active", 0))

            if done:
                break

        history["eventMarkers"].append({
            "time": history["steps"][-1]["t"] if history["steps"] else 0.0,
            "type": "scenario_end",
            "label": "Scenario End",
        })

        return history

    def _build_action_for_runtime(self):
        # v1 placeholder:
        # stable nominal action or externally injected control policy
        return [0.8, 1.0, 0.0, 0.5]

Integration with Playback Controller

The first scenario execution layer must produce everything needed for playback.

Storyboard Mode

Still uses fixed canonical frame presets.

Live Mode

Uses current runtime step or rolling state stream.

Replay Mode

Uses:

event markers
step history
replay windows

This means the scenario runtime must be replay-aware from day one, even if the replay UI is implemented later.

Integration with Machine View

The execution layer should NOT render anything directly.

It must only provide:

state history
event history
current step selection

Machine View then consumes adapted display state from that history.

This separation is critical.

Failure Conditions

The first scenario execution layer fails if:

scenario runs are not reproducible
event markers are missing
disturbance timing is ambiguous
history is not usable for replay
execution logic is hidden inside UI components
scenario inputs mutate unrelated system internals
output is not structured enough to feed playback cleanly

Final Definition

The first scenario execution layer is:

the runtime machinery that turns HydrOS scenarios into reproducible machine-state histories

Without it, HydrOS can show state.
With it, HydrOS can generate, inspect, replay, and study machine reality over time.
