# HYDROS_UI_SCHEMA_SPEC.md

## Purpose

Define the exact UI schema for HydrOS Machine View v1.

This document is the bridge between:
- cinematic concept art
- scientific instrumentation
- React implementation
- Hydrion state binding

This is not a mood board.
This is not a loose dashboard idea.

This is:
> the authoritative interface structure for HydrOS M2

---

# Core Principle

HydrOS UI must behave like a **scientific machine interface**.

That means:

- the machine is central
- telemetry is exact
- labels are stable
- values are bound to state
- nothing important is left implied
- no generated placeholder text is allowed

If the machine is the reality surface, this schema is the instrument panel around it.

---

# Design Doctrine

HydrOS UI should feel:

- precise
- elevated
- minimal
- technical
- premium
- physically interpretable

It should NOT feel:

- messy
- crowded
- game-like
- sci-fi for its own sake
- overly card-driven
- generic SaaS

The outside of the machine should remain:
- clean
- quiet
- readable
- controlled

The inside of the machine is where:
- fluid
- particles
- fouling
- turbulence
- maintenance events
are shown.

---

# Screen Hierarchy

HydrOS Machine View v1 consists of 5 primary regions:

1. Header Bar
2. Core Machine View
3. Top Telemetry Band
4. Bottom Narrative and Control Band
5. Right Advisory Panel

---

# 1. Header Bar

## Purpose
Provide stable system identity.

## Required Elements

### Product Name
- `HydrOS`

### View Name
- optional subtitle:
  - `Machine View`
  - `Simulation Playback`
  - `Live Calibration Mode`

### Run Context Badge
Examples:
- `Scenario Playback`
- `Bench Calibration`
- `Live Sim`
- `Backflush Replay`

## Rules
- header is always centered and calm
- no moving metrics here
- no clutter
- no decorative noise

---

# 2. Core Machine View

## Purpose
This is the primary reality surface.

It must occupy the largest visual area.

## Required Physical Elements

### Flow Path
- inlet
- Stage 1
- Stage 2
- Stage 3
- outlet

### Auxiliary Paths
- backflush line
- bypass branch

### Chamber Labels
- `INLET`
- `STAGE 01`
- `STAGE 02`
- `STAGE 03`
- `OUTLET`
- `BACKFLUSH`

## Rules
- machine geometry stays fixed
- camera angle stays fixed
- no drifting between runs
- all animation happens inside the fixed geometry

---

# 3. Top Telemetry Band

## Purpose
Show machine-critical values immediately.

## Layout
A compact horizontal telemetry strip above or above-left of the machine.

## Required Fields

### Processed Flow
Label:
- `FLOW RATE`

Value format:
- `13.5 L/min`

State source:
- `q_processed_lmin`

### Pressure
Label:
- `PRESSURE`

Value format:
- `45.2 kPa`

State source:
- `dp_total_pa` or selected pressure representation converted to kPa

### Bypass Status
Label:
- `BYPASS`

Value format:
- `OFF`
- `READY`
- `ACTIVE`

State source:
- `bypass_active`
- optional readiness derived from threshold proximity

### Optional Mode Tag
Examples:
- `NORMAL`
- `STRESSED`
- `BACKFLUSH ACTIVE`

## Rules
- every field must always contain a value
- no blank gauges
- no placeholder dashes
- fixed typography and units
- telemetry should update smoothly, not jump

---

# 4. Bottom Narrative and Control Band

## Purpose
Show sequence, burden, and next action.

This region explains what the machine is doing over time.

## 4A. Run Timeline

### Required Label
- `RUN CYCLE`

### Required Value
Example:
- `00:28:47`

### Event Strip
Must support:
- `START`
- `FOULING RISE`
- `PULSE 01`
- `PULSE 02`
- `BYPASS READY`
- `BYPASS ACTIVE`
- `RECOVERY`

## Rules
- this is a machine narrative strip, not a stock timeline widget
- active phase is highlighted
- past phases remain visible but subdued

---

## 4B. Stage Status Readout

### Required Fields
For each stage:

- `STAGE 01`
- `STAGE 02`
- `STAGE 03`

Each stage must show:
- fouling percentage
- optional capture density indicator
- optional pressure burden emphasis

### Value Format
Examples:
- `08% FOULING`
- `25% FOULING`
- `92% FOULING`

State sources:
- `fouling_frac_s1`
- `fouling_frac_s2`
- `fouling_frac_s3`

## Rules
- percentages must always be numeric
- fixed formatting
- all three stages always present

---

## 4C. Circular Instrument Cluster

Two compact circular instruments are allowed in v1.

### Instrument 1 — Fiber Load
Label:
- `FIBER LOAD`

Value options:
- `LOW`
- `MODERATE`
- `HIGH`
- `CRITICAL`

Optional numeric mapping may exist internally, but text output remains categorical.

### Instrument 2 — Pulse Status
Label:
- `PULSE`

Value options:
- `0%`
- `42%`
- `ACTIVE`
- `READY`
- `COOLDOWN`

Can also show:
- `PULSE 01`
- `PULSE 02`
- `PULSE 03`

## Rules
- both instruments must always show real state
- no empty circles
- no decorative meter with no meaning

---

## 4D. Next Action Panel

### Required Label
- `NEXT ACTION`

### Allowed Content
Short imperative recommendations only.

Examples:
- `CONTINUE RUN`
- `MONITOR STAGE 03`
- `INITIATE BACKFLUSH`
- `PREPARE BENCH TEST`
- `BYPASS ON STANDBY`

## Rules
- maximum 2–3 lines
- recommendations must be state-driven
- recommendations must not contradict machine state

---

# 5. Right Advisory Panel

## Purpose
Show system interpretation without overpowering the machine.

## 5A. System Status Box

### Required Label
- `SYSTEM STATUS`

### Allowed Values
- `NORMAL OPERATION`
- `RISING LOAD`
- `MAINTENANCE REQUIRED`
- `BACKFLUSH ACTIVE`
- `STABILIZED / MONITORING`
- `BYPASS ACTIVE`

## Rules
- this is the top right anchor
- one status only
- status color follows severity semantics

---

## 5B. Efficiency Ring

### Required Label
- `EFFICIENCY`

### Value Format
- `82%`

Can later be split into:
- hydraulic efficiency
- capture efficiency

But in v1 it remains one compact percentage.

## Rules
- always numeric
- no decorative empty ring

---

## 5C. Agent Insights / Advisory Box

### Required Label
- `AGENT INSIGHTS`

### v1 Meaning
This is not yet full City of HydrOS.
It is a bounded advisory surface.

### Allowed Lines
- max 3 concise lines
- examples:
  - `System nominal`
  - `Low particle load`
  - `Monitor Stage 03`
  - `Initiate backflush`
  - `Bypass on standby`

## Rules
- insights are subordinate to physical truth
- no speculative language
- no long explanations in v1

---

# Allowed Typography

## Style
- uppercase labels
- high legibility
- disciplined hierarchy
- fixed unit formats

## Label format
Examples:
- `FLOW RATE`
- `PRESSURE`
- `SYSTEM STATUS`

## Value format
Examples:
- `13.5 L/min`
- `45.2 kPa`
- `82%`

## Forbidden
- distorted labels
- autogenerated pseudo-text
- inconsistent unit styling
- missing numbers

---

# Allowed Color Logic

## Base palette
- dark steel / graphite UI shell
- cool blue for nominal fluid state
- amber/orange for burden and maintenance attention
- restrained green for stable operation
- red only for true critical state
- purple reserved for future electrostatic mode only if explicitly enabled

## Semantic use

### Blue
- flow
- nominal hydraulic state
- backflush fluid path when appropriate

### Amber / Orange
- fouling burden
- rising pressure
- maintenance posture

### Green
- normal stable status
- grounded confidence

### Red
- critical warning
- threshold breach
- urgent maintenance state

### Gray / Steel
- UI shell
- non-active regions
- neutral machine structure

## Rules
- color must always mean something
- no neon for its own sake
- no lightning-bolt style electrical language in hydraulic-only views

# Cleanliness Rules

## Outside the machine
Must remain:
- clean
- dark
- quiet
- premium
- low-noise

Forbidden outside the machine:
- floating particles
- chaotic splatter
- visual debris clouds
- unnecessary glow artifacts

## Inside the machine
May show:
- fluid haze
- particles
- bubbles
- turbulence
- fouling
- backflush disturbance

This is critical:
> chaos belongs inside the process path, not outside the instrument panel

---

# Detergent Realism Rules

HydrOS processes laundry outflow, not clean water.

So inside the machine:

- fluid should be slightly hazy
- microbubble presence is allowed
- detergent / surfactant feel should be subtle
- flow should be visibly fluid, not laser-like

But:
- no excessive foam clouds
- no milky over-obscuring sludge
- no messy splashing outside chambers

The correct look is:
> engineered wastewater, not dirty chaos

---

# State Binding Contract

Every displayed value must bind to either:
- a real Hydrion state variable
- a governed derived variable

## Direct state examples
- `q_processed_lmin`
- `dp_total_pa`
- `fouling_frac_s1`
- `fouling_frac_s2`
- `fouling_frac_s3`
- `bf_active`
- `bf_pulse_idx`
- `bypass_active`

## Derived state examples
- fiber load category
- efficiency score
- maintenance required
- bypass ready
- next action
- advisory lines

## Rule
If a metric cannot be bound, it does not belong in v1.

# Frame Story Set for Visual Development

For image and storyboard work, lock the frame count to 5.

## Frame 1
- `CLEAN START`

## Frame 2
- `EARLY CAPTURE`

## Frame 3
- `RISING LOAD`

## Frame 4
- `THRESHOLD STATE`

## Frame 5
- `BACKFLUSH + AFTERMATH`

This removes unnecessary duplication and makes each frame narratively distinct.

HydrOS UI schema is:

> the authoritative structure that keeps the machine view readable, truthful, and professionally controlled

> scientific clarity with cinematic discipline
