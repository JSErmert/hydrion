# CLAUDE RESEARCH → HYDROS INTEGRATION PROMPT

You are integrating real-world research into the HydrOS / Hydrion system.

This is NOT a generic research summary task.

This is a **controlled calibration operation**.

---

# Objective

Take findings from:

- HYDROS_REALITY_CALIBRATION_REPORT.md
- HYDROS_CALIBRATION_TABLE.md
- CALIBRATION_PARAMETER_REGISTER.md

and translate them into:

- concrete parameter updates
- YAML configuration changes
- justified system behavior adjustments

---

# Core Doctrine

HydrOS is a **governed simulation system**.

You must:

1. respect parameter authority (register)
2. respect evidence hierarchy:
   - Direct
   - Proxy
   - Placeholder
3. preserve system architecture
4. avoid ungrounded assumptions

---

# Required Process

For each parameter you modify:

## Step 1 — Identify Parameter
- name
- module (hydraulics, clogging, etc.)

## Step 2 — Source Justification
- which research source supports this
- what quantity is extracted

## Step 3 — Mapping Logic
- how real-world data maps to simulation variable

## Step 4 — Update Value
- proposed new value or range
- units
- confidence level

## Step 5 — Register Update
Update CALIBRATION_PARAMETER_REGISTER.md with:
- new value
- source
- evidence class
- confidence
- replacement condition

## Step 6 — YAML Update
Modify:
- configs/default.yaml

Only for parameters that meet criteria.

---

# Constraints

Do NOT:

- change parameters without justification
- promote proxy → direct without evidence
- break pipeline assumptions
- change multiple interacting parameters blindly
- introduce new parameters without documentation

---

# Allowed Actions

You MAY:

- tighten parameter ranges
- replace placeholder values with proxy values
- replace proxy values with direct values
- introduce bounded uncertainty ranges

---

# Output Format

## A. Parameters Updated
List all changed parameters.

## B. Justification
For each:
- source
- reasoning

## C. YAML Changes
Provide exact edits.

## D. Register Updates
Provide exact additions/modifications.

## E. Expected Impact
Explain how system behavior should change.

---

# Final Standard

Every change must answer:

> “Is this closer to physical reality than before?”

If not, do not apply it.

You are not optimizing performance.
You are improving realism.
