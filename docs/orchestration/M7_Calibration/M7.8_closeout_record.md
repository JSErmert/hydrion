## 5. Benchmark Lineage

### Active canonical baselines after M7

| Artifact | Environment | Obs schema | Provenance | Status |
|----------|-------------|------------|------------|--------|
| `ppo_cce_v2` | ConicalCascadeEnv | 12D truth-derived | M5 | Canonical — unchanged |
| `ppo_hydrienv_v1` | HydrionEnv | obs14_v1 (14D) | M7 | **Canonical — new** |

### Comparability constraints

`ppo_cce_v2` and `ppo_hydrienv_v1` are **not directly comparable.** The two baselines train
on different environments (CCE vs HydrionEnv), different physics, different reward functions,
different observation schemas, and different operating envelopes. A performance number from
one cannot be interpreted relative to a performance number from the other.

### Incompatible pre-M7 HydrionEnv artifacts

Any HydrionEnv policy artifact trained before M7 (under `obs12_v2` or any earlier schema)
is not benchmark-compatible with `ppo_hydrienv_v1`. The VecNormalize statistics are
schema-specific — a 12D VecNorm is incompatible with a 14D obs vector. These artifacts
are implicitly superseded by `ppo_hydrienv_v1` for benchmark purposes. They were not
deleted; their incompatibility is a documented fact.

## 6. What M7 Proved

1. **End-to-end HydrionEnv PPO pipeline works under obs14_v1.** Training, VecNormalize,
   schema lock, ShieldedEnv, and checkpoint saving all function correctly at 500k steps.

2. **Sensor-derived channels (indices 12 and 13) are active during training and evaluation.**
   Both channels are non-zero and varying (AC8). The dual-channel distinction values (2.56
   and 3.73 normalized units) confirm the sensor readings differ meaningfully from their
   truth counterparts (AC9).

3. **A reproducible, metadata-documented canonical HydrionEnv baseline now exists.** Seed,
   schema version, training parameters, and known limitations are all recorded in
   `ppo_hydrienv_v1_meta.json`. Future HydrionEnv policy work has a fixed comparison point.

4. **Benchmark lineage was maintained throughout implementation.** CCE and `ppo_cce_v2`
   were left untouched and verified via git diff. The M7 implementation path never touched
   CCE infrastructure.

5. **A full 7-document governance chain was completed** for a single milestone:
   research brief → sources map → execution document → planning refinement → implementation
   plan → execution prompt → execution report → closeout record.

---

## 7. What M7 Did Not Prove

1. **Deployment validity.** Truth channels 0–9 are privileged simulation-only inputs
   unavailable at hardware. The policy will not function correctly if those channels are
   removed without retraining on a sensor-only observation vector.

2. **Sensor-only control sufficiency.** The policy exploits truth channels. Sensor channels
   12 and 13 being active does not mean they are necessary or sufficient for control. That
   question requires a separate ablation (training a policy with truth channels removed).

3. **Robustness or generalization.** The policy was trained on a fixed noise regime with
   calibration-pending placeholder parameters. Behavior under varied noise, domain
   randomization, or out-of-distribution conditions is unknown.

4. **Calibration-aware realism.** `dp_drift_rate`, `dp_fouling_gain`, and
   `flow_calibration_bias_std` are placeholder values. The sensor noise characteristics
   observed during M7 training will change when calibrated values replace them.

5. **CCE/HydrionEnv comparability.** Return values from `ppo_hydrienv_v1` evaluation are
   not comparable to return values from `ppo_cce_v2` evaluation.

## 10. Correction / Discrepancy Note

### Background execution kill limit

The M7.4 plan assumed training could be run as a background job. In practice, the execution
environment kills background Bash processes at approximately 5 minutes, which is insufficient
for a 500k-step training run (~16 minutes). The user ran training manually in a PowerShell
terminal. This is not a defect in the training script or environment — it is an environment
constraint that applies to M8 training as well.

**Implication for M8:** Any training run exceeding ~5 minutes must be run manually in a
terminal. The M8 planning document should note this explicitly.

### Binary artifact disposition — clarification

M7.4 Section 7 specified "commit training artifacts." The `.gitignore` file has `*.zip` and
`*.pkl` patterns under "Training artifacts." Adding these files would require `-f` (force),
which conflicts with the project's established `.gitignore` policy (confirmed by the fact that
`ppo_cce_v2` artifacts are also not committed). The metadata JSON is committed; the binary
weights and stats files are local-only. This is consistent with the broader project practice,
not a deviation from M7 intent.

### Eval Unicode fix

`eval_ppo_hydrienv_v1.py` as created in `c3f57e9` contained Unicode box-drawing characters
(`U+2500 ─`, `U+2014 —`) in print separator lines. These characters cause
`UnicodeEncodeError` on Windows cp1252 consoles. The fix (replace with ASCII `-` and `N/A`)
was applied in `896c2f3`. The fix does not affect evaluation logic, AC verification, or
sensor diagnostic output.
