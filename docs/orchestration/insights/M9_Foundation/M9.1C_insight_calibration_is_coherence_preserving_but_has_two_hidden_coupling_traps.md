## The insight in one paragraph

Sensor calibration is structurally safe for HydrOS — not because the sensor pipeline is isolated, but because **the reward function is computed from `truth_state` alone (`env.py:304–308`) and is therefore invariant under any sensor parameter change**. This single architectural property rules out an entire category of failure mode ("reward misalignment under new signals") at the structure level rather than at the discipline level. It is the strongest coherence anchor in the system and the reason M9 can proceed without redesign. However, this structural safety ends at the boundary between the environment and the trained-artifact ecosystem: **two hidden coupling traps sit between `configs/default.yaml` and the saved policy artifacts** — (a) `configs/default.yaml` is a shared single source of sensor truth for v1, v2, and CCE eval paths, so a naive in-place edit would silently break the AC11 canonical match (694.639) and invalidate the v2 placeholder-noise reference point; and (b) the saved `*_vecnorm.pkl` running statistics carry pre-calibration distribution moments that would be invalid if ever applied to a calibrated env. Both traps are preventable by **structural** provisions in M9.3 (config isolation via a dedicated `configs/m9_calibrated.yaml` + explicit stale-vecnorm prohibition), not by discipline or vigilance alone.

---

## Why this matters

The integration analysis set out to determine whether calibrated sensor parameters would keep the system coherent, stable, and interpretable. The answer is **yes**, but the reasoning is not "the sensor pipeline is well-isolated" — it is "the reward landscape is calibration-invariant." That distinction is load-bearing. Sensor isolation alone would be a weaker argument because the observation distribution still shifts and the policy is trained end-to-end on that distribution. What makes calibration structurally safe is that the **optimization target** does not move. PPO under calibrated noise is optimizing the same reward surface as PPO under placeholder noise, but through a different observation window. The resulting policy is a different policy, but the "correct behavior" it is being trained toward is unchanged.

This has two consequences worth preserving for future milestones:

1. **Any future milestone that modifies the reward function falls outside this protection.** Reward shaping, reward redesign, or reward-regularization changes must be integration-audited on a different basis. The M9 coherence argument does not transfer to those milestones.
2. **Any future milestone that edits `configs/default.yaml` for sensor or physics parameters inherits the config-coupling trap.** The M9.3 config isolation mechanism (or its equivalent) should become a pattern: when parameters flow into already-trained artifacts, edit through a dedicated config variant, not in-place.

---

## What this insight rules out

- The "sensor calibration might break the system in an unexpected way" concern — the analysis traced a seven-stage propagation chain and identified only quantitative, bounded shifts at each stage, plus the two coupling traps above. There is no hidden instability surface inside the simulation.
- The "reward misalignment under new signals" failure mode — structurally eliminated by the truth-derived reward.
- The "obs8 schema needs to change under calibration" concern — schema is shape-stable; only values flow differently.
- The "the 128/128 test suite will regress" concern — tests are structural, not distributional, and will not break on a config edit.

---

## What this insight does NOT rule out

- A narrowed (or reversed) +21.2% M8 obs8 gap under calibration. The analysis identifies a specific mechanism (`dp_fouling_gain` slow-correlation loss) that could cause narrowing. The direction of the calibrated result remains M9's primary open question and is the reason M9 exists.
- Misinterpretation of the calibrated result as "hardware-validated." The M9 prohibited-claims list must be authored before results are known, using the M8.3 Section 10 pattern.
- The research-access blocker from M9.1R.2 Section 0. Integration readiness does not resolve the need for a verified source pass.

## How to apply this insight

**For M9.3 (execution document):**

1. Specify config isolation: either (a) create `configs/m9_calibrated.yaml` read only by the M9 training script, leaving `default.yaml` unchanged, or (b) snapshot the pre-M9 config commit hash and require v1/v2 re-evaluation against that snapshot. Prefer (a).
2. Explicitly prohibit re-evaluation of `ppo_hydrienv_v1` or `ppo_hydrienv_v2` against the calibrated config using their original `_vecnorm.pkl`. State that any such evaluation is methodologically invalid and will not be reported.
3. Schedule the cross-schema sanity gate: re-run AC11 (expect 694.639) before config change, after config change, before training, after eval, and at closeout. Any deviation halts the pipeline.
4. Treat `dp_latency_steps` as a structural change, not a parameter tweak. Any latency change requires an environment revision note, not a config commit.
5. Confirm that `ConicalCascadeEnv` does or does not read `configs/default.yaml sensors.*` — one-file audit; if it does, extend the isolation mechanism to protect CCE.
6. Bound optical calibration to noise standard deviations only. Do not touch `turbidity_gain_clog`, `turbidity_gain_particles`, or equivalent gain coefficients — those enter the NTU-to-mass mapping flagged in M9.1R.2 §2.3 as experimentally-groundable-only.

**For M9.9 (closeout):**

- Record that M9 was able to proceed safely because reward is truth-derived. Flag this as a structural property of HydrOS for future milestone auditors.
- Note whether the `dp_fouling_gain` parameter remained legitimate after calibration or collapsed to a near-zero structural placeholder. Document the decision.

**For future milestones:**

- Reward-touching milestones must perform their own integration analysis on a different basis. The M9 coherence argument does not carry over.
- Physics-parameter milestones (M10+ domain randomization, DEP Q_crit, turbidity-to-mass experimental pass) inherit the config-coupling protection pattern.

---

## One-line takeaway

**M9 is structurally safe because the reward doesn't move when the sensors do — but the saved-artifact ecosystem coupled to `configs/default.yaml` still needs explicit isolation.**
