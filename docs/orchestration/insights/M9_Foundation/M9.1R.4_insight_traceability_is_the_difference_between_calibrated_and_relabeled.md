# M9.1R.4 Insight — Traceability Is the Difference Between Calibrated and Relabeled

## 1. Core Finding

The M9.1R.3 verified-source pass produced five calibrated values. The M9.1R.4
sources map documents the complete traceability chain for each: device class →
manufacturer specification → unit conversion → committed value. Without this
chain, the calibrated values are indistinguishable from new placeholders.

**The distinction matters because the M9 structural success metric is L2 → L3/L4
sensor-layer advancement.** L2 is "placeholder-bound" — values present but
unanchored. L3/L4 is "source-grounded" — values traceable to named external
sources. The traceability chain is what makes the difference. A value of 1.27e-6
is not inherently more grounded than 0.0005; it is more grounded because it
traces to Rosemount 3051 PDS → MEMS 0.5% FS/year spec → arithmetic conversion.

## 2. Why This Is Not Obvious

Calibration efforts naturally focus on the output: what is the right number?
The traceability work — documenting the chain from source to value — feels
like bookkeeping. But in the HydrOS context, where the M9.1R.1 insight
explicitly warned that replacing one set of unverified numbers with another
does not advance the sensor layer, the traceability chain is the mechanism
that prevents the warning from being violated.

A future reader encountering `dp_drift_rate_kPa_per_step = 1.27e-6` in
`configs/m9_calibrated.yaml` has two possible interpretations:
- "Someone changed the number" (relabeled)
- "This value traces to [source], converted via [method], at [confidence]" (calibrated)

The sources map ensures the second interpretation is available. Without it,
only the first is possible.

## 3. Carry-Forward for M9 Governance

1. **M9.3 must reference the sources map.** Every calibrated value in the
   execution document must carry its source ID (S1–S12) from this map.
2. **`ppo_hydrienv_v2_cal_meta.json` must include provenance fields.** Per
   the M9.2A §4.1 provenance framework, each parameter records: value,
   units, device_class, source_type, source_ref, confidence, assumptions,
   conversion_notes.
3. **M9.9 closeout must confirm that the traceability chain held.** If any
   value was changed between M9.3 and M9.7 (during execution), the chain
   must be updated to reflect the change and its reason.
4. **Future calibration milestones should produce a sources map as a
   standard artifact.** This is not AROD Schema v1 mandatory (it is a
   workflow refinement), but it should be adopted as standard practice for
   any milestone that changes numeric parameter values.

## 4. The 12-Source Coverage Is Sufficient for M9

The sources map covers 12 sources for 9 grounded parameters (5 M9 targets +
4 pre-grounded). Every parameter has at least one external source. Four of
five M9 targets have two or more independent sources. The one MEDIUM-confidence
parameter (`dp_fouling_gain`) has three sources that collectively bound its
value without precisely determining it — and the behavioral impact of the
remaining uncertainty is negligible at episode scale.

No parameter is ungrounded. No source is fabricated. The coverage is complete
for M9 scope and honest about what remains unresolved (optical gains, camera,
model-structure limitations).
