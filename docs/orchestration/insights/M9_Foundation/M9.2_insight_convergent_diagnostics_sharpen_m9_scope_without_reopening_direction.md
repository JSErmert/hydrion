# M9.2 Insight — Convergent Diagnostics Sharpen M9 Scope Without Reopening Direction

---

## 1. Core Finding

**Three independent diagnostic passes, run in parallel on different analytical lenses,
converged on the same conclusion about M9.** M9.1A scored system maturity and layer
alignment. M9.1B classified reality adherence at the file:line level across the
physics and sensor code. M9.1C traced a seven-stage propagation chain through the env,
sensor, obs, vecnorm, policy, reward, and eval pipeline. None of them referenced each
other's conclusions; none of them used the same vocabulary or evidence base. All three
arrived at **sensor calibration is the correct next move**, confirmed the M9.1 refinement,
surfaced zero critical contradictions, and produced compatible refinements that sharpen
the M9 scope without displacing its direction.

This convergence is the strongest pre-execution validation signal available for M9 short
of executing M9 itself. Single-analysis confidence can reflect a single framework's
blind spots. Three-analysis convergence — from distinct frameworks, on distinct evidence —
is structural agreement, not methodological echo.

---

## 2. What the Convergence Produced

The value of the convergence is not that it confirmed M9 — M9.0 refinement and M9.1A
had already done that. The value is the **refinements** that were only visible when
three different lenses met on the same system:

### 2.1 A narrower parameter scope

M9.1R.2 named three sensor parameters as calibration targets:
`dp_drift_rate`, `dp_fouling_gain`, `flow_calibration_bias_std`. M9.1B §5 established
at the code level that **`flow_calibration_bias_std` (code name `flow_bias_std_lmin`) is
already grounded** to the electromagnetic flow meter factory-calibration tolerance
spec. The research effort on flow can be lighter than the brief implied. Only two DP
parameters remain genuine placeholders. The effective M9 parameter scope is narrower
than any single analysis surfaced.

M9.1B also identified additional calibration-pending parameters in the **optical
sensor layer** (`turbidity_noise_std`, `scatter_noise_std`, `camera_noise_std`) that
M9.1R.2 deferred from scope. These become part of M9's parameter set. Net effect:
**one parameter removed from scope, three parameters added** — a more focused, more
specific target set.

### 2.2 A resolved semantic question

M9.1R.2 §3.2 flagged `dp_fouling_gain` as semantically ambiguous: process A
(filter-cake fouling already in truth state, which would mean the parameter
double-counts) or process B (sensor-side bias from line/diaphragm fouling, which would
be a legitimate sensor-error term). The research brief could not resolve this because
the question is not literature-answerable — it is a code-audit question.

M9.1C §2 Stage 1 and §3, tracing the sensor update code path in `pressure.py:99–102`,
**resolved the ambiguity as process B**: the implementation computes
`fouling_offset = dp_fouling_gain · mesh_loading_avg`, applied as an additive bias on
the sensor output, reading `mesh_loading_avg` from truth and writing only to
`sensor_state`. This is a legitimate sensor-error term, not a double-count. M9 keeps
the parameter; its calibrated magnitude is near-zero for episode timescales.

This resolution required the integration analysis to run in parallel with the research
brief. A sequential workflow (R → B → C) would have waited on R's ambiguity to block
M9.3 until C could inspect the code. The parallel workflow closed the question faster.

### 2.3 A structural coherence anchor

M9.1C §2 Stage 6 and §12 surfaced a finding that was not present in any prior artifact:
**the reward function is computed from `truth_state` alone (`env.py:304–308`) and is
therefore invariant under any sensor parameter change**. This single architectural
property rules out the entire category of "reward misalignment under new signals"
failure mode at the structure level rather than at the discipline level.

Prior M9 documents had implicitly assumed coherence without verifying its structural
source. M9.1C made the assumption explicit and established that coherence follows from
a specific property of HydrOS architecture (truth-derived reward) that can be cited,
relied upon, and carried forward to future reward-invariant milestones — and, just as
importantly, **must not be assumed to hold** for future reward-modifying milestones.

### 2.4 Two hidden coupling traps

M9.1C identified two failure modes that were not visible at the refinement level or
at the maturity-diagnostic level, because both require tracing the interaction between
the config file and the saved artifact ecosystem:

- **F1 (config coupling).** `configs/default.yaml` is read by v1, v2, and CCE eval
  paths. A naive in-place edit would silently break AC11 (694.639) and invalidate the
  M8 v2 placeholder-noise reference. Resolution: dedicated `configs/m9_calibrated.yaml`
  read only by the M9 training script.
- **F6 (stale vecnorm cross-eval).** The saved `*_vecnorm.pkl` running statistics
  carry pre-calibration distribution moments. Applying them to a calibrated env
  produces a hybrid number that is neither M8 nor M9. Resolution: explicit prohibition
  in M9.3, paired artifact loading pattern preserved.

Both traps are preventable by **structural** provisions in M9.3, not by vigilance
alone. This is a critical distinction: discipline-based protections can fail; structural
protections cannot. M9.3 must specify both protections as mandatory.

### 2.5 A state-separation violation surfaced

M9.1B §4.3–4.5 and §10 identified that `optical.py:144` writes sensor outputs back into
`truth_state["sensor_turbidity"]` etc. for compatibility with pre-M6.2B callers. This
is a **CLAUDE.md Core Rule #2 (state separation) violation currently active in code**,
annotated as temporary with no removal schedule. M9 is the natural place to remove it
as a zero-risk cleanup (the mirror is a no-op for both obs14_v1 and obs8 observation
assembly, so removing the write to `truth_state` does not change any observation
vector).

Neither M9.1A nor M9.1C surfaced this finding. It required the file-level reality
audit lens of M9.1B to catch.

### 2.6 An adjacent structural flag for M10+

M9.1B §6 surfaced the M4/M5 production/research physics bifurcation: the source-grounded
physics modules in `hydrion/physics/m5/*` are **not wired into the production loop**.
The training and evaluation environment still runs M4 phenomenology. This is the
next-most-important grounding gap after sensor calibration completes, but it is
**larger** than M9 and must not displace M9. It belongs in M10+ planning.

M9.2 records the flag; the M9.1B insight holds it as load-bearing for M10+ scoping.

## 3. Why the Convergence Was Not a Foregone Conclusion

It would have been plausible for the three parallel branches to disagree. Specifically:

- **M9.1B could have argued against M9 direction** by finding that the real priority
  is the M4/M5 integration gap, larger and structurally more important than sensor
  calibration. Instead, M9.1B §10 explicitly supports sensor calibration as the
  correct immediate move and flags the M5 integration gap as a post-M9 concern.
- **M9.1C could have found an integration-breaking risk** that required architectural
  change before M9 could proceed — for example, a reward-side coupling that would
  make calibration unsafe. Instead, M9.1C §6 found the opposite: reward is
  structurally calibration-invariant.
- **M9.1C could have identified contradictory evidence against M9.1A's L2 sensor
  maturity classification** by showing that the sensor layer is actually in better or
  worse shape than L2. Instead, it refined the classification: some parameters are
  already grounded (L3+), others are in worse shape than L2 implied (optical layer).
  The refinement is compatible with M9.1A, not contradictory.

That all three branches produced refinements rather than contradictions — and that
the refinements are mutually compatible — is not automatic. It reflects that HydrOS's
governance work (Contract v3.1, AROD Schema v1, prior M8-class closeout discipline)
has produced a system state where multiple analytical lenses see the same structural
facts. If they had produced contradictions, the synthesis would have had to reopen
the M9 direction. They did not. The direction stays.

## 4. What This Means for M9.3 Authoring

M9.3 now has a single consolidating reference document (M9.2) rather than five parallel
inputs. The execution-document author does not need to hold all of M9.1A, M9.1B,
M9.1C, and M9.1R.2 in mind simultaneously. The synthesis specifies:

- What M9.3 must contain (SP1–SP5, SA1–SA3, parameter scope, artifact versioning,
  prohibited claims, four-scenario classification, inheritance declarations)
- What M9.3 must preserve (protected lineage, schema discipline, physics layer A,
  reward function, test discipline, governance)
- What M9.3 must not do (author results before prohibited-claims list, modify
  protected lineage, expand governance scope, touch optical gain coefficients,
  edit `configs/default.yaml` in place)
- Which gates must close before M9.3 opens (G1 — verified source access, G2 — CCE
  config audit, G3 — M9 prohibited claims list authored at M9.3 opening)

This is a substantial reduction in cognitive load for the M9.3 author and a
corresponding reduction in the risk that some requirement gets dropped.

## 5. What This Insight Generalizes

The convergent-diagnostic pattern is reusable. For future HydrOS milestones that carry
significant risk or structural uncertainty, running **parallel** diagnostic passes
with different analytical lenses — maturity scoring, code-level reality audit,
integration propagation trace — is more informative than running a single deeper pass.
Specifically:

- A maturity/alignment pass alone would have missed the M4/M5 production/research
  split (M9.1A L4 score for physics was defensible but file-existence-level).
- A reality audit alone would have missed the reward-invariance coherence anchor
  (M9.1B did not trace code paths through the training loop).
- An integration analysis alone would have missed the file:line classification
  vocabulary that made the M4/M5 split visible (M9.1C worked at the subsystem level).

Each lens has blind spots that the other lenses fill. Running them in parallel, from
a shared starting point (M9.1 refinement + M9.1A diagnostic), produces a synthesis
that no single lens could have reached.

**Proposed practice:** For milestones with M9-scale structural uncertainty, schedule
three parallel diagnostic passes before the execution document opens. For smaller
milestones (parameter tweaks, documentation cleanups, additive extensions), the
parallel-pass overhead is probably not justified.

## 6. What This Insight Does NOT Say

- **It does not say M9.1R.2 was wasted work.** The preliminary research brief
  established the structural scope (sensor class mapping, noise-model shape,
  fundamental limits) that the subsequent passes built on. It also correctly
  identified the verified-source gate that still blocks M9.3. Without M9.1R.2,
  M9.1C would not have had a parameter-list anchor to trace.
- **It does not say parallel passes are always better than sequential.** Parallel
  runs amplify any premise error — if M9.1 refinement had chosen the wrong direction,
  all three passes would have confirmed it. Parallelism does not verify the starting
  point.
- **It does not declare M9 safe.** M9 remains gated on the verified-source pass
  (G1), the CCE config audit (G2), and the M9.3 prohibited-claims list (G3). The
  synthesis raises confidence in the direction, not in the readiness of all inputs.
- **It does not reduce the standing of M9.1R.1's scoping-pass insight.** The research
  brief is still a scoping pass, not a grounding pass. The numeric values remain
  preliminary. M9.3 cannot commit values drawn only from M9.1R.2.

## 7. Recommended Carry-Forward Actions

1. **Proceed to M9.3 only after G1 closes.** The verified-source gate is the binding
   precondition. M9.2 consolidation does not remove it.

2. **Complete G2 (CCE config-block audit) immediately.** This is a 10-minute
   single-file read of `conical_cascade_env.py`. Do it now so M9.3 has a clean start.
   Record the outcome in a short note appended to this synthesis or committed
   alongside it.

3. **Preserve the parallel-diagnostic pattern as a reusable practice.** Future
   milestones with comparable structural uncertainty should plan for three parallel
   passes (maturity, reality, integration) as a standard pre-execution phase.
   This is not an AROD Schema v1 revision — it is a workflow refinement that can be
   adopted per-milestone and promoted to schema v2 later if it proves consistently
   valuable.

4. **Hold the M4/M5 integration gap as the top M10 candidate.** M9.1B's insight
   names this as the next-most-important structural grounding gap. When M9 closes,
   the first M10 planning conversation should be about whether and how to integrate
   `hydrion/physics/m5/*` into the production loop — with explicit decisions about
   legacy lineage protection for v1, v2, v2_cal, and CCE.

5. **Record the reward-invariance property explicitly in M9.9 closeout and in any
   future reward-modifying milestone's risk analysis.** The property is now
   documented in M9.1C and M9.2 but has not yet reached the closeout or the future
   milestone planning. It should, because it is a durable structural fact about
   HydrOS that future reviewers will need to know before they scope reward changes.

6. **Do not author M9.3 until G1 closes.** Authoring M9.3 with preliminary values
   would commit the M9.1R.2 unverified ranges into an execution document — exactly
   the failure mode M9.1R.1's insight warned against. The synthesis is the last
   pre-execution document; M9.3 is the first execution document; the gap between
   them is the verified-source pass.
