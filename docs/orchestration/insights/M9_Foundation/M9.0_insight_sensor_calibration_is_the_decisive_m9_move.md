# M9.0 Insight — Sensor Calibration Is the Decisive M9 Move

---

## Part I — Strategic Milestone Learning

---

## 1. Core Conclusion

Sensor calibration is the decisive M9 move because it is the **prerequisite** for the
interpretability of every other candidate branch — not because it is the most technically
interesting option, and not merely because it scores highest on realism leverage.

The decisive property is structural: four of the six evaluated branches cannot produce
interpretable results until calibrated sensor noise parameters exist. Calibration is not
one option among six. It is the enabling condition for the other five.

---

## 2. Context

Produced during the M9.0 refinement pass. Six candidate branches were evaluated against
M8's three unresolved confounds (ranked primary/secondary/tertiary per AROD Schema v1
Section 12), each assessed on eleven criteria including realism leverage, confound
reduction, visual validity leverage, and risk of a misleading result.

Candidate branches evaluated: sensor calibration, domain randomization, step-matched v2
benchmark, distillation/teacher-student, obs4_sensor_only ablation, frontend/telemetry.

Governing inputs: M8_achievements.md, M8.9_closeout_record.md, M8.7_execution_report.md,
all M8 insight files, AROD Schema v1, Contract v3.1.

---

## 3. Prerequisite Dependency Map

**The deepest lesson of the M9.0 refinement pass is not "calibration is good."
It is that the order of operations determines whether downstream branches are interpretable at all.**

| Branch | Depends on calibration? | Reason |
|--------|------------------------|--------|
| Domain randomization | **Yes — cannot proceed without it** | Randomizing uncalibrated placeholder parameters samples from an arbitrary distribution, not a realistic operating envelope. Calibration defines the center point that makes range selection meaningful. |
| obs4 ablation | **Yes — result would be confounded** | A 4D sensor-only policy trained on placeholder noise is confounded in exactly the same way as the M8 result. Under calibrated noise, obs4 answers a precise, interpretable question: what is the actuator-channel contribution under realistic sensor conditions? Before calibration, the answer is not separable from noise-regime effects. |
| Frontend / telemetry | **Yes — would be actively misleading** | A polished visualization of placeholder-noise sensor traces looks scientifically grounded while displaying physically arbitrary numbers. The visual validity bridge surfaced this explicitly: the more polished the presentation, the worse the misleading effect. A frontend milestone before calibration violates the visual validity principle. |
| Distillation / teacher-student | **Yes — scope cannot be determined** | Whether distillation is warranted depends on whether the calibrated-noise gap narrows, holds, or reverses. If it narrows significantly, distillation becomes well-motivated. If it remains positive, the case weakens. The distillation question is ill-posed until M9 provides a more stable answer. |
| Step-matched v2 benchmark | **No — but does not address calibration** | The 500k-step rerun reduces the secondary confound (step-budget asymmetry) and is low-cost (one command, no architecture). However, a step-matched policy running on placeholder noise is no more hardware-interpretable than the 360k version. It improves benchmark precision; it does not improve interpretability for deployment purposes. |

**Conclusion from the map:** Calibration is a prerequisite for four branches and a
clarifying input for the fifth. It is not a high-scoring option — it is the enabler.
Any framework that scores branches on individual merit without mapping inter-branch
dependencies will systematically underweight prerequisite work.

---

## 4. Why Calibration Is Load-Bearing, Not Just Valuable

The M8 result (+21.2% deployment gap, v2 vs v1) is the foundation every subsequent
HydrOS deployment-pathway milestone rests on. Three non-exclusive explanations for the
positive direction remain live:

1. The obs8 schema carries genuinely better information content for the policy at this
   training budget.
2. Dimensional reduction (8D vs 14D) produces a simpler optimization landscape for PPO,
   independent of information content.
3. The placeholder noise regime happens to favor the obs8 policy's channel structure.

Explanation 3 cannot be ruled out without M9 calibration. Until it is ruled out, the
deployment gap result cannot be used as evidence for any deployment-relevant claim —
regardless of how strong the number looks. Calibration is the only move that converts
"internally valid simulation measurement" into "hardware-relevant deployment gap estimate."

## 5. Immediate System Implication

M9 is bounded and implementable. The system requires:

1. Research brief (`M9.1R.1`) — ground `dp_drift_rate`, `dp_fouling_gain`,
   `flow_calibration_bias_std` in literature or device datasheets
2. Config update with calibrated values
3. Retrain under calibrated noise (new artifact — M8 v2 is preserved as the
   placeholder-noise reference)
4. Evaluate calibrated gap against v1 baseline using the existing eval script
5. Record: did the gap persist, narrow, or reverse?

All RL infrastructure exists. M9 adds only calibrated parameter values and a retraining
pass. The research brief is the gating artifact.

---

## 6. Carry-Forward Rules (Milestone Learning)

These rules generalize from the M9.0 refinement pass:

**Rule 1 — Prerequisite analysis precedes merit ranking.**
Before scoring branches on individual value, map dependency relationships: which branches
depend on which others for interpretability? A branch that is a prerequisite for three or
four others should be prioritized regardless of its individual merit rank.

**Rule 2 — Benchmark precision ≠ interpretive validity.**
A more precise measurement of an uninterpretable result is still uninterpretable. The
step-matched rerun improves precision; calibration improves interpretability. These are
different contributions. When the choice is between them, prioritize the one that changes
what the result means, not the one that tightens the number.

**Rule 3 — Visual polish amplifies misleading effects.**
A polished visualization of scientifically ungrounded data is worse than a plainly
incomplete visualization because the polish provides false credibility. Frontend/visual
milestones should be scheduled after the underlying data reaches scientific grounding,
not before.

## 7. AROD Validation Note

**This pass is the first live use of AROD Schema v1 in a milestone refinement decision.**
The following records what the schema added, what it did not add, and what should change
in v2.

### AROD elements applied in this pass

| AROD Section | Element applied | How it was used |
|-------------|----------------|-----------------|
| Section 12 | Confound-priority ranking | Ranked M8's three unresolved confounds as primary / secondary / tertiary; used to drive the branch selection |
| Section 11 | Visual validity bridge | Required assessment of each branch against four visual-validity criteria (truth-vs-sensor fidelity, telemetry meaning, policy-view legitimacy, action-trace legitimacy) |
| Section 13 | System-model change requirement | Required stating what changed after M8, what M9 will clarify or correct, and what future milestones must no longer assume if M9 succeeds |
| Section 7 | Artifact identity schema | Populated milestone, iteration, phase, prompt/output/insight paths, governing docs, status, runtime class for the M9.1 artifact |

### How AROD improved the refinement

**Confound-priority ranking (Section 12)** prevented the secondary confound (step-budget
asymmetry) from appearing as equally urgent as the primary confound (sensor noise). Without
the ranking, a step-matched 500k v2 rerun is a plausible first move — it is concrete,
tractable, and produces a cleaner number. The ranking forced the question: does resolving
the secondary confound improve the interpretability of the M8 result for deployment
purposes? The answer is no. This distinction required explicit structure to surface.

**Visual validity bridge (Section 11)** converted the frontend/telemetry candidate from
"strategically aligned with console vision" to "actively misleading under placeholder
noise." This is a non-obvious conclusion. Without the visual validity criterion, the
frontend branch scores well on visual relevance and is easy to rationalize as high-value.
The bridge forced the additional question: does this display hardware-interpretable data?
The answer — not without calibration — is exactly what the criterion is designed to
surface. This is AROD providing governance-at-the-start rather than governance-after-
the-fact.

**System-model change requirement (Section 13)** converted a vague "M9 addresses noise"
statement into a bounded testable question: does the M8 deployment gap persist, narrow,
or reverse under calibrated sensor noise? This framing is the prerequisite for a
governance-safe M9.3 execution document. A refinement output that does not state a
testable question cannot produce binding acceptance criteria.

### Whether AROD added friction

No friction was experienced from the three elements applied above. Each required section
produced a concrete, decision-improving output that was not recoverable from narrative
reasoning alone.

One minor overlap was identified: the confound-priority ranking (Section 12) and the
system-model change requirement (Section 13) have partially overlapping scope in practice.
The primary confound often is the same thing as the most important system-model assumption
to correct. In this pass, both sections pointed to the same conclusion (sensor noise is
the binding constraint). This is not a problem — redundant structure that confirms the
same answer increases confidence — but it does create minor duplication in the refinement
output. A v2 revision could cross-reference rather than duplicate these two sections.

### Carry-forward decision

**Keep with minor refinement.**

AROD Schema v1 added measurable value in the first live pass. The schema should not be
revised before M9.1R.1 (the research brief). The overlap between Sections 12 and 13
is a v2 refinement target, not a v1 correction. After M9 closes, record a second AROD
validation data point in the M9 closeout insight and reassess whether the overlap
warrants a schema revision before M10.

The v2 refinement candidate: consolidate or explicitly cross-reference Sections 12 and
13 so the refinement output does not need to duplicate analysis to satisfy both.

## 8. Recommended Carry-Forward Actions

1. **Open M9.1R.1 (research brief) as the next artifact.** The governing question is
   bounded: what are the realistic noise characteristics for differential pressure
   transducers and flow meters at lab-scale or pilot-scale microplastic extraction?
   The answer determines calibrated values for `dp_drift_rate`, `dp_fouling_gain`,
   and `flow_calibration_bias_std`. Do not write an execution document without this.

2. **Record calibrated parameter provenance in M9.3 (execution document).** Each
   calibrated value must include: source (publication or datasheet), confidence level
   (directly measured vs estimated from analogous devices), and assumptions. Provenance
   is what distinguishes a calibrated result from a better-educated guess.

3. **Preserve the M8 placeholder-noise v2 as the pre-calibration reference.** Do not
   overwrite `ppo_hydrienv_v2.zip`. The M9 calibrated policy must be a distinct artifact.
   The placeholder-noise v2 vs calibrated-noise v2 comparison is itself informative —
   it quantifies how much the noise regime contributed to the M8 result.

4. **Apply prerequisite analysis explicitly in all future refinement passes.** Before
   scoring candidate branches on individual merit, map dependency relationships. A branch
   that is a prerequisite for three or four others should be prioritized accordingly.
   This step should be added as a standard check to future M*.0 refinement prompts.

5. **Record a second AROD validation point at M9 closeout.** This is the first data
   point in the AROD Schema v1 validation record. A second data point at M9 closeout
   (did the schema continue to add value through the research and execution phases?)
   is needed before any schema revision decision is made for v2.
