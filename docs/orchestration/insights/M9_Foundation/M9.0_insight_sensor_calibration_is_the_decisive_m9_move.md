# M9.0 Insight — Sensor Calibration Is the Decisive M9 Move

## 1. Title

Sensor calibration is the decisive M9 move because it is the prerequisite for the
interpretability of every other candidate branch — not because it is the most technically
interesting option.

## 2. Context

Produced during the M9.0 refinement pass. Derived from evaluating six candidate M9
branches against M8's three unresolved confounds, using AROD Schema v1's confound-priority
structure and visual validity bridge.

The refinement pass had six candidate directions: sensor calibration, domain randomization,
step-matched v2 benchmark, distillation/teacher-student, obs4_sensor_only ablation, and
frontend/telemetry. Each was assessed against eleven criteria including realism leverage,
confound reduction, visual validity leverage, and risk of producing a misleading result.

## 3. What Was Learned

**The order of operations matters more than the specific next choice.**

The most important finding from the M9.0 refinement pass is not "sensor calibration is
good" — it is that the other candidate branches all depend on calibration for their
interpretability or safety, and none of them resolve the primary confound. Specifically:

- **Domain randomization** varies sensor noise over ranges. If those ranges are centered
  on placeholder values, the randomization is sampling from an arbitrary distribution
  rather than a realistic operating envelope. Calibration defines the center point that
  domain randomization needs to be meaningful.

- **obs4 ablation** (sensor-only, 4D) would produce a result confounded by placeholder
  noise in exactly the same way as the M8 result is confounded. An obs4 policy trained
  on calibrated noise answers a different and more interpretable question: what is the
  policy contribution of actuator command channels under realistic sensor conditions?

- **Frontend / telemetry milestone** would display sensor traces that reflect placeholder
  noise behavior. The visual console would appear scientifically grounded (it would have
  real-time sensor rendering, comparison traces, policy-view overlays) while showing
  physically arbitrary numbers. This is more misleading than a plainly incomplete
  visualization — the polished presentation amplifies the scientific vacuousness of the
  underlying data. The visual validity bridge evaluation surfaced this explicitly.

- **Step-matched v2 benchmark** resolves the secondary confound (step-budget asymmetry)
  but not the primary confound (sensor noise). A step-matched 500k v2 policy is still a
  placeholder-noise policy. Knowing whether the gap is +22% or +19% at step-matched
  budgets does not change the hardware-interpretability of the result.

- **Distillation / teacher-student** explores a methodology refinement but is premature
  without knowing whether the deployment-realistic result survives calibration. If the
  gap narrows significantly under calibrated noise, the case for distillation strengthens;
  if it remains positive, it weakens. The distillation question cannot be well-scoped
  until M9 calibration provides a more stable answer.

**Calibration is not the most exciting branch. It is the most load-bearing one.**

The M8 result (+21.2%) is the foundation all subsequent HydrOS deployment-pathway work
rests on. If that result changes meaningfully under calibrated sensor noise, the strategy
for domain randomization, reward alignment, obs4 ablation, and frontend rendering all
change accordingly. If it survives, those directions inherit a stronger, more credible
base. In either case, M9 calibration is the prerequisite pass.

This is a general pattern worth carrying forward: when multiple viable directions exist
and one of them is a prerequisite for the interpretability of the others, do that one
first — not because it is most valuable in isolation, but because it enables the value
of everything else.

## 4. Why It Matters

The governance structure (AROD Schema v1, visual validity bridge, confound-priority
ranking) made this conclusion unavoidable rather than implicit. Without explicit
confound-priority ranking, the secondary confound (step-matched benchmark) is a
plausible-looking first move — it is concrete, tractable, and produces a cleaner number.
The ranking mechanism forced the question: does resolving the secondary confound improve
the interpretability of the M8 result for deployment purposes? Answer: no. It only
improves its benchmark precision under placeholder noise conditions. This distinction is
not obvious from narrative reasoning alone; it required explicit ordering.

Similarly, the visual validity bridge prevented the frontend/telemetry candidate from
being assessed as "strategically aligned with the console vision" without the additional
question: would this display hardware-interpretable data? The answer — not without
calibration — is what the visual validity criterion was designed to surface.

## 5. Immediate System Implication

The M9.0 refinement pass is closed with a decisive choice: sensor realism and calibration
is M9. The system now has a bounded, implementable milestone definition:

1. Research brief grounding dp and flow sensor noise parameters in literature/datasheets
2. Config update with calibrated values
3. Retrain ppo_hydrienv_v2 under calibrated noise (new artifact)
4. Evaluate calibrated gap against v1 baseline
5. Record: did the gap persist, narrow, or reverse?

The research brief (M9.1R.1) is the next document. All other infrastructure exists.

## 6. Workflow Implication

**Prerequisite analysis should be explicit in every refinement pass.**

The M9.0 refinement identified that four of the six candidate branches have a dependency
on sensor calibration — yet calibration is listed as only one option among six. Without
explicit dependency analysis, that dependency is easy to miss. Future refinement passes
should include a "dependency graph" check: for each candidate branch, which other
candidate branches does it depend on, and does any branch unlock or enable the others?

If a branch is a prerequisite for three or four others, it is almost certainly the right
first move regardless of its individual merit ranking.

**The visual validity bridge prevented a category error.**

A frontend milestone without calibrated data would have violated the visual validity
principle — not just been premature, but actively misleading. Catching this at the
refinement stage (before any architecture or implementation work) is exactly the function
the visual validity bridge is designed to serve. This is AROD Schema v1 working as
intended: governance-at-the-start rather than governance-after-the-fact.

## 7. Recommended Carry-Forward Action

1. **Open M9.1R.1 (research brief) immediately.** The governing question for the brief
   is bounded: what are the realistic noise characteristics for differential pressure
   transducers and flow meters appropriate to lab-scale or pilot-scale microplastic
   extraction? The answer determines calibrated values for `dp_drift_rate`,
   `dp_fouling_gain`, and `flow_calibration_bias_std`.

2. **Record calibrated parameter provenance explicitly.** When M9.3 (execution document)
   is written, each calibrated parameter value must include: source (publication or
   datasheet), confidence level (directly measured vs estimated from analogous devices),
   and any assumptions. This provenance record is what distinguishes a calibrated result
   from a better-educated guess.

3. **Preserve the M8 placeholder-noise v2 as the pre-calibration reference.** Do not
   overwrite `ppo_hydrienv_v2.zip`. The M9 calibrated policy must be a distinct artifact.
   The comparison between placeholder-noise v2 and calibrated-noise v2 is itself
   informative — it quantifies how much the noise regime affected the M8 result.

4. **Apply the prerequisite analysis pattern in future refinement passes.** Before
   scoring candidate branches independently, map their dependencies. A branch that is
   a prerequisite for others should be weighted accordingly, not just ranked by its
   individual merit score.
