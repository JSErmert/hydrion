# M9.1R.1 Insight — The Research Brief Is a Scoping Pass, Not a Grounding Pass

---

## 1. Core Finding

**A research brief written without verified peer-reviewed source access is a scoping
pass, not a grounding pass.** It can structure the calibration work, map sensor classes
to HydrOS channels, identify noise-model semantics, and flag which questions can and
cannot be resolved by literature. It **cannot** produce defensible numeric values that
M9.3 (execution document) can commit to.

The distinction matters because M9's structural success metric is the advancement of
the sensor layer from L2 (placeholder-bound) to L3 or L4 (grounded). A brief that
looks complete but is based on unverified training knowledge does not lift the sensor
layer — it replaces one set of placeholders with a more confident-looking set of
placeholders. This is worse than the starting condition because it obscures the
placeholder status with a research veneer.

---

## 2. Why This Insight Surfaced

The standing HydrOS research access rule (user memory) requires explicit permission
before training knowledge is used to set physics constants. In this research pass,
such access was not available. The prompt simultaneously required (a) grounded
parameter values, (b) source citations, (c) autonomous execution, and (d) the
explicit instruction "do not invent citations."

The only path consistent with all four constraints was:

1. Produce a brief that maps sensor classes to HydrOS channels (structural scoping)
2. Identify noise-model semantics and flag semantic questions (interpretation)
3. Provide preliminary engineering-typical ranges, every one labeled LOW confidence
4. Explicitly gate M9.3 on a subsequent verified-source pass (M9.1R.3)
5. Not fabricate any specific paper title, DOI, or datasheet part number

Executing this path produced the insight: **the brief divides cleanly into two
types of content — structural scoping (safe and complete) and numeric grounding
(not completable without verified sources)**. The former is useful immediately.
The latter must be deferred.

---

## 3. What the Brief Can Do

### Structural contributions (complete)

- **Sensor class mapping.** dp_sensor_kPa → piezoresistive dp transducer.
  flow_sensor_lmin → one of several flow meter classes (coriolis, ultrasonic,
  electromagnetic, thermal mass, turbine, paddle). sensor_turbidity/sensor_scatter →
  nephelometric optical sensor (ISO 7027 class).

- **Noise-model semantic clarification.** `dp_drift_rate` maps to long-term zero
  drift. `flow_calibration_bias_std` maps to calibration-curve uncertainty.
  `dp_fouling_gain` is **semantically ambiguous** — could be filter-element fouling
  (process A, already in truth state) or sensor-line fouling (process B, a real
  sensor error). This ambiguity is a code-audit question, not a literature question.

- **Fundamental-limit identification.** The NTU-to-microplastic-mass mapping cannot
  be grounded by any literature pass. It requires direct experimental correlation
  against the target particle population. This is a pre-existing HydrOS architectural
  decision (turbidity as proxy) that should be flagged as calibration-pending-by-
  experiment, not calibration-pending-by-literature.

- **Implementation structure.** M9.3 can specify the calibrated-config schema, the
  artifact versioning plan (new `ppo_hydrienv_v2_cal` or `ppo_hydrienv_v3` canonical
  artifact, placeholder-noise v2 preserved), and the metadata provenance fields
  before any specific values are committed.

### Numeric contributions (NOT complete)

- Preliminary engineering-typical ranges for `dp_drift_rate`, `dp_fouling_gain`,
  `flow_calibration_bias_std`, and turbidity noise — every value labeled LOW
  confidence, every source labeled UNVERIFIED.
- No specific paper citations, DOIs, or datasheet part numbers.
- M9.3 must not use any value from the preliminary pass without verification.

## 4. The Two-Phase Research Pattern

This insight suggests a generalizable pattern for any future HydrOS research pass
that runs without verified source access at the outset:

**Phase 1 — Scoping brief (this pass).**
Structures the work, maps concepts to system artifacts, identifies semantic ambiguities,
flags fundamental limits, specifies the implementation structure. Outputs ranges with
LOW confidence labels. Does NOT fabricate citations. Does NOT commit values.

**Phase 2 — Verified source pass (M9.1R.3 or equivalent).**
Runs only after verified source access is granted (peer-reviewed literature OR
datasheet nomination OR both). Replaces preliminary ranges with verified values.
Supplies explicit citations. Produces the brief that M9.3 can commit to.

Phase 1 is useful because it makes Phase 2 fast — the structural work is already
done, so Phase 2 only needs to supply numbers. Phase 1 is useful because it makes
Phase 2 **necessary** — the preliminary ranges are explicitly labeled as unverified,
so M9.3 cannot proceed without the verification step.

Phase 1 is **misleading** if its preliminary ranges leak into an execution document.
The structural value is honest; the numeric value is conditional and must remain so.

---

## 5. Why the Research Access Rule Is Binding, Not Preferential

The user's standing rule ("always ask before using training knowledge for physics
constants") is often read as a process preference — "please ask me first." That
reading is incomplete. The rule encodes a substantive constraint: **unverified
training knowledge is not a safe substitute for peer-reviewed sources when the
values will be used to calibrate a research-grade simulation.**

In the HydrOS context, with M9 specifically designed to lift the sensor layer from
L2 to L3/L4, this is not a courtesy question. It is a load-bearing epistemic
constraint. A calibration pass that replaces placeholders with unverified training
knowledge does not advance the sensor layer. It only renames the problem.

The rule is therefore binding on M9 execution. M9.3 execution document cannot open
without either:

- (a) A successful verified-source pass, or
- (b) Explicit user acknowledgement that unverified training-knowledge values will
  be committed with LOW confidence labels (which would preserve the L2 status)

Option (b) is not an M9-success outcome. Only option (a) is.

---

## 6. Recommended Carry-Forward Actions

1. **Surface the research access gate to the user immediately.** This insight, and
   the M9.1R.2 brief's Section 10 (Critical Status Summary), must be presented as
   a decision point: grant source access, nominate device classes, or accept that
   M9 will not advance the sensor layer.

2. **Begin the M9.1B code audit in parallel.** The `dp_fouling_gain` semantics
   question does not require literature — it requires inspection of
   `sensor_fusion.py` and `init_sensor_state` in the repo. This work can proceed
   immediately and will prepare M9.3 regardless of when source access is resolved.

3. **Record the two-phase research pattern as a reusable orchestration practice.**
   Any future HydrOS milestone that requires research-grounded parameter values
   should explicitly plan for a scoping brief followed by a verified source pass,
   rather than attempting to combine them in one artifact. This is not an AROD
   schema revision — it is a per-milestone workflow refinement.

4. **Preserve the LOW-confidence labeling discipline across all future research
   briefs.** The current brief's value lies substantially in its explicit
   separation of structural (safe) content from numeric (unverified) content.
   This separation must not be blurred in subsequent revisions. If a future pass
   adds verified sources, it should update individual parameter confidence levels
   one at a time with explicit provenance, not sweep-update the entire brief.

5. **Do not allow M9.3 to commit any value from the preliminary brief.** This is
   the most important enforcement point. Every value in M9.1R.2 Section 3 carries
   the LOW label; every value in M9.3 must carry HIGH or MEDIUM confidence based
   on a verified source. The gap between M9.1R.2 and M9.3 is the verified-source
   pass — it is not optional, and it is not compressible.

---

## 7. What This Insight Is NOT

**This insight is NOT:**
- A criticism of the M9 direction (M9.0 refinement and M9.1A diagnostic both confirmed
  sensor calibration as the correct next move)
- A suggestion to delay or defer M9 (M9 should proceed; the research access gate is
  the next action, not an excuse for inaction)
- A claim that training knowledge is worthless (the structural contributions in the
  brief are genuinely useful and safe)
- A request for a schema revision (AROD Schema v1 remains reference-first; this
  insight informs M9 execution, not v2 development)

**This insight IS:**
- A structural observation about what a research brief can and cannot accomplish
  under different source-access conditions
- A gating action: M9.3 cannot open without verified sources
- A generalizable two-phase pattern for future research passes
- A reinforcement of the user's standing research access rule as load-bearing

## 8. Relationship to M9.1A Insights

This insight extends, not contradicts, the three M9.1A insights:

- **M9.0 revised insight (sensor calibration is the decisive M9 move):** confirmed.
  The research access gate does not change the direction; it clarifies the
  prerequisite.
- **M9.1A insight (top-heavy system, sensor layer load-bearing):** confirmed. The
  sensor layer is still the weakest high-leverage layer; M9 is still the structurally
  correct pass. But the prerequisite for actually advancing the sensor layer is
  verified source access, not just running a calibration script.
- **M9.1A final insight (imbalanced, not incomplete):** confirmed. The correction
  is still structural (lift the sensor layer), not feature-additive. The
  verified-source requirement is part of what "structural correction" means in this
  context — it distinguishes real grounding from cosmetic grounding.

All three prior insights remain in force. This insight adds a precondition to
their execution: **verified source access is the gate through which the M9
structural advancement must pass.**
