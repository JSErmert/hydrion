# M9.1B Insight — M5 grounding is not yet wired into the production loop

**Source audit:** `docs/orchestration/M9_Foundation/M9.1B.2_system_reality_adherence_audit.md`
**Date:** 2026-04-15
**Milestone:** M9_Foundation
**Iteration:** M9.1B
**Insight class:** structural reality finding (not a milestone pivot)

---

## The insight

> HydrOS has **two physics layers**. The production simulation loop (`hydraulics.py`, `clogging.py`, `electrostatics.py`, `particles.py`) is **phenomenological — internal-assumption-class** under the M9.1B classification system. The source-grounded physics — `hydrion/physics/m5/*` (Rajagopalan–Tien 1976 capture, Clausius–Mossotti, peer-reviewed nDEP constants for PP/PE/PET, CODATA fundamental constants) — is **the most physically rigorous code in the repo and it is not on the execution path**. The M5 grounding pass produced source-grounded physics modules and a source-grounded physics-basis document; it did not wire those modules into `HydrionEnv`. Every canonical policy (v1, v2, CCE), every reward gradient, every `capture_eff` trajectory, every `E_capture_gain` trace is Layer A. The +21.2% M8 deployment gap is a Layer-A result observed through partially-grounded sensors. The external story "HydrOS is grounded in peer-reviewed physics" is truthful about Layer B and imprecise about the running simulation.

---

## Why this is surprising relative to M9.1A

M9.1A scored the physical-truth layer at **L4 (Structured)** with justification: "M5 grounded the core physics in peer-reviewed sources … Hydraulics, clogging, backflush, electrostatics, particles are in place per the realism roadmap. The physics model is documented, source-traceable, and reference-grounded for its own domain."

At the **file-existence level** this is correct. At the **execution-path level** it conceals a split:

- The peer-reviewed constants live in `hydrion/physics/m5/materials.py`, `m5/dep_ndep.py`, `m5/capture_rt.py`.
- The running simulation imports `hydrion/physics/hydraulics.py`, `clogging.py`, `electrostatics.py`, `particles.py`.
- `particles.py` does **not** import `m5/capture_rt.py`. `electrostatics.py` does **not** import `m5/dep_ndep.py`.
- M5 documents the correct physics for three polymers in water (negative DEP: particles repelled from the high-field region toward the cone apex). M4 `electrostatics.py` still computes a capture gain that rises with `|E|` and residence time, i.e., a pDEP-shaped response. These are physically inconsistent models.

M9.1A's L4 score is correct for Layer B (research code) and optimistic for Layer A (production code). The audit does not suggest re-scoring M9.1A — maturity was assessed under a different lens and produced a defensible top-line verdict. It suggests that **when the M9.1A score is cited going forward, the two-layer structure should be cited with it**.

---

## What is dangerous about this

1. **The governance layer names M5 as grounding.** Every milestone artifact that references "M5 physics grounding" is referencing Layer B. Every milestone artifact that reports a trained policy result is reporting Layer A. A reader connecting those two references reasonably infers that the trained policy was trained against grounded physics. It was not.
2. **The M9 sensor calibration effort will grounds sensors on top of Layer A.** When M9 closes, HydrOS will have grounded DP and flow sensors observing an un-grounded `capture_eff` curve, an un-grounded `E_capture_gain`, and an un-grounded `dep_rate_s*`. That is a half-step toward reality, not a full step.
3. **The correct M5 physics (nDEP, RT filtration) would change control trajectories.** Under nDEP, the optimal control is not "maximize field magnitude everywhere" — it is "shape the field so particles drift to the apex." Under RT filtration, the optimal capture is not a fixed `(d_p/d_ref)^exp` curve — it has explicit Péclet and gravity terms that make the optimal size-band different per stage. A policy trained on Layer A and then deployed against a Layer-B-behaving real filter will not only lose efficiency — it may pursue an actively wrong strategy.

## What this insight does NOT say

- It does not say M9 direction is wrong. M9 sensor calibration is still the correct immediate next move: it is smaller, faster, higher-leverage-per-effort, and the binding constraint on M8 interpretability.
- It does not say M5 code is wrong. M5 is the highest-quality code in the repo by grounding standards.
- It does not say any existing artifact must be retracted. The M5 physics basis is accurate. The M8 result is internally valid. The M9.1A maturity assessment is defensible at its top-line score.
- It does not say "Layer A is broken." Layer A is functioning, tested (128/128), and stable. It is not grounded, but it is not wrong by construction — just by physics.

## What it does say

- When M9 closes, the next structural priority for reality adherence is **M5 integration**, not another sensor-calibration pass or a domain-randomization sweep.
- The M10 planning conversation should include an explicit decision about when and how M5 capture and nDEP physics become the production model — and specifically, how the existing canonical policies (v1, v2, CCE) transition to a Layer-B engine (do they retrain? do they become historical? does AROD Schema v1 protect them as legacy lineage?).
- Any milestone that references "HydrOS is grounded in peer-reviewed physics" should add one of two qualifiers: (a) "via the M5 reference implementation in `hydrion/physics/m5/`", or (b) "for the sensor and physics-basis layers; the production simulation still runs M4 phenomenology pending M5 integration."

## Smallest immediate action

No code changes are implied by this insight for M9. One documentation action is appropriate:

- **Add a one-line note to `M5_PHYSICS_BASIS.md`** stating that the module is a reference implementation and that integration into `HydrionEnv` is scheduled for a future milestone (TBD). This closes the credibility-halo risk without changing any physics or any milestone.

The insight itself is the load-bearing artifact. The M10+ planning conversation is where it gets converted to action.

## AROD validation note

This insight exists because the M9.1B audit applied **source-grounded / derived-grounded / internal-assumption / placeholder / unknown** classification at the **file:line** level rather than at the **subsystem** level. A subsystem-level classification would have categorized "electrostatics" as partially grounded and moved on. The file-level pass forced a distinction between `electrostatics.py` (internal assumption) and `m5/dep_ndep.py` (source-grounded), which made the split visible. This is an AROD Schema v1 benefit: the classification vocabulary is fine-grained enough to catch structural bifurcations that a narrative pass would smooth over.
