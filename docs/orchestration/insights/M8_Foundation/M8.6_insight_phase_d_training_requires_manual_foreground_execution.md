# M8.6 Insight — Phase D Training Requires Manual Foreground Execution

## Execution Summary

Phases A–C complete. 128/128 tests pass. All code artifacts committed. Training (Phase D)
cannot be executed autonomously due to a platform constraint: the execution environment
kills background processes at approximately 5 minutes, and training requires ~15–16 minutes.

---

## Finding 1: The Background Process Kill Constraint Is the Hard Blocker

`python -m hydrion.train_ppo_hydrienv_v2` requires ~15–16 minutes. Both available
execution paths fail:
- `run_in_background=True` on the Bash tool: environment kills background processes at ~5 min
- Foreground Bash with 600s timeout: 10-minute ceiling, insufficient

This is not a code problem. All code is implemented and tested correctly. The blocker is
purely environmental. The fix is manual foreground execution by the user via:

```
! python -m hydrion.train_ppo_hydrienv_v2
```

**Implication for future training milestones:** Any training run exceeding ~8–9 minutes must be
flagged as a manual step in the implementation plan before execution begins. M9+ training
scripts should default to `--steps 10000` (proof-of-concept mode) for autonomous validation,
with full-budget runs documented as manual foreground steps from the start.

## Finding 2: The Additive Schema Pattern Held at Full Integration Scale

The `obs_schema` parameter and `_observe()` dispatch implemented in M8 added two schemas
(obs14_v1 default, obs8_deployment_v1) with 17 tests and zero regressions against the 111
existing tests. The pattern is:

- New function (`build_observation_obs8`) is additive — existing `build_observation` is untouched
- Schema parameter defaults to `"obs14_v1"` — all existing callers see identical behavior
- `observation_space.shape` is now parameterized by `_obs_dim` — schema contract is
  structurally enforced before any model or VecNormalize object is created

This pattern is extensible: a third schema (e.g., `obs6_minimal_v1`) requires only a new
`build_observation_*` function and a new entry in `_VALID_SCHEMAS`. No existing path changes.

## Finding 3: The Two-Checkpoint Test Protocol Caught Nothing — That Is the Signal

Both M8.5 Section 10 checkpoints passed cleanly:

- Checkpoint 1 (after F1/F2 — sensor_fusion + env.py changes, before scripts): 111 passed

Zero regressions at either checkpoint. This confirms:
- obs14_v1 default behavior was not disturbed by the obs_schema parameter addition

## Finding 4: AC11 Is the Load-Bearing Acceptance Criterion

AC11 requires that ppo_hydrienv_v1 (obs14_v1) returns within 5% of its M7 canonical value
(694.639) when re-evaluated by the M8 eval script. If AC11 fails, all v2-vs-v1 comparison
results must be halted — the comparison baseline is invalid.

The M8 eval script enforces this as a printed halt warning:

```
[AC11 WARN] v1 return deviates more than 5% from canonical 694.639.
HALT: Do not interpret v2-vs-v1 delta until deviation is investigated.
```

The v1 environment path in eval is: fresh `HydrionEnv(obs_schema="obs14_v1")`, loaded from
`models/ppo_hydrienv_v1_vecnorm.pkl`, identical `ShieldedEnv` config. If AC11 fails, likely
causes are: VecNorm statistics mismatch, ShieldedEnv config drift, seed handling difference,
or noise model change since M7. All are diagnosable from the eval output.

---

## Finding 5: The Prohibited-Claims List Is an Operational Protocol, Not a Disclaimer

M8.3 Section 10 defines six claims that cannot be made regardless of evaluation outcome.
The eval script prints them at every run header. This is not boilerplate — it is an
operational control. When a numerical result (e.g., v2 ≈ v1 within 2%) appears in output,
the natural language description "the deployment schema performs equivalently to the full
schema" would be factually misleading without the prohibited-claim constraints.

The four-scenario classification table (M8.3 Section 9) provides the binding interpretation
for each numerical outcome range. The eval script dispatches to the correct scenario
automatically and prints the must-not-conclude statement. This removes interpretation ambiguity
at the point of result consumption.
