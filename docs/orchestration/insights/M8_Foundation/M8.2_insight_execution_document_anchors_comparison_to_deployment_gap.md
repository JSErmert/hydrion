# M8.2 Insight — The Execution Document Anchors the M8 Comparison as a Deployment-Gap Measurement, Not a Policy Quality Comparison

## 1. Title

M8.3 establishes a precise, bounded claim structure for the `ppo_hydrienv_v2` vs
`ppo_hydrienv_v1` comparison. The most important architectural decision in the execution
document is the insistence that the comparison is a deployment-gap measurement — a
cross-schema instrument — not a quality ordering between two policies. This framing
determines what M8 can and cannot prove regardless of outcome.

## 2. Context

Produced during M8.2 execution document pass (M8.3_execution_document.md). Integrates
governing inputs: M8.1 refinement output, M8.1R.2 research brief, M8.1R.4 sources map,
and all M8.0–M8.1R.3 insights.

M8.3 is the authoritative execution document. M8.4 (implementation plan) will inherit
its ACs and methodology validity conditions directly.

## 3. What Was Learned

**The four-scenario table is the most load-bearing element of M8.3:**
The result interpretation table (Section 9) defines, in advance, what each possible
outcome proves and does not prove. Its purpose is to prevent the most common
failure mode in RL ablation reporting: the researchers interpret the result through
their prior expectations rather than through what the methodology can actually support.
The table must be reproduced verbatim in the closeout record — it should not be
reconstructed from memory during reporting.

**The methodology validity conditions are not style — they are preconditions for result
validity:**
MVC-1 (VecNorm refit) and MVC-2 (matched conditions) are the technical preconditions
for the comparison to be interpretable. If either is violated, the performance delta
between v2 and v1 could be explained by normalization mismatch or hyperparameter
difference rather than the observation-space change. Enforcing them at the implementation
level (schema lock assertion, separate VecNorm pkl paths) is the implementation plan's
responsibility, but M8.3 establishes them as binding requirements.

**The three-class channel taxonomy is now locked into the execution document:**
physics_truth (0–5) / actuator_feedback (6–9) / sensor_derived (10–13) appears in
Section 2.2 of M8.3 as the canonical taxonomy with a formal table. It is no longer a
recommended correction to M7 metadata — it is the normative classification for all M8+
work. Future milestones that reference obs14_v1 must use this taxonomy.

**obs8_deployment_v1 is not sensor-only — this is a prohibited mischaracterization:**
Section 10 explicitly prohibits the claim "obs8_deployment_v1 is sensor-only control."
This matters because the interpretation of a near-equivalent M8 result changes
significantly depending on which framing is used:
- If obs8 is sensor-only: the result suggests sensors are sufficient for control.
- If obs8 is deployment-realistic (actuator feedback + sensors): the result suggests
  deployment-available signals together are sufficient — a weaker but more honest claim.
The prohibited-claims section enforces the honest framing.

**The large-gap investigation trigger is a required protocol step, not optional:**
Section 9 specifies that if a large performance gap is observed (> ~20%), three
alternative explanations must be evaluated before concluding channel insufficiency:
training budget extension, distillation from v1, calibration noise quality. This
prevents a failure result from being misread as evidence that deployment is impossible.

## 4. Why It Matters

M8 is the first result in HydrOS that directly bears on deployment viability. The
natural temptation when a deployment-realistic policy underperforms is to interpret the
gap as evidence that deployment is hard or impossible. The natural temptation when it
performs well is to interpret it as deployment-ready.

Both interpretations are wrong for this experiment. The execution document builds the
interpretation rules in advance so neither error can be committed under time pressure
during reporting.

The explicit prohibited-claims section (Section 10) is unusual for an execution document
but is necessary here: the stakes of misinterpreting a deployment-gap result in a
hardware-forward system are high. Overclaiming (deployment-ready) wastes development
effort on a policy that isn't. Underclaiming (deployment impossible) abandons a direction
that may require only more training or distillation.

## 5. Immediate System Implication

M8.4 implementation plan must:
1. Implement MVC-1 at the code level: training script initializes fresh VecNormalize,
   eval script uses separate load paths per policy. Include a runtime assertion that
   verifies VecNorm dimension matches the observation space shape before training.
2. Implement MVC-2 at the code level: define a shared config that ppo_hydrienv_v1 and
   ppo_hydrienv_v2 training scripts both import, asserting identical PPO hyperparameters,
   episode length, and ShieldedEnv parameters. Changes to this config affect both.
3. Implement AC2 at the code level: add a `OBS_DIM = 8` constant and assert
   `env.observation_space.shape == (OBS_DIM,)` at training script startup.
4. Implement AC12 at the code level: add a `tests/test_obs8_schema.py` that validates
   obs8_deployment_v1 shape, channel order, dtype, and value range.
5. Reproduce the four-scenario interpretation table verbatim in the M8 closeout record
   (do not rewrite or paraphrase it during reporting).

## 6. Workflow Implication

**M8.3 is the authoritative interpretation contract for M8 results.** Any future
milestone document (M9+) that references M8 evaluation results should cite M8.3 Section 9
(interpretation table) and Section 10 (prohibited claims) — not just the numbers.

**Benchmark lineage is now explicitly governed:**
The relationship between ppo_hydrienv_v1 (full schema baseline) and ppo_hydrienv_v2
(deployment-gap comparison artifact) is defined in M8.3 Section 4.3. Future milestones
that introduce additional policies must decide which schema and comparison role they
occupy. The benchmark lineage convention is: version number (v1, v2, ...) tracks
canonical HydrionEnv policies; schema name tracks the observation regime. These are
independent dimensions.

**Distillation path is formally named:**
If M8 shows a large gap, M8.3 Section 9 and M8.1R.4 Section 6 together specify the
Lee et al. 2020 DAgger-based distillation path as the recommended M9 response. This path
uses ppo_hydrienv_v1 as the teacher and trains a student on obs8_deployment_v1 via
imitation. The teacher artifact already exists.
