# M8.4 Insight — The Additive Schema Selector Pattern Is the Narrowest Correct Implementation Path for obs8_deployment_v1

## 1. Title

The M8.5 implementation plan resolves the obs8_deployment_v1 implementation problem by
adding a parallel `build_observation_obs8()` function and an `obs_schema` parameter to
`HydrionEnv` — neither modifying existing logic nor requiring a separate environment class.
This additive pattern is the implementation-level realization of M8.3's requirement to
preserve benchmark lineage while introducing the deployment-realistic schema.

## 2. Context

Produced during M8.4 implementation plan pass (M8.5_implementation_plan.md). Inherits all
governing inputs from M8.3: ACs, methodology validity conditions, interpretation rules,
and protected artifact list. M8.5 is the final document in the M8 planning chain; it
transitions M8 from design to execution-ready state.

Three implementation approaches were evaluated:
- (A) Additive function + parametric env parameter — chosen
- (B) Separate HydrionEnv subclass — rejected (physics pipeline duplication)
- (C) VecNormalize-level obs slicing wrapper — rejected (VecNorm normalizes 14D first, schema semantics are wrong)

## 3. What Was Learned

**The additive pattern satisfies all constraints simultaneously:**
The choice to add `build_observation_obs8()` as a parallel function rather than modifying
`build_observation()` achieves four things at once:
1. obs14_v1 behavior is unchanged — `build_observation()` is untouched, zero arguments added.
2. All 111 existing tests continue to pass — `HydrionEnv()` defaults to `obs_schema="obs14_v1"`.
3. Schema drift is detectable at the implementation boundary — `_VALID_SCHEMAS` dict
   raises `ValueError` for unknown names; training script asserts shape before any training.
4. Future schemas (obs10, obs12_deployment, etc.) can be added without modifying existing
   code — they require only a new function in sensor_fusion.py and a new dict entry in env.py.

**The VecNormalize path separation is the most operationally fragile point:**
The schema lock assertion pattern (named constants `_V2_VECNORM_PATH`/`_V1_VECNORM_PATH`
that are each used in exactly one `VecNormalize.load()` call) prevents the most critical
failure mode: loading obs14_v1 VecNorm statistics for an 8D observation space. This
failure would either raise a dimension error or — worse — silently produce invalid
normalized observations if SB3 does not check dimensions at load time. The named-constant
pattern makes this failure impossible to commit by accident.

**The schema lock assertion must run before VecNormalize initialization:**
In `train_ppo_hydrienv_v2.py`, the schema check instantiates `HydrionEnv` and asserts
shape before `DummyVecEnv` or `VecNormalize` are initialized. This ordering matters: if
the assertion fires after VecNormalize initializes with an incorrect dimension, the error
message is less clear and cleanup may be required. The implementation plan specifies this
explicit ordering.

**AC11 is the most important acceptance criterion for M8 result validity:**
AC11 requires that ppo_hydrienv_v1 mean return is within 5% of the M7 canonical value
(694.639) during M8 evaluation. If AC11 fails, the v2-vs-v1 comparison is invalid —
the evaluation environment may have changed, breaking the cross-schema comparison's
matched-conditions requirement (MVC-2). The implementation plan handles this explicitly:
if AC11 fails, the eval script prints a warning and instructs the user to halt and
investigate before interpreting any deployment-gap result.

**The 20-step execution order has two full test-suite runs:**
Steps 5 and 12 both run `pytest tests/`. Step 5 runs immediately after F1/F2 changes
(before any new scripts are created) to verify that the env.py + sensor_fusion.py changes
introduce no regressions. Step 12 runs after all five files are created to verify
end-to-end correctness. This double-gate prevents a scenario where F3/F4/F5 creation
masks a regression in the core env changes.

## 4. Why It Matters

M8 is the first HydrOS milestone where a change to `HydrionEnv` observation behavior is
intentional and permanent rather than a correction or extension. Unlike M6.2B (which added
sensor indices to an existing schema), M8 introduces a second independently-specified
observation regime. The implementation approach must therefore satisfy a constraint that
did not apply before: the two regimes must coexist in the same codebase without
interference.

The additive pattern is the architectural answer to this requirement. Its value is not just
technical elegance — it is that it makes the following guarantee provable by test rather
than by inspection: "adding obs8_deployment_v1 did not change obs14_v1 behavior." T5a
and T5b in `test_obs8_schema.py` encode this guarantee as executable assertions.
