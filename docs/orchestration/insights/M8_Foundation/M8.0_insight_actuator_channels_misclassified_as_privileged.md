# M8.0 Insight — Actuator Command Channels Misclassified as Privileged in obs14_v1

## 1. Title

obs14_v1 indices 6–9 (actuator commands) are sourced from `truth_state` and labeled
"privileged / not available at hardware deployment" — but they are deployment-available.
The true physics-truth gap is only 6 channels (indices 0–5), not 10.

## 2. Context

Produced during M8.0 refinement pass (M8.1 — Refinement Output). Discovered by reading
`hydrion/sensors/sensor_fusion.py` to understand the exact source and deployment
availability of all 14 obs14_v1 channels.

M7 metadata (`ppo_hydrienv_v1_meta.json`) records:
`"truth_channels": "0-9 (privileged -- not available at hardware deployment)"`

This is imprecise. Inspection of `sensor_fusion.py` shows:
- Indices 0–5: physics quantities (flow, pressure, clog, E_field_norm, C_out,
  particle_capture_eff) — computed by the physics simulation, not measurable at
  hardware deployment.
- Indices 6–9: actuator command values (valve_cmd, pump_cmd, bf_cmd, node_voltage_cmd)
  — the controller's own previous-step action outputs. These are always known to the
  controller and are trivially available at deployment.

## 3. What Was Learned

The "deployment gap" in obs14_v1 is 6 channels, not 10. The deployment-realistic
observation schema is not 4D (sensors only, 10–13) but 8D (actuator feedback 6–9 plus
sensors 10–13). A "remove all truth channels" ablation that also removes actuator
feedback (6–9) would be testing a scenario more restrictive than any realistic deployment
configuration.

The actuator command channels may carry significant policy-useful information. In
partial-obs control, action history is a common substitute for hidden state (the
controller knows what it has been doing even if it cannot observe the result fully).
Removing this information from the ablation would confound the result.

## 4. Why It Matters

The ablation design determines what M8 proves. If the ablation removes more channels
than a deployed system would actually lack, a policy failure would be misleading —
attributed to "deployment conditions" when it was actually caused by a more extreme
information restriction than deployment requires. Conversely, if the ablation is too
conservative, the policy might appear deployment-viable without being tested at the
right boundary.

Getting the ablation target right is prerequisite to all downstream deployment-relevance
claims. The actuator-channel finding fixes the target.

## 5. Immediate System Implication

The M8 target schema is `obs8_deployment_v1` (indices 6–13), not `obs4_sensor_only`.
This is the deployment-accurate observation vector:
- 4 actuator command channels (6–9): always known to the controller
- 4 sensor-derived channels (10–13): hardware-measurable

The obs14_v1 metadata field `"truth_channels": "0-9"` is imprecise and should be
corrected in M8 metadata. M8's `ppo_hydrienv_v2_meta.json` must include a
`channel_taxonomy` field explicitly classifying each index by deployment availability.

This correction does not invalidate ppo_hydrienv_v1. It clarifies what ppo_hydrienv_v1's
metadata means and what the deployment gap actually is.

## 6. Workflow Implication

When writing M7-style metadata for future training artifacts, the `truth_channels`
field should not group all "simulation-sourced" channels together. The correct taxonomy
distinguishes:
- physics_truth_channels: channels requiring a simulation model to compute
- actuator_feedback_channels: controller-issued commands, always available
- sensor_derived_channels: hardware-measurable, available with appropriate sensors

This three-class taxonomy should be the standard going forward, not the binary
"truth vs sensor" framing.

Future milestones that reference obs14_v1 as having "10 privileged channels" should be
read as imprecise. The corrected count is: 6 privileged (physics truth) + 4
deployment-available (actuator commands) + 4 sensor-derived = 14 total.

## 7. Recommended Carry-Forward Action

1. Define `obs8_deployment_v1` in M8.2 (execution document) using the correct 8-channel
   set: indices 6–9 (actuator feedback) + 10–13 (sensor-derived).
2. Add a `channel_taxonomy` field to `ppo_hydrienv_v2_meta.json` with per-channel
   deployment availability classification.
3. Update M8 test suite to include a test that asserts the deployment-availability
   classification of each obs14_v1 channel (a living test, not just a metadata note).
4. In all M8+ communications, use "physics truth channels" (0–5) and "actuator command
   channels" (6–9) as distinct categories. Retire the imprecise "truth channels 0–9"
   shorthand unless referring to the legacy obs14_v1 metadata only.
