# M10.0 Insight — Particle capture is reward-decoupled in the current production config

**Source analysis:** M10.1 refinement pass — code-level tracing of reward coupling paths
**Date:** 2026-04-16
**Milestone:** M10_Physics
**Iteration:** M10.0
**Insight class:** structural coupling finding (risk-reducing for M10 integration)

---

## The insight

> **The particle capture subsystem (`particles.py`) and the electrostatics subsystem (`electrostatics.py`) are reward-decoupled in the current production configuration. Neither feeds into the reward function.** The coupling path that would connect them to reward — `C_fibers` from `particles.py` into `clogging.py` deposition rate — is disabled because `psd.enabled: false` and `shape.enabled: false` in all canonical configs. When both are disabled, `particles.py:344-346` unconditionally sets `C_fibers = 1.0`, severing the particle→clogging→reward feedback loop. **However, `C_out` from the particle model does feed into the optical sensor (`optical.py:107`), which writes `sensor_turbidity` and `sensor_scatter` into obs8 indices 4-5. Capture is invisible to reward, but partially visible to the policy through this optical sensor pathway.**

*[Corrected by M10.1A patch — 2026-04-16. Original version incorrectly stated capture does not flow into any observation channel.]*

The reward function (`env.py:303-308`) reads five values from `truth_state`:

| Reward term | State source | Subsystem |
|-------------|-------------|-----------|
| `q_proc / Q_norm` | `q_processed_lmin` | Hydraulics |
| `(pressure − 0.50)²` | `pressure` | Hydraulics (derived) |
| `mesh_avg − 0.40` | `mesh_loading_avg` | Clogging |
| `q_bypass / Q_norm` | `q_bypass_lmin` | Hydraulics |
| `bf_active` | `bf_active` | Backflush state |

None of these depend on `capture_eff`, `C_out`, `E_capture_gain`, `eta_system`, or any particle-model output. The particle model computes `C_out` and `particle_capture_eff`, but these values do not flow into any reward term. `C_out` does flow into the optical sensor, affecting obs8 indices 4-5 — this is a correct downstream consequence, not a hidden coupling.

---

## Why this matters for M10

1. **Integration risk is dramatically lower than expected.** Replacing M4 power-law capture with M5 RT capture does NOT perturb the reward gradient. Training stability with respect to reward is preserved by architecture, not by luck.

2. **The toggle architecture is sufficient.** Because capture is reward-decoupled, a config-level toggle between `capture_mode: "m4"` and `capture_mode: "m5_rt"` produces identical reward behavior. obs8 indices 4-5 (sensor_turbidity, sensor_scatter) will differ between modes due to the C_out → optical sensor pathway — this is expected and correct. obs8 indices 0-3 and 6-7 are unaffected. M9 reproducibility is guaranteed by the toggle under `"m4"` mode.

3. **The same decoupling applies to nDEP.** `E_capture_gain` feeds into `particles.py` as a multiplier on capture efficiency (`alpha_E × E_capture_gain`). But since capture efficiency doesn't flow into the reward, changing the electrostatics model from pDEP to nDEP would also be reward-neutral. This de-risks Phase 2.

### Observation coupling — partial, expected

*[Added by M10.1A correction patch — 2026-04-16]*

The reward-decoupling does NOT extend to full observation-decoupling. The pathway:

```
particles.py:351 (C_out) → optical.py:107 → sensor_turbidity (obs8[4]), sensor_scatter (obs8[5])
```

means the policy sees different optical sensor readings under M5. The gain coefficient is 0.6 per channel. This is a correct physical consequence — when capture physics change, the outflow particle concentration changes, and the optical sensor correctly observes that change. It should be treated as an expected difference in the evaluation plan, not as a failure signal.

---

## The danger: the coupling path exists but is latent

The decoupling depends on a single conditional (`particles.py:344-346`):

```python
if self._psd_cfg["enabled"] and self._shape_cfg["enabled"]:
    C_fibers = self._shape_cfg["fiber_fraction"] * C_in
else:
    C_fibers = 1.0
```

If any future milestone enables PSD and shape, `C_fibers` becomes a function of `C_in`, which is a function of capture efficiency. At that point:

```
capture_eff → C_out → C_fibers → clogging dep_rate → fouling_frac → mesh_loading_avg → reward
```

The decoupling is not architectural — it is configurational. It holds because of a config flag, not because the code enforces separation. Any M10+ work that enables PSD/shape must re-validate reward coupling.
