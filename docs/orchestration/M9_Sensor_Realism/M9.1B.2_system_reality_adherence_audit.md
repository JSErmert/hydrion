## 1. Executive Reality Verdict

HydrOS is **physically plausible but only partially grounded in experimentally validated physics**. Source-grounded physics exists in two distinct forms: (a) the M5 research pass in `hydrion/physics/m5/` (Rajagopalan–Tien 1976, Clausius–Mossotti, nDEP-confirmed on PP/PE/PET, CODATA constants), and (b) the M6 Phase-1 sensor noise and latency parameters for `dp_sensor` and `flow_sensor` (PMC 2024, Nature 2023, mag-meter and factory-calibration spec ranges). The **strongest grounded area** is the M5 physics module — every constant traces to a peer-reviewed source and `M5_PHYSICS_BASIS.md` is a real provenance record. The **weakest grounded area** is the optical sensor layer (`turbidity`, `scatter`, `camera`): gain coefficients and noise floors are domain-guess design choices with zero literature citation, and `optical.py` mirrors sensor outputs back into `truth_state` in violation of the architectural state-separation rule (explicitly marked temporary in code, but currently active). The **most dangerous realism gap** is not the sensor noise placeholders identified in M9.1A — it is that **the M5 source-grounded physics (RT 1976 capture, nDEP dielectrophoresis) is not wired into the production simulation loop**. The running engine still uses M4 phenomenological curves for particle capture and a pDEP-flavored radial-field electrostatics model, while the peer-reviewed M5 code sits adjacent to it unused. This creates a two-track grounding split: HydrOS has done the grounding research but has not yet collected it into the loop the policy trains on.

## 3. Subsystem Deep Dive

### 3.1 Hydraulics (`hydrion/physics/hydraulics.py`)

**What is implemented.** Pump curve intersection solving `u² + β·u − 1 = 0` from a parabolic pump saturation curve (`P = P_max·(1 − (Q/Q_max)²)`) against a linear system resistance (lines 207–222). Per-stage mesh resistance of the form `R_mi = R_mi_base + k_mi_clog · n_i` where `n_i` is normalized fouling (lines 188–199). Passive pressure-relief bypass with hysteresis (lines 224–242, 65 kPa threshold, 90% deactivation band). Explicit per-stage ΔP decomposition writes `dp_stage_1/2/3 = R_mi · Q_proc` to state (lines 247–259).

**Equations / models.** Pump affinity law (standard, uncited specific pump). Linearized Darcy–Weisbach-style resistance `ΔP = R·Q` per element. First-pass pore-size inverse-proportionality for `k_m1/2/3` (2e7 / 8e7 / 3e8 Pa·s/m³).

**Classification.** `derived-grounded`. The structural physics (pump saturation curve intersected with series linear resistance) is textbook. The bypass hysteresis is a standard safety relief model.

**Parameter grounding.** `Q_max_Lmin=20.0`, `P_max_Pa=80000` are locked-envelope design constraints. `R_pipe=2.0e7` and `k_m1/2/3` are design defaults. `configs/default.yaml:34-36` explicitly labels these "first-pass area-scaling approximation" — not lab-calibrated against a physical filter's measured ΔP(Q, fouling) surface. `bypass_pressure_threshold_pa=65000` is "81% of P_max," a design choice rather than a hardware set-point.

**Risks.**
- Any ΔP-driven reward or obs channel inherits an uncalibrated resistance slope. Policies that use ΔP as a clogging proxy are learning from a slope whose magnitude is not anchored to any real filter.
- `bypass_pressure_threshold_pa` is coupled to `P_max`. If `P_max` ever changes to reflect real hardware, the bypass behavior does not re-scale.
- Pump curve is not device-specific; no datasheet reference in repo.

### 3.2 Clogging dynamics (`hydrion/physics/clogging.py`)

**What is implemented.** Four-component decomposition per stage (cake, bridge, pore, irreversible) updated through deposition, shear, and backflush recovery (lines 232–280). Nonlinear deposition `dep_total = dep_rate · dep_base · Q · C_fibers · max(ff, dep_floor)^dep_exponent · dt` (lines 439–447). Linear shear removal `shear_total = shear_coeff · shear_scale · recoverable · dt`, distributed proportionally across recoverable components (lines 458–465). Irreversible stress accumulation above 70% fouling (5% of new deposition becomes irreversible; lines 451–455). Non-monotonic capture-efficiency curve `capture_eff = base + gain · mesh_load_avg · (1 − mesh_load_avg)^exp` clipped to [0.30, 0.98] (lines 299–308).

**Equations / models.** Phenomenological power-law deposition with a nucleation floor; first-order linear shear removal ("proportional to flow"); empirical non-monotonic capture-efficiency curve. None are derived from a filtration-theory source — the system reference implementation for filtration theory lives in `m5/capture_rt.py` (Rajagopalan–Tien 1976) and is **not** invoked from `clogging.py` or `particles.py` production paths.

**Classification.** `internal assumption`. This is not phenomenology on top of a grounded baseline — it is a free-standing parameterized model whose coefficients were chosen by design intent and then adjusted by fix decree in M1.5.

**Parameter grounding.** `dep_exponent=1.0` (changed from 2.0 in M1.5 because 2.0 produced bistability that prevented fouling growth from clean; this is an RL-viability fix, not a physical correction). `dep_floor=0.05` is a nucleation-rate design choice. `shear_coeff=5e-3 s⁻¹` is ungrounded (no Reynolds/Sherwood correlation cited). `dep_rate_s1/2/3 = 0.5 / 0.8 / 1.5` are "first-pass estimates based on physical intuition … NOT validated against lab data" (configs/default.yaml:16-18). Capture-efficiency `baseline / gain / exponent = 0.80 / 0.12 / 0.50` are purely empirical, YAML-tunable with no derivation.

**Risks.**
- **Bistability fix by decree.** The exponent 2→1 change was made because the original model had a physically implausible fixed point at clean state. Neither the old nor the new exponent is literature-anchored, so the fix restored trainability without closing the grounding gap.
- **Non-monotonic capture curve as an RL attractor.** `capture_eff` peaks at moderate fouling. This creates a local optimum in any reward that includes capture — policies may learn to hold the filter at a fouling level that is an artifact of the curve shape, not a real filter property.
- **Component weights arbitrary.** The 60%-bridge / 55%-cake per-stage allocations are narrative ("fiber entanglement at coarse mesh") without quantitative grounding and are not on the M1.5 calibration list.

### 3.3 Backflush behavior (`clogging.py` + `hydraulics.py` pulse logic)

**What is implemented.** Pulse state machine reads `bf_active` and cumulative burst count, applies diminishing factor `0.80^(max(n_bursts−1, 0))`. Per-component recovery fractions cake 35%, bridge 20%, pore 8%, scaled by source-cleaning efficiency and diminishing factor (`clogging.py:467-476`). Hydraulic diversion applies an 80% flow penalty during a pulse (`hydraulics.py:262`). Three pulses per burst, 0.4 s each, 9 s cooldown (`configs/default.yaml:126-129`).

**Equations / models.** Empirical exponential-style recovery: each component removed as a fraction of its current value per pulse window, scaled by a diminishing factor. No connection to a published backflush recovery model.

**Classification.** `internal assumption`. The pulse duration and cooldown are within the locked system constraints (09_REALISM_ROADMAP.md), but the recovery fractions and diminishing factor are design heuristics.

**Parameter grounding.** `bf_cake_recovery=0.35`, `bf_bridge_recovery=0.20`, `bf_pore_recovery=0.08`, `bf_diminishing_factor=0.80`, `effluent_cleaning_efficiency=0.70` — all uncited. The 80% hydraulic diversion penalty is also not physics-derived.

**Risks.**
- The observed policy behavior around backflush ("when to pulse, how often, when to wait for cooldown") is being tuned against a recovery model whose absolute rate is unvalidated. A policy that learns "one extra pulse recovers X% more" is learning a synthetic gradient.
- Component-sum clipping (`clogging.py:483-491`) is a post-hoc safety check rather than a consequence of the model. That it is needed at all means some parameter combinations produce internally inconsistent states.

### 3.4 Electrostatics / DEP (`hydrion/physics/electrostatics.py` + `hydrion/physics/m5/dep_ndep.py`)

**What is implemented — production (M4).** First-order voltage dynamics `V_node += ((V_target − V_old)/τ − leak · V_old)·dt` (lines 142–146). Radial cylindrical field `E_r(r_outer) = V / (r_outer · ln(r_outer/r_inner))` (lines 99–107). Two-subsystem combined capture gain: ring contribution `0.30 · tanh(V_norm) · tanh(t_residence/t_E_ref)` + node contribution `0.70 · E_field_norm · tanh(t_residence/t_E_ref)`, summed to `E_capture_gain ∈ [0, 1]` (lines 161–174). This value is consumed by `particles.py` (lines 295–299) as a multiplicative capture boost.

**What is implemented — research (M5).** `m5/dep_ndep.py` implements the Clausius–Mossotti factor `Re[K] = (ε_p − ε_m)/(ε_p + 2ε_m)` with Maxwell–Wagner relaxation, explicit nDEP confirmation for PP/PE/PET (Re[K] ≈ −0.47 to −0.50), and DEP force `F_DEP = 2π·ε_m·ε₀·r³·Re[K]·∇|E|²`. Critical capture velocity `v_crit = ε_m·ε₀·r²·|Re[K]|·∇|E|² / (3μ)` balances DEP force against Stokes drag and feeds a sigmoid capture probability. Peer-reviewed sources: Pethig 1992, Gascoyne & Vykoukal 2002, RSC 2025.

**Equations / models.** M4: correct radial field formula, but capture-gain function is tanh-saturation phenomenology. M5: Clausius–Mossotti + DEP force + Stokes drag, all textbook and primary-source-cited.

**Classification.** M4 production electrostatics = `internal assumption`. M5 research electrostatics = `source-grounded`. **The subsystem is split.** The production loop runs M4; the grounded M5 physics is adjacent but not wired.

**Parameter grounding.**
- M4: `r_inner_m=0.005`, `r_outer_m=0.040`, `stage_volume_L=0.25`, `stage_height_m` are all explicitly marked "placeholder; bench calibration target" in `electrostatics.py:153-158`. `V_max_realism=2500 V` is a locked system constraint. `tau_charge=0.5 s`, `t_E_ref=2.0 s`, `V_ring_ref=500 V`, `ring_weight/node_weight=0.30/0.70` are all design defaults; the 30/70 split is locked in `06_LOCKED_SYSTEM_CONSTRAINTS.md` §D but is not physics-derived.
- M5: `ε_PP / ε_PE / ε_PET` and `σ_p` from Brandrup 1999, Ku 1987, Neagu 2002. `ε_water(20°C)=80.20±0.03` from Fernández 1995 (IAPWS 1997). Hamaker `H=0.5×10⁻²⁰ J` from Visser 1972 / Gregory 1981. CODATA 2018 fundamental constants. `grad_E2 ≈ (β·E_mean)² / electrode_gap` with `β ≈ electrode_gap/tip_radius` is flagged in the code and in `M5_PHYSICS_BASIS.md §7` as "APPROXIMATE — replace with FEM simulation value for precision."

**Risks.**
- **pDEP-flavored capture function in production.** M4 `E_capture_gain` treats higher |E| as higher capture. M5 establishes that all three target polymers exhibit **negative** DEP — particles are repelled from the high-|E| region toward the low-field center of the cone, where capture occurs by a different mechanism (apex accumulation, not wall attraction). The production capture-gain function is not consistent with nDEP geometry.
- **Uncalibrated geometry.** `r_inner`, `r_outer`, `stage_volume` are placeholders, so even the correct radial field formula produces a numerically arbitrary E at the inner electrode. Training against this field shape is training against a number that has no hardware anchor.
- **Research-to-production gap.** The grounding work in `m5/` is the most physically rigorous code in the repo and is not in the training loop. The value of M5 as a reliability uplift is currently zero until integration happens.

### 3.5 Particle modeling (`hydrion/physics/particles.py` + `hydrion/physics/m5/capture_rt.py`)

**What is implemented — production (M4).** Density classification splits particles 70% dense (PET, PA, PVC) / 15% neutral / 15% buoyant (PP, PE); only dense and neutral are captured (lines 32–35, 243–246). Per-stage capture curves: `η_s1 = (d_p/500µm)^1.5` coarse, `η_s2 = (d_p/100µm)^1.2` medium, `η_s3 = (d_p/5µm)^0.8 · exp(−0.04·max(Q−10, 0))` fine + flow penalty (lines 75–103). Stokes settling boost `v_s = (ρ_p − ρ_w)·g·d²/(18μ)` capped at 5% (lines 61–72, 301–309). System efficiency as series-stage combination `η_sys = 1 − ∏(1 − η_i)`. A deterministic reference `η_nominal` at d=10 µm, Q=13.5 L/min is exported as a normalization baseline (lines 334–341).

**What is implemented — research (M5).** `m5/capture_rt.py` implements the Rajagopalan–Tien 1976 single-collector efficiency `η₀ = 4·A_s^(1/3)·N_Pe^(−2/3) + A_s·N_Lo^(1/8)·N_R^(15/8) + 0.00338·A_s·N_G^1.2·N_R^(−0.4)` (diffusion + interception/vdW + gravity). Dimensionless numbers are computed with textbook definitions; mesh solidity uses the woven-mesh formula `α = 2(d_w/L) − (d_w/L)²`; Happel sphere-in-cell parameter `A_s = 2(1−p⁵)/(2 − 3p + 3p⁵ − 2p⁶)`; Brownian diffusivity from Stokes–Einstein. Sources: Rajagopalan & Tien 1976 (AIChE Journal), Elimelech et al. 1995.

**Equations / models.** M4 capture: phenomenological power laws. M5 capture: primary-source filtration theory.

**Classification.** M4 = `internal assumption`. M5 = `source-grounded`. **Same split as electrostatics: grounded code exists but is not integrated.**

**Parameter grounding.**
- M4: `dense_fraction=0.70` is a design allocation (not literature; all three polymers can both sink and float under weathering/biofouling). `ρ_dense=1380 kg/m³` is a PET handbook value. `μ_water=1e⁻³ Pa·s` (water @ 20°C) is source-grounded. `fouling_gain_s1/2/3 = 0.15 / 0.20 / 0.10` and `s3_flow_penalty_coeff=0.04` are first-pass estimates without lab data. `η_ref_d=10 µm` and `η_ref_Q=13.5 L/min` are locked reference conditions.
- M5: Rajagopalan–Tien formula peer-reviewed primary. `MESH_S1/2/3` wire diameters marked `[DESIGN_DEFAULT]` in `m5/capture_rt.py:87-100` — wire diameters estimated as opening/4 or opening/3.3 without device measurement. Brownian diffusivity and Stokes–Einstein are textbook.

**Risks.**
- **M4 capture curves are the exclusive production physics for micro-particle capture.** The three exponents 1.5, 1.2, 0.8 are not derived from any filtration theory and have no error bars. A policy optimizing capture is optimizing against a shape that M5 explicitly exists to replace.
- **Density split is not physical.** A fixed 70/15/15 allocation cannot represent the real distribution of aged, biofouled, or irregular-shaped microplastics. The "only dense and neutral captured" rule is a simplification that M5's nDEP + gravity term is designed to replace with a calculated outcome per particle.
- **Fouling-capture coupling not physics-based.** `fouling_gain_s*` terms boost capture as fouling rises — this is the source of the non-monotonic `capture_eff` curve seen in clogging.py. It is unvalidated against any filter dataset.

### 3.6 Sensor modeling (`hydrion/sensors/*.py`, `hydrion/sensors/sensor_fusion.py`)

**What is implemented.**
- **`dp_sensor`** (`pressure.py`): `measured = clip_below(0, truth_dp_kPa + noise + drift + fouling_offset)` where `noise ~ N(0, 0.25 kPa)`, `drift` is a clipped random walk (`±2.0 kPa` envelope, `0.0005 kPa/step`), and `fouling_offset = 0.5 · mesh_loading_avg`. One-step (100 ms) latency via circular buffer.
- **`flow_sensor`** (`flow.py`): `sensor_q = truth_q · (1 + N(0, 0.01)) + bias_episode`, where `bias_episode ~ N(0, 0.2²)` is sampled once at reset and held fixed through the episode.
- **`optical`** (`optical.py`): `turbidity = 0.8·mesh_avg + 0.6·C_out + 0.4·(1 − capture_eff) + N(0, 0.01)`, clipped [0, 1]. `scatter = 0.4·(Q/Q_ref) + 0.6·C_out + N(0, 0.01)`, clipped [0, 1]. `camera` fuses brightness and turbidity with `N(0, 0.02)`.
- **`sensor_fusion`**: assembles `obs14_v1` (14 channels: physics_truth 0–5, actuator_feedback 6–9, sensor_derived 10–13) and `obs8_deployment_v1` (actuator_feedback 0–3, sensor_derived 4–7).

**Equations / models.** DP: additive-noise transducer model with drift and linear fouling offset. Flow: multiplicative-noise transducer with persistent factory bias. Optical: linear-combination-of-state transducers with Gaussian noise.

**Classification.** See §4 for per-channel detail. Summary: `dp_sensor noise + latency` and `flow_sensor noise + bias` are source-grounded; `dp_sensor drift + fouling offset` are placeholders (mechanism confirmed, magnitude unvalidated); `turbidity / scatter / camera` are internal assumptions.

**Risks.** See §4 and §6.

### 3.7 Reward / control interface — physical-meaning slice

**What is implemented.** The reward composition in the HydrionEnv is built on `capture_eff`, `q_processed_lmin`, control-effort terms, and (in obs8 deployment) a sensor-derived slice. No reward term injects physics constants directly; all terms are downstream consequences of the state produced by §3.1–§3.6.

**Classification.** `derived-grounded with caveat`. The **form** of the reward is a control-engineering construction, not a physics claim. The **grounding** of each term is inherited from whatever produced its input. So: `capture_eff_term` inherits the L2 classification of M4 particle capture; `Q_term` inherits the derived-grounded classification of hydraulics; `ΔP_term` (if used) inherits the uncalibrated resistance slope.

**Risks.** No novel risk beyond what the upstream subsystems contribute, but one aggregation effect: **the reward is a weighted sum of signals with heterogeneous grounding**. Source-grounded Stokes/CODATA constants mix with placeholder `dp_fouling_gain` and phenomenological `capture_eff` curve shapes in the same scalar reward. The policy cannot distinguish which part of its gradient is real.

## 4. Sensor Reality Analysis (CRITICAL)

### 4.1 `dp_sensor` (`hydrion/sensors/pressure.py`)

- **How it is computed.** Reads `truth_dp_kPa = truth_state["dp_total_pa"] / 1000.0`. Applies white Gaussian noise `N(0, σ=0.25 kPa)`. Adds a clipped random-walk drift (lines 96–97). Adds a fouling-induced linear offset `dp_fouling_gain · mesh_loading_avg` (line 102). Clamps below 0. Passes through a 1-step circular deque for latency. Writes only to `sensor_state["dp_sensor_kPa"]` (no `truth_state` writes).
- **Noise model.**
  - White Gaussian (σ=0.25 kPa) — **source-grounded** to PMC 2024 and Nature Microsystems & Nanoengineering 2023 as a conservative low-cost MEMS noise floor.
  - Clipped random-walk drift (0.0005 kPa/step, ±2 kPa envelope) — **placeholder**. The mechanism (port fouling, long-term aging) is confirmed in STS Sensors technical notes and in the research brief, but the drift rate and envelope are application-specific and labeled "calibration-pending" in `pressure.py:24`.
  - Fouling offset (gain 0.5) — **placeholder**. The mechanism (port blockage producing a bias proportional to accumulated mesh load) is documented, but no peer-reviewed quantitative relationship exists for laundry-service DP sensors. `pressure.py:26` and `research_brief.md:49` flag it "calibration-pending."
  - 1-step latency (≈100 ms) — **source-grounded** to Nature 2023 and Eastsensor/Analog Devices response-time specs (1–50 ms range).
- **Real-world class.** Low-cost MEMS differential pressure transducer, 0–50 kPa range, piezoelectric strain gauge. The device class is named in the research brief but not in a datasheet-level sense; the target hardware is not yet selected.
- **What is missing for realism.** Field calibration of `dp_drift_rate_kPa_per_step`, `dp_drift_max_kPa`, and `dp_fouling_gain` against a candidate DP sensor under laundry-wastewater service. A published provenance record per parameter with uncertainty bands.

### 4.2 `flow_sensor` (`hydrion/sensors/flow.py`)

- **How it is computed.** Reads `truth_q = truth_state["q_processed_lmin"]`. Applies multiplicative white noise `truth_q · (1 + N(0, 0.01))` and adds a per-episode calibration bias sampled at `reset()`. Writes only to `sensor_state["flow_sensor_lmin"]`.
- **Noise model.**
  - Multiplicative 1%-of-reading Gaussian — **source-grounded** within mag-meter specification range (±0.5–1% reading for electromagnetic flow meters in dirty-water service; 1% is a conservative dirty-water bound).
  - Per-episode calibration bias `N(0, 0.2 L/min)` — **source-grounded** to factory-calibration tolerance (±0.5–1% FS at 15 L/min FS = ±0.075–0.15 L/min; 0.2 is a conservative upper bound). Held fixed through the episode, which correctly models factory calibration as a persistent offset, not tick-by-tick jitter.
- **Real-world class.** Electromagnetic (Faraday-principle) flow meter, 0–15 L/min range, conductive dirty-water rated. Mag meters are drift-free with temperature, which justifies the absence of a drift term in the flow path.
- **What is missing for realism.** Essentially nothing for the noise model itself under the stated sensor class. What is missing is commitment to a concrete device (datasheet reference) and confirmation that the M8 policy's error envelope matches that device's specified repeatability under real flow disturbances.

### 4.3 `turbidity` (`hydrion/sensors/optical.py`)

- **How it is computed.** `turbidity = 0.8·mesh_loading_avg + 0.6·C_out + 0.4·(1 − capture_eff) + N(0, 0.01)`, clipped [0, 1]. Reads truth_state values; **writes a compatibility mirror back into `truth_state["sensor_turbidity"]`** (`optical.py:144`, marked temporary).
- **Noise model.** Additive Gaussian σ=0.01 on the [0, 1] output scale. **Unsourced.**
- **Real-world class.** **Not specified.** No OEM name, no measurement principle beyond "optical density," no range, no spectral band.
- **What is missing for realism.** Everything sensor-specific: what device class is this (nephelometric ISO 7027? transmission photometer?), what is its response curve to laundry-wastewater turbidity (NTU range, calibration standard), what noise floor and drift does the OEM datasheet specify, what interference does fibre content produce? The gain coefficients 0.8 / 0.6 / 0.4 are symbolic design weights, not calibration constants.

### 4.4 `scatter` (`hydrion/sensors/optical.py`)

- **How it is computed.** `scatter = 0.4·(Q / Q_ref) + 0.6·C_out + N(0, 0.01)`, clipped [0, 1]. Same compatibility mirror to `truth_state`.
- **Noise model.** Additive Gaussian σ=0.01. **Unsourced.**
- **Real-world class.** **Not specified.** The name "scatter" suggests a side-scatter optical sensor, but no device model, wavelength, or scattering angle is named.
- **What is missing for realism.** A physical model relating particle concentration, size distribution, and refractive-index contrast to scattered-light intensity. Currently the sensor outputs a symbolic linear combination of flow and concentration with no physics underneath.

### 4.5 Bottom-line sensor findings

- The **sensor subsystem's grounding ceiling is set by the DP + flow channels** (source-grounded noise + latency + factory bias), which are the channels most directly tied to the M8 deployment decision. This is a credit.
- The **sensor subsystem's grounding floor is the optical triplet**, which is currently not a sensor at all in the physical sense — it is a policy-visible function of state with noise on top. This is the largest single reality gap inside the sensor subsystem.
- The **`optical.py` state-separation violation** (mirrors sensor outputs into `truth_state`) is a CLAUDE.md rule breach that is active today, despite being annotated as temporary. It means that anything downstream that reads `truth_state["sensor_turbidity"]` is reading a synthesized, noisy mirror, not ground truth. This is a latent correctness hazard.

## 6. Most Dangerous Reality Gap

> **The production simulation loop runs M4 phenomenology while the source-grounded M5 physics sits uncalled in `hydrion/physics/m5/`. The most physically rigorous code in the repo is not on the execution path — and the policy, the sensor fusion, and the reward are all built on top of the M4 layer that M5 was designed to replace.**

**Why it is dangerous.**

This is not a single placeholder — it is a **structural bifurcation**. HydrOS has two physics layers:

- **Layer A (production):** `hydraulics.py`, `clogging.py`, `electrostatics.py`, `particles.py`. Phenomenological, uncalibrated, internally consistent, wired into `HydrionEnv`, trained against by PPO.
- **Layer B (research):** `hydrion/physics/m5/*`. Peer-reviewed, CODATA-consistent, nDEP-corrected, Rajagopalan–Tien-compliant, **not wired into `HydrionEnv`**.

The governance layer, the M5 physics basis document, and the external story of HydrOS all describe the system as "grounded in peer-reviewed physics" — and that is truthful about Layer B. But the **training and evaluation loop runs on Layer A**. Every M8 result, every canonical policy's +21.2% deployment gap, every `capture_eff` curve, every DEP-weighted reward — all are Layer A observations. The M5 insight that PP / PE / PET undergo **negative** DEP (repulsion from wall, apex accumulation) contradicts the Layer-A implicit assumption that higher field → higher wall capture. M4 electrostatics still uses that assumption.

**What it affects.**

- Every milestone that cites "M5 grounded physics" as a credibility anchor inherits an implicit claim that is only half true.
- The policy's learned gradient from `E_capture_gain · (1 − buoyancy_penalty)` in `particles.py` is a gradient against an M4-shaped field model, not an nDEP-consistent one.
- The "deployment-realistic" obs8 channels (sensor-derived) are honest about their noise grounding, but the physics_truth channels in obs14 (capture_eff, C_out) are Layer-A phenomenology. Any narrative that positions obs14 as "ground truth" is ground truth relative to Layer A, not to Layer B.
- The research-brief effort currently scoped for sensor calibration (M9.1R.1) does not include the Layer-A → Layer-B integration work. That work is invisible in the current M9 plan.

**How it could mislead interpretation.**

- A reviewer reading `M5_PHYSICS_BASIS.md` and `m5/capture_rt.py` will see source-grounded physics and reasonably assume the simulation is built on it.
- A downstream milestone that imports a trained obs8 policy for evaluation under "deployment-realistic" conditions inherits Layer A under a Layer B label.
- The M9 sensor calibration effort, when it completes, will produce grounded sensor parameters that feed a Layer-A physical-truth process. The system will then have grounded sensors sampling ungrounded physics, which is only a half-step from the current state.

This is not a criticism of the M5 work — M5 did exactly what M5 was asked to do (produce a source-grounded reference implementation). It is a **surfaceable gap** that belongs in M9's peripheral vision even if it is not M9's primary target.

## 7. Reality vs System Behavior Risk

**Where the system might behave "correctly" but for the wrong physical reasons.**

- `capture_eff` rising with moderate fouling (clogging.py:299-308) looks like realistic cake-layer behavior (a loaded cake can improve capture by reducing effective pore size). The underlying curve is not derived from filtration theory — it is an empirical non-monotonic function whose peak location is a design choice. A policy that learns to hold the filter at a mid-fouling state will look like it has discovered cake-enhanced capture. It has actually discovered the peak of an arbitrary curve.
- Backflush recovery producing a ΔP drop and a capture-efficiency restoration looks like physical backflush. The magnitude is set by the 35/20/8 recovery fractions and the 80% diminishing factor, neither of which are measured. A policy that learns "pulse three times, wait for cooldown, repeat" is learning the timing of a synthetic recovery curve, not a filter's actual hydraulic cleaning behavior.
- Pressure bypass activating at 65 kPa looks like a safety relief. The threshold is 81% of `P_max`, which is 81% of an envelope constraint, not 81% of a real burst pressure. The behavior is correct in shape, but the set-point has no hardware anchor.

**Where policy learning might exploit unrealistic dynamics.**

- **`dp_fouling_gain` exploitation.** The fouling offset is **linear and deterministic** in `mesh_loading_avg`, and `mesh_loading_avg` is observable via the clogging channels. A sufficiently expressive policy can learn a **defouled** DP signal by subtracting an estimated offset. If real hardware shows `dp_fouling_gain ≠ 0.5`, the policy's learned compensation is wrong in exact proportion.
- **`optical.py` state-separation mirror exploitation.** Because `optical.py` writes `turbidity / scatter / camera` back into `truth_state`, any downstream logic that reads `truth_state["sensor_turbidity"]` is reading the same noisy mirror the policy sees. There is no "truth turbidity" to compare against — the system cannot self-audit its optical sensors.
- **pDEP-shaped capture exploitation.** M4's `E_capture_gain` saturates with both voltage and residence time. A policy can learn to hold voltage at saturation continuously, because the M4 model does not penalize arbitrary |E|. M5's nDEP physics would produce a different optimal control trajectory (field shape matters more than field magnitude once above `v_crit`).

**Where outputs could look valid but are not physically meaningful.**

- Stage-level ΔP traces: mathematically plausible, slope is uncalibrated.
- Rendered dp_sensor telemetry: Gaussian noise and latency are realistic; drift and fouling shape are arbitrary.
- `capture_eff` time series: shape is a non-monotonic empirical curve; absolute level is a design choice.
- Turbidity / scatter / camera traces: not physical measurements at all — they are noisy linear combinations of `truth_state` fields with ad-hoc weights.

**Per-subsystem movement from current state to one realism level above.**

Replace M4 tanh-saturation capture gain with M5 nDEP-consistent capture. Replace placeholder geometry with FEM-derived field maps. Measure mesh wire diameters and remove `[DESIGN_DEFAULT]` flags. Revisit the 70/15/15 density split.
- **Sensor — DP drift + fouling (placeholder → source-grounded).** M9 research brief (already scoped) produces `dp_drift_rate`, `dp_drift_max`, `dp_fouling_gain` with literature or datasheet provenance.

- **Reward / control interface.** No direct uplift — reward grounding is inherited. It advances automatically as the upstream layers advance.

**What is required for functional realism** (every output is interpretable as a measurement of a real-like physical process): M5 integration + sensor calibration + optical sensor class selection + removal of the optical state-separation mirror.

**What is required for robust realism** (outputs reproduce measurement statistics from real hardware): all of the above + lab calibration passes on hydraulics/clogging/backflush + at least one hardware cross-check against a candidate filter.

## 10. Strategic Implication

**Does this audit support or challenge the M9 direction?**

**It supports M9 sensor calibration as the correct immediate move, and it surfaces one adjacent concern that M9.1A did not name.**

- **Support.** Every M9.1A conclusion about the sensor layer being load-bearing is confirmed at the code level. `dp_drift_rate` and `dp_fouling_gain` are genuinely placeholder. The optical layer is worse than M9.1A implied (it is not merely calibration-pending; it is not a physical sensor model at all). Sensor calibration is the correct next move.
- **Refinement.** `flow_calibration_bias_std` is **already grounded** at the code level under the mag-meter class. The M9 research brief effort can be weighted accordingly: heavy on DP, lighter on flow, targeted on optical device-class selection rather than noise-floor tuning.
- **Adjacent concern.** The M4/M5 production/research split (Section 6) is not on the current M9 critical path, and it should not displace M9 sensor calibration. But it deserves explicit acknowledgment as the **next-most-important grounding gap** after sensor calibration completes. Specifically: once sensors are calibrated, the limiting factor on reality adherence shifts from the sensor layer to the Layer-A physics that the sensors observe. If M5 integration is not scheduled, HydrOS will reach a state where the sensors are grounded but the process they measure is not.

**Is sensor calibration still the correct next move?**

Yes. Nothing in this audit contradicts the M9 direction. The M4/M5 integration gap is older, larger, and more structural than sensor calibration, but it is also a larger lift (new wiring, new validation of capture curves, new policy retraining) and would defer M9's load-bearing benefit to `dp_drift_rate` and `dp_fouling_gain`. The correct sequencing is: close M9 sensor calibration, then open an M5-integration branch in M10 or later.

**Does any subsystem unexpectedly require priority?**

One subsystem's unexpectedness deserves a note, but not a priority swap:

- **Optical sensor layer.** M9.1A classified the sensor layer as L2 based on `dp_drift_rate`, `dp_fouling_gain`, and `flow_calibration_bias_std`. This audit shows that the optical triplet (`turbidity`, `scatter`, `camera`) is in worse shape than those three: it has no OEM class, no literature source for any parameter, uses ad-hoc linear gains, and actively violates the state-separation rule via the `truth_state` mirror in `optical.py:144`. This does not change the M9 direction (DP calibration is still the highest-leverage move for deployment realism), but it means M9's research brief should also scope an **optical sensor class selection** task. That is a research-brief scoping addition, not a milestone pivot.

- **State-separation rule compliance.** The `optical.py` → `truth_state` mirror is a **CLAUDE.md Core Rule #2 violation** that is presently active in code. It is annotated as temporary but has no explicit removal schedule. M9's sensor calibration milestone is the natural place to fix it — it costs little and resolves a latent correctness hazard.

**Bottom line.** M9 direction is confirmed. Sensor calibration remains the decisive move. One refinement (flow is already grounded), one additional scope item (optical device-class selection + state-separation fix), and one post-M9 flag (M5 integration as the next-most-important structural gap).
