## 1. Executive Priority Summary

**The system's improvement path is dominated by a single prerequisite move:** lifting the sensor layer out of placeholder-bound L2. Everything else either depends on it or is lower-leverage than it. The top three priorities are structurally ordered, not discretionary.

### Top 3 improvement priorities

**Priority 1 — Sensor calibration (M9 scope).**
Replace placeholder values for `dp_drift_rate`, `dp_fouling_gain`, and `flow_calibration_bias_std` with literature- or datasheet-grounded values. Retrain under calibrated noise as a new canonical artifact (M8 v2 preserved as pre-calibration reference).
*Why it matters:* Directly lifts the primary confound blocking every downstream interpretation. Converts the M8 +21.2% deployment gap result from internally valid to hardware-interpretable.
*What it unlocks:* Domain randomization, obs4 ablation, distillation scoping, frontend validity, and a meaningful M10 scope definition.

**Priority 2 — Step-budget alignment (optional M9 infrastructure).**
Run a full 500k-step `ppo_hydrienv_v2` training path (single command: `python -m hydrion.train_ppo_hydrienv_v2`). Resolves the secondary confound (360k vs 500k asymmetry) and strengthens benchmark precision.
*Why it matters:* Eliminates the step-budget MVC-2 partial deviation from the M8 comparison. Low cost; high benchmark-discipline value.
*What it unlocks:* Step-matched cross-schema comparisons under both placeholder-noise and calibrated-noise conditions.

**Priority 3 — Multi-seed validation (tertiary; M10+).**
Run `ppo_hydrienv_v2` with two or three additional seeds to produce a cross-seed variance estimate. Gives the deployment gap magnitude an error bar attributable to training stochasticity.
*Why it matters:* Currently the gap result has no variance estimate — single seed = no error bar. Resolves the tertiary confound.
*What it unlocks:* Statistical interpretation of the gap (is +21.2% within one sigma of zero, or is the direction robust?).

These three priorities **together** reduce all three M8 confounds. Priority 1 is the prerequisite for the interpretability of 4 downstream branches. Priority 2 is a cheap precision improvement. Priority 3 is statistical due diligence.

## 4. Opportunity vs Risk Balance

### Improvements that reduce risk most

**P1 (sensor calibration)** — reduces the largest risk in the system: the stacked misalignment where three L5 layers rest on an L2 sensor distribution. Every other risk reduction is secondary to this.

**Prohibited-claims propagation discipline (ongoing, not a priority in this list)** — the only thing currently preventing the stacked-misalignment risk from producing false confidence. This is a governance discipline, not an improvement, but it is what keeps risk bounded until P1 completes.

**P3 (multi-seed)** — reduces the risk that the headline +21.2% result is seed-specific rather than structural.

### Improvements that improve capability most

**P1 (sensor calibration)** — lifts the system's ability to produce hardware-interpretable results, which is a categorical capability change, not an incremental one.

**P4 (domain randomization)** — once P1 completes, DR converts the system from point-estimate to robustness-envelope capability.

**P5 (visual console)** — once P1 completes, the console becomes a scientifically grounded operator tool. Before P1, it cannot.

### Improvements that could mislead if done too early

**P5 (visual console before calibration)** — polished visualization of placeholder data is more misleading than no visualization. The visual validity bridge explicitly catches this failure mode.

**P4 (domain randomization before calibration)** — DR over uncalibrated parameters samples from an arbitrary distribution. The result would look like robustness evaluation but would not be.

**P6 (obs4 ablation before calibration)** — would produce a result confounded identically to the M8 result. Looks like ablation evidence but is not interpretable.

**P8 (AROD schema expansion before M9 closes)** — expanding governance scope without advancing physics/sensor layers would worsen the top-heavy condition. Looks like maturation; actually imbalance.

**Common pattern:** All four premature-execution failure modes produce results that look more mature than they are. This is the same structural risk as the stacked misalignment itself — polish creates implicit credibility. The sequencing rules in Section 5 exist to prevent this.

## 5. Sequencing Strategy

### Must happen first (M9)

1. **M9.1R.1 — Research brief prompt** (next artifact, per M9.1 and M9.1A.2 recommendations)
2. **M9.1R.2 — Research brief** (literature / datasheet grounding for sensor noise parameters)
3. **M9.3 — Execution document** (with its own prohibited-claims list defined before results are known)
4. **M9.5 — Implementation plan**
5. **M9.6 — Execution prompt**
6. **Sensor calibration implementation** (config update + retraining pass)
7. **M9 evaluation** (calibrated gap vs v1 baseline; four-scenario classification; AC checks)
8. **M9 closeout + achievements + insight** (complete the M8-class governance chain)

### Can happen next (M10 or parallel to late M9)

- **Step-budget alignment (P2):** optional M9 auxiliary if execution time allows; otherwise early M10
- **Multi-seed validation (P3):** early M10, on calibrated-noise policy
- **Physical truth extension (P7):** parallel-safe; can run in a dedicated calibration sub-pass or be folded into M10 scope
- **Domain randomization (P4):** mid-M10 after multi-seed variance is known
- **obs4 ablation (P6):** mid-M10 after DR, or standalone late-M10 ablation milestone
- **AROD Schema v2 refinement (P8):** gated on M9 closeout third-data-point AROD assessment

### Must wait

- **Visual console milestone (P5):** after M9 closes and sensor calibration data is available; should NOT be scheduled before M9 closes regardless of apparent readiness
- **Any "deployment-ready" claim:** blocked indefinitely under current M8.3 prohibited-claims list; remains blocked through M9 regardless of calibrated result direction
- **Governance scope expansion:** no new AROD layers, no new schemas, no new governance artifacts until physics/sensor layers catch up (earliest M10)

## 7. System Uplift Definition

### What "better HydrOS" means after the prioritized improvements complete

**After Priority 1 (M9 core) completes, HydrOS will be:**

**More realistic** —
Sensor noise characteristics will reflect literature- or datasheet-grounded values for dp transducers and flow meters. The measurement layer moves from L2 to L3 or L4. The gap between truth layer (L4) and sensor layer closes. The system passes from "peer-reviewed physics observed by placeholder sensors" to "peer-reviewed physics observed by grounded sensors."

**More interpretable** —
The +21.2% M8 deployment gap becomes meaningful as a hardware-relevant comparison rather than an internally-valid-only measurement. Every downstream claim (robustness, distillation value, frontend validity) inherits calibrated noise as the baseline interpretation context. The stacked misalignment is reduced from 3 levels (L5 policy over L2 sensor) to ~1 level (L5 policy over L4 sensor).

**More trustworthy** —
The load-bearing textual discipline (prohibited claims, MVC-3 calibration confound) remains in force, but is supported by actual calibration rather than defending against placeholder ambiguity. External readers can rely on the deployment gap figure with explicit confidence bounds (after P3) and a grounded noise model. The risk of "deployment-ready" overclaim propagating through downstream artifacts is structurally reduced.

**More visually valid** —
A post-calibration frontend milestone becomes safe: sensor trace rendering displays grounded data; truth-vs-sensor comparisons are scientifically meaningful; policy-view rendering reflects what a deployed system would actually perceive. The visual validity bridge criteria (truth-vs-sensor fidelity, telemetry meaning, policy-view legitimacy, action-trace legitimacy) become achievable rather than aspirational.

**Balance restored** —
The four-level maturity spread narrows. The top-heavy condition moves toward balance: governance L5, benchmark L5, policy L5, physical truth L4, sensor L3/L4, visual L2 (unchanged until a post-calibration console milestone), deployment L1 (unchanged until hardware contact). The system is no longer dominated by a single load-bearing weakness. The remaining gap (L1 deployment) is a lifecycle gap, not a structural imbalance — it reflects where HydrOS is, not a misalignment.

### The uplift is not about adding features

The prioritized improvements in this document do not propose new capabilities. Every priority is a correction or extension of something that already exists:

- P1 replaces placeholders with grounded values (existing infrastructure)
- P2 extends a training run from 360k to 500k steps (existing script, one command)
- P3 reruns existing training with additional seeds (existing script)
- P4 varies parameters that already exist (new config ranges, no new parameters)
- P5 builds on existing console infrastructure (K-O locked baseline, Vite/React app)
- P6 is a narrower version of the existing obs8 schema pattern
- P7 closes calibration-pending items that were already flagged in M5/M6
- P8 refines an existing governance schema based on observed use

**The improvement is structural, not additive.** HydrOS does not need more features to reach a scientifically trustworthy state. It needs to lift its sensor layer out of placeholder-bound L2 and close the resulting alignment gap. The priorities in this document are ordered to accomplish exactly that.
