## 1. Executive Integration Verdict

The system will remain structurally coherent after sensor calibration. The shape of the noise model in `hydrion/sensors/` (`pressure.py`, `flow.py`, `optical.py`) is compatible with calibrated parameter values; the reward landscape is computed from `truth_state` and is therefore calibration-invariant; the obs8 schema shape is unchanged; and the RL training + evaluation pipeline supports a fresh canonical artifact without touching any protected lineage. The biggest integration risk is **not runtime instability** — it is a **hidden coupling between `configs/default.yaml` and the already-trained `ppo_hydrienv_v1`/`ppo_hydrienv_v2` artifacts**: both trained policies use `configs/default.yaml` sensor values at eval time, so a naive in-place edit to the shared sensors block would silently invalidate the v1 500k canonical baseline (AC11 = 694.639) and the M8 v2 placeholder-noise reference point. The biggest expected improvement is **shift of the primary confound from the data to the parameter provenance record** — the measurement layer moves from L2 (placeholder-bound) toward L3/L4 (source-grounded), and the +21.2% M8 obs8 gap acquires hardware-interpretable meaning regardless of its direction after retraining. **Overall integration confidence: HIGH for structural coherence; MEDIUM-HIGH for clean execution, contingent on (a) config-snapshot isolation so v1/v2 baselines are reproducible and (b) resolution of two specific hidden-coupling traps identified in Section 6.**

## 2. Change Propagation Analysis (CRITICAL)

This section traces a calibrated parameter edit through the full seven-stage pipeline. At each stage: **what changes**, **why**, **expected magnitude**, **risk**.

### Stage 1 — Sensor parameters change

**What changes:** Values in the `sensors:` block of `configs/default.yaml`. Specifically the parameters identified in M9.1R.2 as calibration targets:
- `sensors.pressure.dp_noise_kPa` (currently 0.25)
- `sensors.pressure.dp_drift_rate_kPa_per_step` (currently 0.0005)
- `sensors.pressure.dp_drift_max_kPa` (currently 2.0)
- `sensors.pressure.dp_fouling_gain` (currently 0.5) — semantically confirmed as **process B** (sensor-side bias proportional to `mesh_loading_avg`) in `hydrion/sensors/pressure.py:99–102`, which resolves the M9.1R.2 Section 3.2/4.2 ambiguity in favor of a legitimate sensor-error term rather than a double-count of filter-cake physics
- `sensors.pressure.dp_latency_steps` (currently 1)
- `sensors.flow.flow_noise_frac` (currently 0.01)
- `sensors.flow.flow_bias_std_lmin` (currently 0.2)
- Optionally `sensors.optical.*` noise standard deviations (currently 0.01 for turbidity/scatter; M9.1R.2 flags as calibratable but turbidity-to-mass mapping is excluded from M9 scope)

**Why:** M9 replaces placeholder magnitudes with literature- or datasheet-grounded values for a nominated sensor device class. The **shape** of the noise model (additive Gaussian + random-walk drift + per-episode calibration bias + mesh-loading-proportional sensor offset + 1-step latency buffer) is not changed by M9 — per M9.1R.2 §4.2–4.3, only magnitudes and provenance change.

**Expected magnitude of change:** Depends on the verified-source pass (M9.1R.3, currently gated). The preliminary M9.1R.2 ranges suggest:
- `dp_noise_kPa` may move by a factor of 0.5–2× of the current 0.25 value (realistic MEMS σ range)
- `dp_drift_rate_kPa_per_step` is likely to **shrink by orders of magnitude** toward zero — M9.1R.2 §3.1 shows that realistic per-second drift on an 80 kPa span is ~1e-6 kPa over a 400-step episode, far below the current 0.0005 kPa/step
- `dp_fouling_gain` is likely to **shrink toward zero** for episode timescales — per M9.1R.2 §3.2, episode-scale sensor-line fouling accumulation is negligible
- `flow_noise_frac` and `flow_bias_std_lmin` may move by a factor of 0.5–3×

**Risk introduced:** **Config-file coupling to already-trained artifacts.** This is the single most important integration risk. Editing `configs/default.yaml` in place affects **every** future invocation of `hydrion.eval_ppo_hydrienv_v1`, `hydrion.eval_ppo_hydrienv_v2`, and `hydrion.eval_ppo_cce` that constructs a fresh `HydrionEnv` / `ConicalCascadeEnv` from the default config. Both `ppo_hydrienv_v1` (obs14_v1, indices 12–13 are `flow_sensor_norm` and `dp_sensor_norm`) and `ppo_hydrienv_v2` (obs8, indices 6–7 same) observe the live sensor output. If the sensor pipeline silently produces statistically different outputs after the edit, the saved `_vecnorm.pkl` running statistics become stale and the AC11 exact canonical match (694.639) will not reproduce — a Category-A governance breach.

### Stage 2 — `sensor_state` changes

**What changes:** Per-step values written to `sensor_state["dp_sensor_kPa"]`, `sensor_state["flow_sensor_lmin"]`, `sensor_state["sensor_turbidity"]`, `sensor_state["sensor_scatter"]`. Upstream `truth_state` is unaffected — the sensor update functions read `truth_state` and write `sensor_state` only, per the architectural constraint in `pressure.py:42–45`, `flow.py:41–44`, and `optical.py` (noted: the optical sensor retains a compatibility mirror back to `truth_state`, which is a M9-orthogonal legacy shim and does not affect the calibration propagation).

**Why:** The same `update()` methods execute; only their parameter values differ. All four sensor instances are constructed once at `HydrionEnv.__init__` (env.py:113 etc) from the config and retain their parameter values for the full training/eval run.

**Expected magnitude:**
- `dp_sensor_kPa`: mean unchanged (noise and drift are zero-mean; fouling offset is a small DC bias proportional to `mesh_loading_avg` and was already small). Episode-scale standard deviation of the sensor-error term scales with calibrated `dp_noise_kPa`. If the calibrated value halves to ~0.125 kPa, σ of the normalized obs channel (dp_sensor / 80) drops from ~0.003 to ~0.0015. If it doubles, σ rises to ~0.006. Both are well below the signal range [0, 0.75].
- `flow_sensor_lmin`: per-step noise σ = `flow_noise_frac` × `q_processed`. At Q=13 L/min and 1% noise: σ ≈ 0.13 L/min. Per-episode DC offset ~ N(0, 0.04 L/min²). Normalized to [0,1] via /20, the per-step σ is ~0.0065 and the per-episode bias σ is ~0.01.
- `sensor_turbidity`, `sensor_scatter`: deterministic functions of `mesh_loading_avg`, `C_out`, `capture_eff`, `flow_norm` + small Gaussian noise (σ=0.01). If M9 tightens or loosens these σ values modestly, the channels' noise floor shifts proportionally.

**Risk introduced:** Low at the sensor-write step itself. The risk is not that `sensor_state` misbehaves — it is that the **distribution of `sensor_state` over an episode** acquires new first and second moments.

### Stage 3 — Observation (obs8) distribution changes

**What changes:** `build_observation_obs8(truth, sensors)` in `sensor_fusion.py:109–126` returns a vector whose indices 4–7 (the four sensor-derived channels) now reflect the new `sensor_state` distribution. Indices 0–3 (actuator commands: `valve_cmd`, `pump_cmd`, `bf_cmd`, `node_voltage_cmd`) are read from `truth_state` and are **unchanged** by calibration — they are controller-issued values, not sensor readings.

**Why:** The obs vector assembly function is deterministic; its outputs follow the new sensor distribution by construction.

**Expected magnitude:** Half of the obs8 vector (indices 4–7) sees a distribution shift. Indices 0–3 are identically distributed to pre-calibration. The marginal distribution shift on any single sensor index is on the order of a few percent of its normalized range (see Stage 2). The **joint** distribution shift is more subtle: channels 6 (flow) and 7 (dp) now have different temporal autocorrelation structures because `dp_fouling_gain` shrinking toward zero removes a slow time-correlated component of the dp signal. This is a harder-to-bound shift because the policy may have learned to use that slow correlation.

**Risk introduced:** **Temporal autocorrelation loss in dp_sensor_norm** if `dp_fouling_gain` calibrates to ≈0. The M8 obs8 policy trained under placeholder `dp_fouling_gain=0.5` saw a dp channel that slowly biased upward with `mesh_loading_avg`. Under calibrated near-zero fouling gain, that slow bias disappears and the dp channel becomes purely noise around true dp. A policy that had learned to use the slow-bias signature as an implicit fouling estimator will lose that feature. This is the single most specific way in which the +21.2% gap could narrow under calibration.

### Stage 4 — Policy input distribution shifts

**What changes:** The PPO policy's observation input is routed through `VecNormalize`, which maintains a running mean and variance over obs indices. During **fresh training** for the M9 calibrated artifact (proposed `ppo_hydrienv_v2_cal` or `ppo_hydrienv_v3`), a new `VecNormalize` instance is instantiated and adapts to the calibrated distribution. This is the intended path. During **evaluation of the existing `ppo_hydrienv_v2`** against the calibrated env, the saved `ppo_hydrienv_v2_vecnorm.pkl` running statistics are stale — the pre-trained policy sees a distribution shift it was not trained to handle.

**Why:** VecNormalize stores mean/variance per index; calibrated distribution has different statistics; existing policy's input normalization is no longer centered/scaled correctly.

**Expected magnitude:** Small per-index (a few percent shift in normalized input), but the **cumulative effect over 400 steps** on a trained policy can be significant because small systematic input shifts compound into consistent action biases. For a freshly trained policy the effect is zero (vecnorm adapts). For an existing-policy-on-new-env eval the effect is a meaningful but bounded policy drift.

**Risk introduced:** The protocol **must not** evaluate `ppo_hydrienv_v2` against a calibrated env using its saved vecnorm — that would be a methodologically invalid comparison. The M9 protocol is **new artifact under calibration**, not **old artifact under new env**. This aligns with M9.1 §7 ("retrain ppo_hydrienv_v2 under calibrated noise to produce a new canonical policy artifact") and the M9.1A.4 protected-lineage rules. The integration risk is that someone (or a future automation) invokes the existing eval script with the calibrated config and draws a misleading conclusion. **This needs to be blocked by the M9.3 execution document, not assumed.**

### Stage 5 — Policy behavior changes

**What changes:** The new policy trained under calibrated noise learns a different state→action map than the M8 placeholder-noise policy. Its internal representations reflect the new noise regime. Actuator trajectories (valve_cmd, pump_cmd, bf_cmd, node_voltage_cmd) will differ at the fine scale even when the macroscopic behavior (backflush timing, flow target) is similar.

**Why:** PPO optimizes against the reward landscape **conditional on the observed state distribution**. Same landscape + different observed state → different optimal policy. The difference can be small (if calibrated noise is similar to placeholder) or large (if calibrated noise is very different).

**Expected magnitude:** Uncertain in direction and magnitude; this is precisely the question M9 exists to answer. A priori, based on the likely direction of calibrated values (M9.1R.2 §3):
- **Lower dp_drift_rate → cleaner dp signal → potentially better obs8 policy** (more predictable dp channel)
- **Lower dp_fouling_gain → loss of slow mesh-loading correlate on dp channel → potentially worse obs8 policy** (loses an implicit fouling proxy)
- **Modestly changed `flow_noise_frac` and `flow_bias_std_lmin` → marginal change to flow-channel reliability**
- **Net direction on the +21.2% gap:** plausibly unchanged to moderately narrowed. A reversal (v2 < v1) is possible but would require the loss of the fouling-gain correlation to dominate the gain from cleaner dp noise — both effects are small relative to the 21.2% headroom.

**Risk introduced:** The M9 gap result may end up in any of the four scenarios from M8.3 §9. This is not a risk — it is the purpose of M9. The real risk is **overinterpretation of whichever direction emerges**, which is addressed in Section 9 below and must be protected by an M9 prohibited-claims list defined before results are known.

### Stage 6 — Reward response shifts

**What changes:** **Nothing, structurally.** Reward is computed at `env.py:304–308` from `truth_state` quantities: `q_processed`, `pressure` (normalized), `mesh_avg` (`mesh_loading_avg`), `q_bypass`, and `bf_active`. **None** of these are sensor channels. The reward function is **invariant** under sensor calibration.

**Why:** The reward-shaping design separates truth-side objectives from measurement-side observability. Sensor calibration changes what the policy **sees**, not what the environment **pays**.

**Expected magnitude:** Zero direct change. Indirect change is possible: if the calibrated policy takes different actions, those actions traverse different truth-state trajectories and therefore accumulate different reward. But the reward **mapping** is unchanged.

**Risk introduced:** **None** at the reward level. This is the strongest coherence anchor in the integration analysis. Because the reward is truth-derived, the M9 calibration pass cannot introduce reward misalignment. The previously identified "reward misalignment under new signals" failure mode (see Section 5) is **structurally ruled out** by the existing HydrOS architecture. This is a significant finding that reduces the effective risk surface of M9.

### Stage 7 — Evaluation metrics change

**What changes:** The M9 calibrated-noise policy's evaluation return vs the v1 baseline (694.639 at 500k steps, obs14_v1) and vs the pre-calibration v2 reference (841.839 at 360k steps, obs8). New metrics are computed by the existing eval script (`hydrion.eval_ppo_hydrienv_v2`). Four-scenario classification (M8.3 §9) is re-applied to the calibrated result.

**Why:** Eval scripts run deterministic rollouts through the same environment construction path. Under the calibrated config, the same eval pipeline produces calibrated-noise metrics.

**Expected magnitude:** The absolute return numbers will differ from the M8 placeholder-noise numbers. The comparison-of-interest is:
- **Calibrated v2 return** vs **v1 return (694.639)** — the headline deployment gap figure
- **Calibrated v2 return** vs **M8 v2 return (841.839)** — the within-schema comparison showing what calibration cost or gained relative to placeholder noise

**Risk introduced:** Two distinct risks:
1. **AC11 drift risk.** If the same config edit that feeds calibration values into v2 training is used for v1 re-evaluation, the v1 return will shift away from 694.639. AC11 is the environmental-stability gate for all cross-schema comparisons. Loss of AC11 exact match is a Category-A governance breach and is recoverable only by isolating the calibrated config from the v1 eval path. **This is the highest-priority integration risk and demands an explicit protection mechanism in M9.3.**
2. **Step-budget asymmetry preservation risk.** The M8 v2 ran 360k steps; if the M9 calibrated artifact runs at a different budget (either 360k for comparability to M8 v2, or 500k for comparability to v1), the comparison semantics must be declared in M9.3. A mixed-budget comparison without declaration would re-introduce the MVC-2 partial deviation under a new noise regime.

## 3. System Coherence Check

### truth_state ↔ sensor_state consistency

**Status: CONSISTENT.**

The sensor modules (`pressure.py`, `flow.py`, `optical.py`) follow a strict one-way read contract: `truth_state` is read, `sensor_state` is written. No sensor module writes to `truth_state` except for one legacy compatibility mirror in `optical.py` which writes back into `truth_state["sensor_turbidity"]` etc. for backward compatibility with pre-M6.2B callers. This mirror is a no-op for both `build_observation` (obs14_v1) and `build_observation_obs8`, both of which read the sensor fields from the dict they were given (the mirror produces the same value in both dicts). Under calibration, the separation holds — calibrated parameters only affect write-path values, never read-path sources.

### Sensor signals remain physically interpretable

**Status: MORE interpretable, not less.**

Before calibration, the four sensor channels are labeled as "noisy observations" but their noise characteristics do not correspond to any named device. After calibration, each channel corresponds to a nominated device class (per M9.1R.2 Section 2: piezoresistive dp transducer, electromagnetic flow meter, nephelometric turbidity sensor). A viewer can map the displayed value to a real instrument type with known operating characteristics. Interpretability monotonically improves.

**One caveat:** The `dp_fouling_gain` semantics question is resolved by code inspection (process B, sensor-side bias — `pressure.py:99–102`), but the calibrated magnitude is essentially zero for episode-scale simulation. This means the code retains a parameter that in the calibrated regime does nothing observable within an episode. This is acceptable (the parameter remains available for long-duration simulation should M10+ scope expand), but M9.3 should explicitly record that the episode-scale effective value is near zero, and M9.9 closeout should document that the parameter was intentionally retained at a small nonzero value for structural completeness, not for behavioral effect.

### Observation schema validity

**Status: VALID, unchanged.**

`obs8_deployment_v1` is a shape and index-ordering contract. M9 changes the numeric values flowing through indices 4–7; it does not change the index assignments or the schema dimensionality. The `_VALID_SCHEMAS` dict in `env.py` is untouched. Schema lock assertions at train and eval time continue to pass because the assertion checks shape and index mapping, not value distribution. The 17 obs8 schema tests in the 128/128 test suite will continue to pass (they are structural, not distributional). The backward-compatibility gate T5 (obs14_v1 default behavior unchanged) holds **only if** v1 eval is run against a config snapshot that matches v1's training config — see the config-coupling risk in Section 6.

### Policy inputs remain meaningful

**Status: MEANINGFUL, with one subtlety.**

For the **new** M9 calibrated policy, inputs are meaningful by construction — the policy is trained on the distribution it will be evaluated on. For the **pre-existing** M8 `ppo_hydrienv_v2`, its inputs cease to be meaningful if evaluated against a calibrated env through its saved vecnorm. This is why M9 must produce a new artifact, not re-evaluate the old one. The coherence of "policy inputs are meaningful" is therefore contingent on the artifact-versus-env pairing discipline that M9.3 must specify.

### Mismatches identified

| Mismatch | Severity | Resolution |
|----------|----------|-----------|
| `configs/default.yaml` is shared by v1, v2, and CCE eval paths | **HIGH** | M9.3 must snapshot the pre-M9 config or introduce a separate `configs/m9_calibrated.yaml` used only by the M9 training script, leaving `default.yaml` unchanged until a later "promotion" commit |
| `dp_fouling_gain` is architecturally legitimate (process B, confirmed in code) but calibrates to a near-zero value | LOW | Document in M9.3 and M9.9 that the parameter is retained for structural completeness; not a code change |
| Optical sensor compatibility mirror writes to both `truth_state` and `sensor_state` | LOW | Pre-existing legacy shim; M9-orthogonal; do not touch |
| VecNormalize stale-stats risk if pre-existing policies are naively re-evaluated | HIGH | M9.3 must explicitly prohibit re-evaluation of v1 or v2 against the calibrated config using their original vecnorm |

## 4. Policy Stability Risk

### Will the current policy degrade under calibrated noise?

**Indeterminate in direction, bounded in magnitude.**

The existing `ppo_hydrienv_v2` (360k obs8 policy) will not be used to run against calibrated noise — that comparison is invalid per the stale-vecnorm issue above. The relevant question is whether a **freshly trained** policy under calibrated noise will be weaker, equal, or stronger than the 841.839 reference.

**Factors favoring preservation or improvement:**
- Lower calibrated `dp_drift_rate` removes an intra-episode random-walk component that the M8 policy had to tolerate
- Lower calibrated `flow_noise_frac` (if it calibrates smaller) cleans up the flow channel
- The reward landscape is unchanged, so the learning target is the same

**Factors favoring degradation:**
- Loss of the `dp_fouling_gain` slow-correlation feature that may have functioned as an implicit fouling proxy
- Possibly increased `flow_bias_std_lmin` if the calibrated meter class has a wider factory-calibration tolerance than the placeholder
- Calibrated `dp_noise_kPa` could go either direction

**Net assessment:** The most likely outcome is a **bounded change** in either direction. Catastrophic degradation (policy failing to learn, or failing to match v1) is unlikely because the calibrated noise magnitudes are all within a small constant factor of the placeholder values — nothing in M9.1R.2's preliminary ranges suggests a qualitative regime change. However, this assessment is conditional: if the verified-source pass (M9.1R.3) returns values **very** different from current placeholders (e.g., `flow_bias_std_lmin` at 1.0 L/min instead of 0.2, a 5× increase), then degradation is more plausible.

### Behavior collapse risk

**Low.** Behavior collapse requires either pathological reward-signal loss or a regime change in observation that makes the policy's learned value function invalid. Neither applies here: reward is truth-derived (invariant); obs shape is unchanged; and the distribution shift is quantitative, not qualitative. PPO with standard hyperparameters and fresh vecnorm is well-conditioned against modest input distribution shifts during training.

### Gap reversal risk

**Low to moderate.** The +21.2% M8 gap has substantial headroom (21.2% is not a marginal effect). For a gap reversal (v2_cal < v1) to occur, the combined effect of calibrated noise on the obs8 policy would have to exceed 21.2% in the downward direction. The plausible adverse effect (loss of `dp_fouling_gain` slow correlation) is unlikely on its own to exceed that threshold because the dp channel is only one of four sensor channels and the policy retains three others plus the four actuator-feedback channels. Gap **narrowing** (from 21.2% to something between 0% and 21%) is the more likely direction. Gap reversal is possible but not the central expected outcome.

### Learning dynamics

**Similar, but not identical.** PPO's clipping, advantage normalization, and value-function loss all work with whatever distribution the environment presents. A modest shift in the sensor distribution should not change the learning dynamics qualitatively — the same PPO hyperparameters will converge on a similar timescale (360k–500k steps). The **shape of the learning curve** may differ slightly (the early-training noise floor is different), but the asymptote should be reached in the same budget.

**Cross-policy variance caveat:** The M9 calibrated artifact is still single-seed (seed=42) unless Priority 3 from M9.1A.5 is invoked. A one-seed comparison between M9 calibrated and M8 placeholder-noise v2 inherits the tertiary confound. This is acceptable as long as the M9.3 prohibited-claims list explicitly preserves the "no confidence interval" qualifier.

## 6. Hidden Dependency Detection

### Subsystems implicitly depending on placeholder noise

**Three identified:**

1. **`ppo_hydrienv_v1` (obs14_v1) eval path.** v1 was trained against placeholder-noise sensor channels at indices 12–13 (flow_sensor_norm, dp_sensor_norm). A v1 evaluation uses the default config to construct the env and then loads the saved vecnorm. If the config sensor values change, v1 eval returns drift. This is the AC11 drift risk (F1).

2. **`ppo_hydrienv_v2` (obs8_deployment_v1) eval path.** Same mechanism as v1 but even more sensitive because the policy has no physics-truth channels to fall back on. Four of its eight observation channels are sensor-derived.

3. **`ppo_cce_v2` (obs12_CCE) CCE eval path.** CCE uses a truth-derived observation (no sensor channels in obs12_CCE per M9.1 §1), but `ConicalCascadeEnv` still instantiates the sensor pipeline if it constructs the same sensor modules. Need to verify: if CCE does not reference the `sensors:` config block at all, CCE is isolated from M9 risk; if it does, the AC11-equivalent CCE stability gate is also at risk. **Action required in M9.3:** confirm `ConicalCascadeEnv` does or does not read `configs/default.yaml sensors.*`. This is a single-file audit, not a design question.

### Assumptions that may break under calibration

| Assumption | Where it lives | How it could break |
|-----------|---------------|---------------------|
| `flow_bias_std_lmin = 0.2` is small enough that per-episode bias doesn't dominate flow obs | `flow.py` (`reset()` samples once per episode) | If calibrated value is much larger (e.g., ±1 L/min), episode-to-episode variance in the flow channel becomes a major source of input noise that the policy cannot average over within a single episode |
| `dp_drift_max_kPa = 2.0` is a conservative cap that is rarely reached | `pressure.py` `_drift_max` | If calibrated drift rate increases enough that the cap is reached in a single episode, the dp channel develops a biased saturation the policy hasn't seen. Highly unlikely under preliminary M9.1R.2 ranges. |
| `dp_latency_steps = 1` (≈100 ms at dt=0.1) matches real MEMS latency | `pressure.py` `_latency` | If the calibrated device class has multi-step latency, the buffer length must grow. A change from 1 to 2 steps changes the phase relationship between action and observation — this is a non-trivial environment change, not a parameter tweak. **M9.3 must flag any latency change as a structural change, not a parameter change.** |
| The optical sensor's `turbidity_gain_clog = 0.8` coupling to `mesh_loading_avg` is grounded | `optical.py` `OpticalSensorParams` | M9.1R.2 §2.3 explicitly flags the NTU-to-microplastic-mass mapping as not literature-groundable. If M9 attempts to calibrate this gain coefficient, it enters experimental-correlation territory. M9 should **not** modify the optical gain coefficients — only the noise standard deviations. This boundary must be explicit in M9.3. |
| `configs/default.yaml` is the single source of sensor truth for all policies | Runtime config loader | As noted: this shared config is the main coupling hazard |

### Places where calibration could have unexpected effects

- **VecNormalize statistics file drift.** If calibrated training produces a new vecnorm file and someone ever mixes it with the pre-calibration policy zip (or vice versa), the result is nonsensical. M9.3 must enforce paired artifact loading — which is already the M8 discipline per the M9.1A.4 "named constant VecNorm path separation" protection. M9 should extend this pattern with new constants (`_V2_CAL_VECNORM_PATH` or similar).
- **Logging / telemetry pipelines.** The env.py info-dict includes `dp_sensor_kPa` and `flow_sensor_lmin` as scalars for logging (env.py:346–347, 389–390). Any analysis tool consuming these logs will see different distributions after calibration. This is not a breaking change — the field names are stable — but analysis scripts may need a provenance label.
- **`sensor_state` mirror in the optical sensor compatibility shim.** Calibrated optical noise values flow into both `sensor_state` and `truth_state` through the shim. Nothing currently reads `truth_state["sensor_turbidity"]` for a policy input (obs8 reads from `sensors` dict; obs14_v1 also reads from `sensors` dict in `sensor_fusion.py`). So the mirror is calibration-safe, but it is an additional line of defense that should be preserved unchanged.

## 7. System Behavior Validity

### Could the system still behave "correctly" but for wrong reasons?

**Yes, and this is one of the diagnostic goals of M9.** The M8 +21.2% deployment gap is a correct result under the training distribution — but its *interpretation* as hardware-relevant depends on the sensor noise being hardware-like. Under calibration, the same system may produce a similarly positive result (same apparent correctness) because the obs8 policy has real signal content, OR it may produce a much smaller or reversed result because the placeholder-noise regime was favorable. Either outcome is more honest than the placeholder-noise result. The post-M9 system behavior is **more trustworthy** regardless of whether the numeric outcome is better or worse than M8.

**The "wrong reasons" scenario specifically:** The M8 obs8 policy may have been implicitly using the `dp_fouling_gain × mesh_loading_avg` slow drift as a proxy for fouling state, effectively reconstructing a physics-truth channel from sensor data. Under calibration with near-zero fouling gain, that reconstruction channel disappears. If the gap narrows sharply post-calibration, this is the likely explanation — and it is a scientifically meaningful finding. The failure mode is only if the finding is misreported as "the obs8 architecture is invalid" rather than "the obs8 architecture was partly benefiting from a placeholder-noise artifact."

### Could calibration expose previously hidden issues?

Yes. Three specific issues could surface:

1. **`dp_fouling_gain` semantic confirmation.** Already resolved in this analysis: code inspection confirms process B (sensor-side bias). No issue here, but the analysis exposes that the current placeholder value of 0.5 kPa at `mesh_loading_avg=1.0` is an order of magnitude larger than realistic episode-scale sensor-line fouling. This means the M8 obs8 policy saw a placeholder dp channel with a **larger-than-real** slow upward bias when fouling accumulated. This is one of the specific ways placeholder noise could have been "favorable."
2. **Flow meter calibration bias distribution.** The current `flow_bias_std_lmin = 0.2` is a per-episode DC offset — the policy sees a consistent bias within an episode and a mean-zero bias distribution across episodes. This is an unusually clean noise regime. Real per-episode flow calibration error is likely in the same ballpark (M9.1R.2 §3.3: 0.5–2% of reading) but verifying this is a M9.1R.3 question.
3. **Optical sensor noise vs gain-coefficient grounding gap.** M9 should only touch noise std values, not the gain coefficients (`turbidity_gain_clog`, `turbidity_gain_particles`, etc.). If M9 attempts to calibrate those, it oversteps into the turbidity-to-mass mapping that M9.1R.2 flags as experimentally-groundable-only. M9.3 must enforce this boundary.

### Does calibration increase or decrease interpretability?

**Increases, monotonically.** Every sensor channel moves from "noisy observation with unnamed parameters" to "observation with provenance-tagged parameters corresponding to a nominated device class." Downstream interpretation — of the deployment gap, of policy behavior, of telemetry renderings — gains a definite meaning it did not previously have. The interpretability gain is the primary structural benefit of M9 and is the reason the M9.1A.2 assessment correctly identified calibration as the single highest-leverage move.

5. **Calibrated v2 vs v1 delta** — the headline gap figure, with four-scenario classification applied before interpretation
6. **Per-episode variance** — if episode return variance rises sharply, calibrated noise may be too large for the current training budget

- **Entropy staying in PPO's healthy range** (not collapsing, not exploding)
