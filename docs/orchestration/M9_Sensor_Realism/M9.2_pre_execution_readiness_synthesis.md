## 1. Executive Synthesis Verdict

**M9 direction is unchanged and now reinforced by three independent diagnostic passes.** M9.1A (maturity + alignment), M9.1B (reality adherence), and M9.1C (integration consistency) all converge on sensor realism & calibration as the correct next move, and all produce **refinements** rather than contradictions. 1. **Parameter scope is narrower than M9.1R.2 alone suggested.** Code-level inspection (M9.1B §5) establishes that `flow_bias_std_lmin`, `dp_noise_kPa`, and `dp_latency_steps` are **already grounded** to peer-reviewed sources. The parameter is **process B** (sensor-side bias ∝ `mesh_loading_avg`), not process A (double-count of filter-cake physics). ## 4. Resolved Questions

### 4.1 `dp_fouling_gain` semantics — RESOLVED (process B)

M9.1R.2 §3.2 / §4.2 flagged this as a semantic ambiguity that could be process A (filter-cake fouling already modeled in truth state) or process B (sensor-side bias from line/diaphragm fouling). M9.1C §2 Stage 1 and §3 resolve the ambiguity by code inspection of `hydrion/sensors/pressure.py:99–102`: the implementation applies the gain as `fouling_offset = dp_fouling_gain · mesh_loading_avg` added to the sensor output, reading `mesh_loading_avg` from `truth_state` and writing only to `sensor_state`. This is **process B** — a legitimate sensor-error term, not a double-count.

Consequence: `dp_fouling_gain` stays in the noise model. Its calibrated magnitude is expected to be near-zero on episode timescales per M9.1R.2 §3.2 (episode-scale sensor-line fouling accumulates negligibly). M9.3 and M9.9 should document the parameter as retained for structural completeness (future multi-hour simulation support), not for intra-episode behavioral effect.

### 4.2 Reward invariance under sensor calibration — CONFIRMED

M9.1C §2 Stage 6 establishes by code inspection of `env.py:304–308` that reward is computed from `truth_state` quantities (`q_processed`, `pressure`, `mesh_avg`, `q_bypass`, `bf_active`) and **does not read any `sensor_state` field**. Sensor calibration therefore cannot affect the reward signal, the reward shape, or the optimization target.

Consequence: M9 calibration cannot induce reward misalignment. The "reward misalignment under new signals" failure mode is structurally ruled out, not just governance-managed. This is a finding worth preserving for future milestones: **reward-touching milestones fall outside this protection** and must be integration-audited on a different basis.

### 4.3 `flow_calibration_bias_std` grounding status — ALREADY GROUNDED

M9.1A and M9.1R.2 named `flow_calibration_bias_std` as a placeholder calibration target. M9.1B §5 establishes that the parameter (code name `flow_bias_std_lmin = 0.2`) is already inside the mag-meter factory-calibration tolerance spec (±0.5–1% FS at 15 L/min FS ≈ ±0.075–0.15 L/min; 0.2 is a conservative upper bound). The naming mismatch between roadmap and code is cosmetic; the parameter is the same.

Consequence: M9 does not need to recalibrate the flow bias magnitude. M9.3 should record the conservative-bound rationale and move on. The research-brief effort allocated to flow calibration can be redirected to the optical layer.

### 4.4 Architecture change requirement — NONE

All three branches agree that M9 does not require any architectural change. No new files, no new schemas, no new obs indices, no new env classes. The `obs_schema` selector, `build_observation_obs8()`, `HydrionEnv`, `ConicalCascadeEnv`, `_VALID_SCHEMAS`, VecNorm path constants, and the 128/128 test suite all remain unchanged.

The only in-repo edits M9 should make:
1. New `configs/m9_calibrated.yaml` (new file, additive)
2. New `hydrion/train_ppo_hydrienv_v2_cal.py` (new file, additive) — or a `--config` argument on the existing script, equivalent effect
3. Optional removal of the `optical.py:144` `truth_state` mirror (SA2 above) — zero-impact cleanup gated on 128/128
4. New metadata file `models/ppo_hydrienv_v2_cal_meta.json` recording calibrated values, provenance, confidence, and calibration-pending items
5. New trained artifact `models/ppo_hydrienv_v2_cal.zip` + `_vecnorm.pkl`
