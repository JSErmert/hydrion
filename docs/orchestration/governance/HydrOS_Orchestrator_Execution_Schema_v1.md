## 1. Identity and Purpose

This schema is the formal execution reference for HydrOS milestone orchestration work.

It defines how HydrOS milestone artifacts should be structured, sequenced, named, stored,
inherited, and returned. It is not a milestone-specific document. It is a reusable template
that makes future milestone prompt generation faster, more consistent, and more governance-safe.

**Why it exists:**
The M6–M8 orchestration chain produced a repeatable, high-value pattern of structured
prompts, outputs, insights, and handoffs. That pattern was executed by trial-and-refinement
across three milestones. This schema formalizes it — not to freeze it, but to make it
available as a generation framework for future work without rediscovering it from scratch.

**What it covers:**
- Artifact classes and their roles
- Phase sequencing
- Identity fields per artifact
- Runtime class system
- Inheritance structure
- Truth-status and confound vocabulary
- Visual validity requirements
- Manual-runtime handoff protocol
- Return formatting
- Storage discipline

**What it does not cover:**
Physics, control theory, sensor modeling, RL methodology, or HydrOS-specific design
decisions. Those are governed by Contract v3.1, the research layer, and execution documents.

## 3. Reference-First Usage Status

**This schema is reference-first, not mandatory-first.**

- It is available in-repo as an orchestration reference for all future milestone work.
- It is intended to be consulted when generating milestone prompts, sequencing artifacts,
  structuring execution reports, formatting returns, and designing handoffs.
- It is not yet a rigid dependency. Future milestones are not required to cite it formally
  in every artifact until its value has been validated through repeated use.
- It will be considered for promotion to a stronger mandatory-dependency role after M9
  and at least one additional milestone have demonstrated whether it improves sequencing
  quality in practice.

The correct posture: use it when it helps; note when it is being applied; refine it when
practice reveals gaps or unnecessary constraints.

## 4. Recursive Self-Application Role

This schema is an instance of its own pattern. It was:
1. Drafted from the M6–M8 artifact lineage (inputs = historical prompt chain)
2. Structured using the same artifact-class vocabulary it defines
3. Committed with an insight artifact per its own insight-generation rule

This is its recursive property: **the schema can generate future prompts of the same
class as the prompts that generated it.**

Applied intentionally, this means:
- A future M9 refinement prompt can be generated by populating the Refinement Prompt
  template in Section 5 with M9-specific inputs.
- A future closeout prompt can be generated by populating the Closeout Prompt template
  with the relevant milestone's governing artifacts.
- An entirely new milestone chain can be seeded by filling in the Artifact Identity
  Schema (Section 7) for the new milestone and then generating each phase prompt from
  the standard class definitions in Sections 5–6.

**Governed limits of recursive self-application:**
- It does not produce new milestones autonomously. The user defines the milestone goal.
- It does not change the governing contract. Philosophical guardrails are Contract v3.1.
- It does not override domain-specific decisions. Physics, RL methodology, and sensor
  design are governed by research artifacts and execution documents, not this schema.

## 5. Standard Artifact Classes

Each HydrOS orchestration artifact belongs to one of the following classes.
Artifact numbering convention: prompts use `.N_prompt.md` suffix or even-number prefix;
outputs use descriptive names or odd-number prefix; insights live in
`docs/orchestration/insights/<Milestone>/`.

---

### Refinement Prompt
**Path convention:** `<Milestone>/<Milestone>.0_refinement_prompt.md`
**Purpose:** Activates a new milestone. Asks: what is the true objective? What is the
right scope? What are the candidate approaches? What should and should not be attempted?
Produces the refinement output.

---

### Refinement Output
**Path convention:** `<Milestone>/<Milestone>.1_refinement_output.md`
**Purpose:** The authoritative milestone definition output. Records the AlignFlow See →
Understand → Protect → Act analysis. Defines the milestone title, scope, candidate
branches, chosen branch with rationale, and carry-forward risks. Governs all downstream
M.N artifacts in this chain.

---

### Research Brief
**Path convention:** `<Milestone>/<Milestone>.1R.2_research_brief.md`
(odd number, `.R` suffix marks research layer)
**Purpose:** Documents peer-reviewed and authoritative literature relevant to the
milestone design decisions. Labels source-derived claims separately from internal
architecture decisions. Produces source-grounded justification for key choices.

---

### Sources Map
**Path convention:** `<Milestone>/<Milestone>.1R.4_sources_map.md`
**Purpose:** Full source traceability record. Lists every source used in the research
brief with citation type (PJ/PC/PR/TR/ID), what each source supports, and which M
design decisions each source informs. Governs claim validity boundaries for the
milestone.

---

### Execution Document
**Path convention:** `<Milestone>/<Milestone>.3_execution_document.md`
(odd number — output)
**Purpose:** Authoritative milestone scope document. Defines: system state inherited,
formal artifact definitions (schemas, names, structures), acceptance criteria (ACs),
methodology validity conditions (MVCs), interpretation rules, prohibited claims, and
protected artifacts. This document binds all downstream implementation.

---

### Implementation Plan
**Path convention:** `<Milestone>/<Milestone>.5_implementation_plan.md`
**Purpose:** Converts execution intent into file-by-file, step-by-step implementation
reality. Specifies exact code changes, test checkpoint positions, schema lock locations,
and artifact save paths. The last document before execution begins.

---

### Execution Prompt
**Path convention:** `<Milestone>/<Milestone>.6_execution_prompt.md`
**Purpose:** The activation document that initiates the implementation phase. References
the implementation plan directly. Specifies the runtime class, exact commands if manual
runtime is required, and what to return after execution.

---

### Execution Report
**Path convention:** `<Milestone>/<Milestone>.7_execution_report.md`
**Purpose:** Records exactly what happened during execution. States which phases
completed, which are pending, actual vs expected results, test checkpoint outcomes,
AC results, MVC status, and closeout readiness. Updated post-runtime if training or
evaluation was deferred.

---

### Runtime Handoff / Runtime Update
**Path convention:** Section within execution report, or `<Milestone>.7b_runtime_update.md`
**Purpose:** Used when execution has a manual-runtime phase that generates new results.
Updates execution report sections that were PENDING before runtime. Records actual
training output, eval metrics, AC outcomes, and scenario classification.

---

### Closeout Prompt
**Path convention:** `<Milestone>/<Milestone>.8_closeout_prompt.md`
**Purpose:** Activates the formal milestone closeout pass. Specifies what the closeout
record must contain and which governing artifacts to use as inputs.

---

### Closeout Record
**Path convention:** `<Milestone>/<Milestone>.9_closeout_record.md`
**Purpose:** The authoritative milestone record. States formal closeout status, what was
implemented, what was observed, what was established, what was not established,
step-budget deviations, protected-lineage confirmation, and next-milestone inheritance.
This is the document that formally closes a milestone.

---

### Achievements Prompt
**Path convention:** `<Milestone>/<Milestone>.10_achievements_prompt.md`
**Purpose:** Activates the achievements summary pass. Distinct from closeout — intended
to produce a reusable, high-signal summary rather than a governance record.

---

### Achievements Document
**Path convention:** `<Milestone>/<Milestone>_achievements.md`
**Purpose:** Concise, reusable milestone-value summary. Covers: what was built, what was
proved, what was corrected, what was preserved, what was unlocked. Includes resume/portfolio
summary bullets. Does not repeat the full closeout record. Designed for later reuse without
re-reading the full milestone chain.

---

### Insight Artifact
**Path convention:** `docs/orchestration/insights/<Milestone>/<Milestone.Iteration>_insight_<short_slug>.md`
**Purpose:** Captures one discrete thing that became more true, more clear, or more
actionable because of this pass. Not a summary — a learning artifact. Required after
every major orchestration pass. Structure: Title, Context, What was learned, Why it
matters, Immediate system implication, Workflow implication, Recommended carry-forward
action.

## 6. Standard Phase Sequencing

The default HydrOS milestone phase sequence is:

```
Phase 0 — Refinement
  → .0_refinement_prompt.md
  → .1_refinement_output.md
  → Insight

Phase 1 — Research (when needed)
  → .1R.2_research_brief.md
  → .1R.4_sources_map.md
  → Insights

Phase 2 — Execution Document
  → .3_execution_document.md (+ optional prompt at .2)
  → Insight

Phase 3 — Implementation Plan
  → .5_implementation_plan.md (+ optional prompt at .4)
  → Insight

Phase 4 — Execution
  → .6_execution_prompt.md
  → .7_execution_report.md
  → [Runtime handoff if manual_runtime_required]
  → Insight

Phase 5 — Closeout
  → .8_closeout_prompt.md
  → .9_closeout_record.md
  → Insight

Phase 6 — Achievements
  → .10_achievements_prompt.md
  → <Milestone>_achievements.md
  → Insight
```

**Not every milestone uses every phase.** The research layer (Phase 1) is required when
the milestone involves physics, sensor modeling, RL methodology, or other realism-critical
claims. It may be abbreviated for purely architectural or governance milestones.

**Phase sequencing is a default template, not a rigid requirement.** A milestone may
skip the implementation plan phase if the execution is trivially constrained by a
single-file change. A milestone may merge closeout and achievements into one pass if the
result set is simple. The schema provides the full template; judgment determines which
phases are applied.

## 7. Artifact Identity Schema

Every major orchestration artifact should include the following identity fields
in its header or frontmatter:

```markdown
**Document type:** <artifact class>
**Milestone:** <milestone ID and name>
**Iteration:** <M.N — the pass number>
**Phase:** <refinement | research | execution_document | implementation_plan |
           execution | closeout | achievements | governance>
**Prompt path:** <path to the prompt that generated this artifact, if applicable>
**Output path:** <path to this artifact>
**Insight path:** <path to the insight created for this pass, if applicable>
**Governing contract:** docs/orchestration/governance/HydrOS_x_AlignFlow_Orchestration_Contract_v3.1.md
**Governing schema:** docs/orchestration/governance/HydrOS_Orchestrator_Execution_Schema_v1.md
**Status:** <draft | complete | pending_runtime | post_runtime_update | closed>
**Runtime class:** <fully_autonomous | autonomous_until_manual_runtime |
                   manual_runtime_required | post_runtime_update_required>
```

Governance artifacts (contract, schema) omit milestone and iteration fields.
Insight files use frontmatter (milestone, iteration, slug) per Contract v3.1 Section 11.

## 8. Runtime Class System

Every execution-phase artifact must declare a runtime class. This class governs how the
execution prompt is written, how the execution report handles pending sections, and what
the handoff to the user looks like.

---

### `fully_autonomous`
All phases can be completed without user intervention.
- No training runs, external commands, or user-pasted output required.
- Execution report can be written complete in a single pass.
- **Example:** Schema extension implementation with test-only validation.

---

### `autonomous_until_manual_runtime`
All code, tests, and infrastructure phases complete autonomously. One or more phases
require a user-run terminal command (typically a long training run) that exceeds the
autonomous execution ceiling.
- Execution prompt must specify the exact manual command(s) with `!` prefix.
- Execution report is written with PENDING markers in runtime-dependent sections.
- After user runs the command and pastes output (or completes the run), the report is
  updated via a Runtime Update pass.
- **Example:** M8 — implementation complete, training requires `! python -m hydrion.train_ppo_hydrienv_v2`.

---

### `manual_runtime_required`
The primary deliverable requires user-driven execution from the start. No meaningful
autonomous execution precedes the manual step.
- Execution prompt leads with the user command.
- No PENDING sections — the report is written only after runtime results are available.
- **Example:** A hardware-in-the-loop validation pass.

---

### `post_runtime_update_required`
An execution report exists and is partially complete. This class is declared when
re-entering a milestone after the manual runtime phase completes.
- Governing input is the existing execution report plus runtime output.
- All PENDING sections are updated with actual results.
- Closeout eligibility is assessed after this update.
- **Example:** M8.7 update after `python -m hydrion.eval_ppo_hydrienv_v2` produced results.

---

### Runtime class in planning

When writing an implementation plan or execution prompt, state the runtime class
explicitly and include:

For `autonomous_until_manual_runtime`:
```
Runtime class: autonomous_until_manual_runtime
Manual runtime required after Phase D:
  Command: ! python -m hydrion.train_ppo_hydrienv_v2
  Expected output: [SCHEMA LOCK OK] + training progress + final model paths
  Execution ceiling note: ~800s required; bash ceiling = 600s; full budget run is manual
  Post-runtime: update M.7 Sections 3–6 with actual results
  Success criterion: models/ppo_hydrienv_v2.zip + _vecnorm.pkl + _meta.json all present
```

## 9. Inheritance Input Structure

Every artifact must specify what it inherits. The inheritance structure prevents
orphaned artifacts and makes milestone chain traceability explicit.

**Standard inheritance fields in artifact headers:**

```markdown
**Governing inputs:**
- <path to refinement output>         — defines milestone scope
- <path to execution document>        — defines ACs, MVCs, prohibited claims
- <path to implementation plan>       — defines file-level changes
- <path to execution report>          — records runtime results
- <path to closeout record>           — records formal milestone status
- <path to relevant insight files>    — carries forward strategic learning
- <any protected artifacts>           — confirms what must not be modified
```

**Minimum inheritance per phase:**

| Phase | Minimum inherited inputs |
|-------|-------------------------|
| Research | Refinement output |
| Execution document | Refinement output + research brief + sources map |
| Implementation plan | Execution document |
| Execution | Execution document + implementation plan |
| Closeout | Execution document + execution report + all insights |
| Achievements | Closeout record + all insights |

## 10. Truth-Status and Validity-Status Structure

All major milestone documents should classify claims using the following vocabulary.
This vocabulary should be surfaced explicitly in execution documents, implementation
plans, execution reports, and closeout records.

| Status label | Meaning |
|-------------|---------|
| `source-grounded` | Derived from peer-reviewed or authoritative literature. Citation present in sources map. |
| `internal architecture decision` | Not derived from literature. Made within HydrOS. Must be labeled explicitly and not presented as externally justified. |
| `implemented` | Specified design decision has been written into code or configuration. Not yet runtime-tested. |
| `runtime-verified` | Confirmed correct at actual runtime (test pass, schema lock fired, eval output recorded). |
| `confounded` | Result is valid under current conditions but its interpretation is bounded by one or more named confounds. |
| `unresolved` | Open question, assumption, or limitation that has not yet been addressed. Should be carried forward explicitly. |
| `calibration-pending` | Parameter or model uses a placeholder value. Result validity is bounded by the uncalibrated parameter. |

**Application rule:** When a claim in an execution document, execution report, or
closeout record could be misread without a truth-status qualifier, add one explicitly.
Do not leave truth-status implicit.

## 11. Visual Validity Bridge

HydrOS milestones M6 and beyond must be evaluated not only for backend realism and
control performance, but also for their effect on the future visual simulation console's
**visual validity**.

**Definition — Visual validity:**
The degree to which future rendered telemetry, truth-vs-sensor comparisons,
policy-visible observations, and action traces are scientifically grounded,
interpretable, and not misleading to an observer of the console output.

**Requirement:**
Every milestone refinement output must include an assessment of whether the milestone:

1. Improves future truth-vs-sensor visualization fidelity
   (e.g., calibrated sensor channels produce renderable, meaningful time series)

2. Improves the scientific meaning of displayed telemetry
   (e.g., a channel that was calibration-pending and displayed becomes grounded)

3. Reduces confounds that would make a future visual console misleading
   (e.g., a placeholder noise parameter that produces implausible sensor traces
   is removed or bounded before it reaches the console)

4. Strengthens the legitimacy of future policy-view and action-trace rendering
   (e.g., the obs8 policy's action traces are now interpretable because its
   observation channels are deployment-realistic, not simulation-privileged)

**Visual validity is not a cosmetic concern.** A console that renders scientifically
ungrounded data is not just inaccurate — it creates false confidence in system behavior.
The visual console must be as governed as the backend.

## 12. Confound-Priority Structure

Every milestone refinement output should explicitly rank the confounds it inherits:

```markdown
## Inherited Confounds

**Primary confound (address first):**
<Name and description — this is the confound that most directly limits the
interpretability or deployment relevance of current results>

**Secondary confound:**
<Name and description — meaningful but addressable only after the primary>

**Tertiary confound(s):**
<Name and description — lower priority; may persist across multiple milestones>

**Recommended confound-reduction target for this milestone:**
<State which confound this milestone's work will reduce, and by how much>
```

**Rationale:** Confounds that are named but not ranked tend to be silently inherited
without a clear next-action. Ranking forces an explicit decision about which uncertainty
to reduce first, which is the core question for milestone prioritization.

## 13. System-Model Change Requirement

Major refinement outputs and closeout records must explicitly state:

```markdown
## System-Model Changes

**What changed in the system model:**
<What is now represented differently — observation schema, channel taxonomy,
sensor noise model, reward function, etc.>

**What assumption was corrected:**
<What prior assumption or metadata imprecision was superseded by this milestone>

**What future milestones must no longer assume:**
<Explicit list of things that were previously permissible to assume but are
now known to be incorrect or insufficient>
```

**Example from M8:**
- Changed: obs14_v1 channel taxonomy (binary "truth vs sensor" → three-class
  physics_truth / actuator_feedback / sensor_derived)
- Corrected: indices 6–9 labeled "privileged" — now classified as actuator_feedback
  (deployment-available)
- Must not assume: "remove truth channels 0–9" is the correct deployment ablation target;
  correct target is channels 0–5 only

## 14. Manual-Runtime Handoff Pattern

When a milestone has runtime class `autonomous_until_manual_runtime`, the execution
prompt must include a clearly delimited manual-runtime handoff block:

```markdown
## Manual Runtime Required — Phase [N]

**Runtime class:** autonomous_until_manual_runtime

**Command(s) to run in terminal:**
```
! python -m <module.name> [--args]
```

**Expected output indicators (confirm these before proceeding):**
- [specific line or artifact confirming success]
- [artifact paths expected to exist after run]

**Execution ceiling note:**
<State the bash ceiling, estimated runtime, and why this exceeds the ceiling>

**What to paste back / confirm:**
After running, confirm one of:
(a) Paste the final output block here for automated report update, OR
(b) Confirm artifacts exist: `ls models/<artifact>*`

**What report is updated after runtime:**
`<Milestone>.7_execution_report.md` — Sections [N] through [M] will be updated
with actual results once runtime confirmation is received.

**Success criterion:**
<Specific, verifiable condition — e.g., "models/ppo_hydrienv_v2.zip exists and
eval output shows [SCHEMA LOCK OK] for both v2 and v1">

**Failure condition:**
<What constitutes a failed run — e.g., "schema lock assertion fires, missing
artifact, AC11 halt rule triggered">
```

## 15. Standard Return Structure

Every orchestration prompt must specify a return block at the end. The standard format:

```markdown
## Required Return

Return:
1. <prompt path>
2. <primary output path>
3. <insight path>
4. <short summary — one to three sentences: what was produced and what it established>
5. <next artifact — what the user should expect or activate next>
```

**For runtime-dependent milestones, return item 5 becomes:**
```
5. manual runtime required — run: ! python -m <module>
   then return to activate the post-runtime report update
```

**Return items must be in this order.** The short summary (item 4) must be specific
and honest — not promotional. It should be readable as a one-line status update for
the milestone log.

```
  <Milestone>.8_closeout_prompt.md

  <Milestone>.10_achievements_prompt.md
  <Milestone>_achievements.md
```

**Insight artifacts:**
```
docs/orchestration/insights/<Milestone>/
  <Milestone.Iteration>_insight_<short_slug>.md
```

**Governance artifacts:**
```
docs/orchestration/governance/
  HydrOS_x_AlignFlow_Orchestration_Contract_v3.1.md
  HydrOS_Orchestrator_Execution_Schema_v1.md
  HydrOS_Orchestrator_Execution_Schema_v1_prompt.md
```

**Chat echo requirement (per Contract v3.1 Section 15):**
After every markdown artifact is written, print the complete file in chat as a single
fenced code block (` ```markdown ... ``` `). This provides a copy-pasteable backup
and allows immediate inspection. The echo must be materially identical to the stored
file. It must be complete — not summarized or broken into sections.

**Git commit discipline:**
- Prompt files, output files, and insight files should be committed together per
  orchestration pass.
- `models/*.zip` and `models/*.pkl` are gitignored (binary artifacts).
- `models/*_meta.json` is committed (structured metadata).
- Commit messages should follow: `docs(<Milestone>): <pass description> — <key result>`

## 17. M6–M8 Proven Pattern Note

This schema is a direct formalization of the orchestration pattern proven across
milestones M6 through M8.

That pattern:
- Used alternating prompt/output numbering (`.N_prompt.md` / `.N+1_output.md`) with
  `.R` suffixes for research-layer artifacts
- Stored insights under `docs/orchestration/insights/<Milestone>/` and generated them
  at each major pass
- Separated research (literature grounding) from execution documents (binding scope)
  from implementation plans (file-level precision)
- Identified the `autonomous_until_manual_runtime` class when training exceeded the
  bash execution ceiling (~600 seconds) and handled it via explicit user-run handoffs
- Used closeout records to state formal milestone status and what the next milestone
  inherits, separately from achievements documents which serve reuse and portfolio use
- Enforced prohibited-claim discipline through eval script output headers, not just
  documentation language

Each of these patterns was discovered by executing a milestone, observing what worked
and what caused confusion, and refining the next pass. The schema formalizes the result.

It is being introduced at M8 close because that is the first point where three complete
milestone chains exist as evidence. Future work may refine it.

## 18. Validation-Through-Use Note

This schema is v1. It has not yet been applied prospectively to a new milestone.

M8 was the derivation source; M9 will be the first validation test.

**Expected validation criteria for promotion to mandatory-dependency status:**
- M9 execution prompts generated using this schema produce artifacts that require fewer
  mid-pass corrections than M8 prompts did
- The runtime class declaration prevents at least one planning ambiguity in M9
- The confound-priority structure in M9's refinement output improves the precision of
  M9's scope decision compared to M8's refinement output
- The artifact identity schema fields are populated correctly without additional guidance
  for at least two M9 artifacts

If M9 demonstrates these improvements, consider updating this schema to v2 after M9
closes — incorporating any M9-specific refinements and reducing any sections that
proved unnecessary overhead in practice.

If M9 reveals that sections of this schema add friction without value, revise or remove
those sections rather than preserving them for completeness.

The goal is not a complete schema. The goal is a useful one.
