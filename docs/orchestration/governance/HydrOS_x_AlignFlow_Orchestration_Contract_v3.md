## 1. Core Operating Principle

HydrOS work must always be interpreted through the AlignFlow loop:

**See → Understand → Protect → Act → Stabilize → Insight / Improve**

This is the default reasoning and execution structure for all future HydrOS work.

### See
Identify the real current state of the system:
- relevant files
- active schemas
- active benchmark artifacts
- sensor paths
- truth paths
- frontend/backend boundaries
- milestone status
- open gaps
- accepted limitations

### Understand
Interpret what that state means:
- physically
- architecturally
- operationally
- in relation to realism
- in relation to benchmark lineage
- in relation to deployment relevance
- in relation to the current milestone goal

### Protect
Preserve what must not be broken:
- benchmark lineage
- schema/version discipline
- truth_state vs sensor_state separation
- CCE integrity
- accepted scope boundaries
- simulation-only claim discipline
- artifact naming consistency
- milestone documentation integrity

### Act
Determine and execute the next correct scoped action:
- research
- execution document
- implementation plan
- implementation
- audit
- validation
- closeout
- handoff

No action should be taken unless it is the correct next action for the current milestone state.

### Stabilize
Produce outputs that leave the system cleaner, more governed, and easier to continue:
- explicit documents
- verified artifacts
- pass/fail checks
- corrected assumptions
- closeout records
- milestone handoff clarity

Every HydrOS action should improve system intelligibility, not just add code.

### Insight / Improve
After major orchestration passes, extract what the system and workflow learned:
- what became more true
- what became clearer
- what should be preserved
- what should be changed
- what should be prioritized next

The system should not only complete work. It should also improve how future work is done.

### Realism discipline
Do not overclaim:
- simulation performance is not hardware proof

- policy success does not imply deployment readiness

### Separation discipline
Preserve:
- truth_state vs sensor_state distinction
- backend vs frontend distinction
- simulation truth vs policy perception distinction
- benchmark path independence when required

### Scope discipline
Do not broaden milestones casually.
If the user is in:
- research
- planning
- implementation
- audit
- closeout

stay in that phase unless there is a strong reason to change.

## 5. Research Standard for HydrOS

HydrOS research must default to high-quality, trusted, technically relevant sources.

### Preferred source order
1. Peer-reviewed scientific journal papers
2. Peer-reviewed conference papers
3. Authoritative technical standards or reference documents
4. High-quality preprints only when clearly labeled and when they add genuine value
5. Vendor specifications only for implementation constraints or device characteristics, never as stand-alone scientific proof unless explicitly identified as such

### Research quality rules
- Prefer primary sources over summaries
- Prefer sources with clear technical methodology
- Prefer sources directly relevant to the HydrOS physical, sensing, control, or simulation problem
- Do not use weak secondary commentary as the basis for technical design claims
- Do not convert vague literature into overconfident design conclusions
- If literature is thin, say so explicitly

### Required research outputs
When research is performed for a HydrOS milestone, it should usually produce:
- a research brief
- a source map / traceability record
- clear separation between source-derived claims and internal architecture decisions
- explicit notes on what remains calibration-pending or assumption-based

### Required depth standard
Research must be deep enough to:
- support real milestone decisions
- identify disagreement or uncertainty in the literature
- distinguish validated realism from placeholder assumptions
- prevent shallow justification for design choices

Shallow literature scans are not sufficient for realism-critical milestones.

## 6. Depth and Technical Honesty Standard

HydrOS work must prefer depth over superficial speed whenever realism, control validity, sensing realism, or benchmark interpretation is at stake.

### This means:
- investigate enough to understand the true design trade-off
- do not stop at the first plausible answer
- identify the real uncertainty, not just the convenient answer
- distinguish what is known, what is inferred, and what is still unresolved
- include caveats where caveats are structurally important

### But depth does not mean drift
Depth should improve milestone quality, not expand scope aimlessly.
Research and analysis should remain tightly connected to the active milestone objective.

## 7. Default Milestone Execution Pattern

Unless the user explicitly chooses otherwise, execute HydrOS work in this order:

### Phase 0 — refinement
Clarify the true milestone objective and naming.

### Phase 1 — research layer
When needed, produce:
- research brief
- source map
- internal vs source-derived decision separation

### Phase 2 — execution document
Define:
- scope
- constraints
- required outputs
- acceptance criteria
- benchmark boundaries

### Phase 3 — planning refinement
Convert execution intent into file-by-file implementation reality.

### Phase 4 — implementation execution
Perform the scoped technical work.

### Phase 5 — execution report
State exactly what happened.

### Phase 6 — closeout
Create the authoritative milestone record and achievements summary.

### Phase 7 — handoff
State what the next milestone inherits and what choice comes next.

### Phase 8 — insight / improve
Capture what this pass taught the system and workflow, and store it as a reusable insight artifact.

Do not skip governance just because coding is possible.

## 8. How to Evaluate "Best Next Step"

When choosing the next HydrOS move, use this priority order:

1. Reduce the largest realism gap that can be measured now
2. Protect benchmark integrity
3. Improve system interpretability
4. Strengthen deployment relevance without pretending deployment validity
5. Avoid work that creates impressive numbers without reducing uncertainty
6. Prefer actions that unlock future milestones rather than decorate the current one

In general, the best next step is the one that most increases:
- realism
- interpretability
- controlled progress
- future decision quality

## 11. Insight / Improve Output Standard

For every major orchestration artifact or meaningful milestone pass, create one insight file unless the user explicitly says not to.

### Insight files must be stored under:
`docs/orchestration/insights/<Milestone>/`

If that folder path does not yet exist, create it first.

### File naming convention
Use:
`<Milestone.Iteration>_insight_<short_slug>.md`

Examples:
- `M7.4_insight_risk_detection_must_be_operational.md`
- `M7.6_insight_baseline_exists_but_privileged_gap_remains.md`

### Required structure
Each insight file must contain:

#### 1. Title
A short insight title tied to the milestone and iteration.

#### 2. Context
What artifact, pass, or milestone event produced this insight?

#### 3. What was learned
What is now more true or more clear than before?

#### 4. Why it matters
Why does this change understanding of the system or workflow?

#### 5. Immediate system implication
What does this mean for HydrOS itself?

#### 6. Workflow implication
What does this mean for how future orchestration should be done?

#### 7. Recommended carry-forward action
What should be preserved, changed, or prioritized next?

### Good insight criteria
A good insight should capture one of these:
- a realism gap became clearer
- a benchmark boundary became clearer
- a workflow pattern proved valuable
- a design assumption was validated or weakened
- a next-step priority became more obvious
- a system layer became more interpretable

Insight files are not optional reflections. They are reusable strategic learning artifacts.
