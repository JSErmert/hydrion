## 1. Executive Objective

M8 trains a second canonical HydrionEnv policy, `ppo_hydrienv_v2`, on a deployment-
realistic 8-dimensional observation schema, `obs8_deployment_v1`, that retains only
channels available at hardware deployment: actuator command feedback (indices 6–9) and
sensor-derived readings (indices 10–13). By comparing `ppo_hydrienv_v2` against the
existing `ppo_hydrienv_v1` (trained on the full 14D `obs14_v1` schema) and a random
baseline, M8 quantifies the performance cost of removing the 6 genuinely non-deployment
physics-truth channels (indices 0–5). This measurement is the first deployment-gap
quantification for HydrionEnv and the gating result for all downstream deployment-
relevance, calibration, and reward-design work.

| 1 | 7 | pump_cmd | `truth.get("pump_cmd")` | actuator_feedback | [0, 1] — inherits obs14_v1 normalization |
| 2 | 8 | bf_cmd | `truth.get("bf_cmd")` | actuator_feedback | [0, 1] — inherits obs14_v1 normalization |
| 3 | 9 | node_voltage_cmd | `truth.get("node_voltage_cmd")` | actuator_feedback | [0, 1] — inherits obs14_v1 normalization |

| obs14_v1 index | Channel name | Reason for removal |
|----------------|--------------|-------------------|
| 0 | flow (truth) | physics_truth — not available at hardware deployment |
| 1 | pressure (truth) | physics_truth — not available at hardware deployment |
| 2 | clog (truth) | physics_truth — not available at hardware deployment |
| 3 | E_field_norm (truth) | physics_truth — not available at hardware deployment |
| 4 | C_out (truth) | physics_truth — not available at hardware deployment |
| 5 | particle_capture_eff (truth) | physics_truth — not available at hardware deployment |

### 3.2 Observation space specification

```
shape: (8,)
dtype: float32
low:   0.0
high:  1.0 (nominal; sensor channels may exceed 1.0 under extreme noise — clip enforced)
```

The `HydrionEnv.observation_space` must declare `shape=(8,)` when obs8_deployment_v1
is active. This requires the observation space to be parameterized by schema.

### 3.3 Normalization continuity

All 8 channels in obs8_deployment_v1 retain the same normalization references as their
corresponding obs14_v1 counterparts. No new normalization constants are introduced. No
existing normalization constants are changed.

**VecNormalize statistics are NOT continuous.** The VecNormalize running statistics
(mean, variance) from `ppo_hydrienv_v1_vecnorm.pkl` are obs14_v1-specific — they encode
statistics for a 14D vector. They cannot be sliced, truncated, or adapted for 8D use.
A fresh VecNormalize statistics file must be fit to obs8_deployment_v1 during training.
See Section 6 — Methodology Validity Condition 1.

### 3.4 Naming and version rules

- Schema label: `obs8_deployment_v1`
  - "8" = dimensionality
  - "deployment" = intent (deployment-realistic, not privileged)
  - "v1" = first schema at this dimensionality and intent
- obs8_deployment_v1 applies to HydrionEnv only
- obs8_deployment_v1 is **not** equivalent to obs12_CCE or obs14_v1
- obs8_deployment_v1 is **deployment-realistic**, not **sensor-only**:
  it includes 4 actuator-feedback channels that are not sensed hardware quantities —
  they are controller-generated signals. Sensor-only would be indices 10–13 only (4D).
  That is a more restrictive-than-deployment scenario and is outside M8 primary scope.

### 4.3 Role of ppo_hydrienv_v2 relative to ppo_hydrienv_v1

`ppo_hydrienv_v2` is a deployment-gap comparison artifact. It does not replace
`ppo_hydrienv_v1`. The relationship is:

- `ppo_hydrienv_v1` is the full-schema baseline (obs14_v1): the best policy we can train
  with complete privileged simulation state.
- `ppo_hydrienv_v2` is the deployment-realistic artifact (obs8_deployment_v1): the best
  policy we can train with only hardware-available observations.
- The comparison between them measures the deployment gap, not relative policy quality.
  A higher return from v2 would not make it "better" than v1 in any general sense.

### 4.4 Random baseline

A random-action baseline must be evaluated under `obs8_deployment_v1` (using the same
HydrionEnv + ShieldedEnv configuration as v2 training). This baseline anchors the scale:
any return above the random baseline indicates the policy learned above-chance control.

The M7 random baseline result (mean return ≈ 586.028 under obs14_v1) is not directly
reusable for M8 comparison under obs8_deployment_v1 — environments may behave differently
under a different observation schema if the policy interacts differently. A fresh random
baseline must be collected alongside the M8 evaluation.

## 6. Methodology Validity Conditions

The following four conditions are **non-negotiable execution requirements** grounded in
peer-reviewed sources. Violating any of them renders the M8 result methodologically
unsound. These are not stylistic preferences.

### MVC-1: VecNormalize refit required

**Condition:** A new VecNormalize statistics file must be fit from scratch to
`obs8_deployment_v1` during ppo_hydrienv_v2 training.
`ppo_hydrienv_v1_vecnorm.pkl` must NOT be loaded for any M8 training or evaluation path.

**Source basis:** M8.1R.4_sources_map.md — S2 (Andrychowicz et al. 2021 ICLR). Observation
normalization has a large, consistent effect on PPO policy performance. VecNormalize
statistics encode the dimensionality and running statistics of the observation vector —
a 14D stats file cannot be validly used for an 8D observation.

**Implementation requirement:** The training script for ppo_hydrienv_v2 must initialize
VecNormalize fresh. The evaluation script must load `ppo_hydrienv_v2_vecnorm.pkl` when
evaluating v2, and `ppo_hydrienv_v1_vecnorm.pkl` when evaluating v1. These must never
be swapped.

### MVC-2: Matched training conditions required

**Condition:** ppo_hydrienv_v2 must train with the same PPO hyperparameters, episode
length, ShieldedEnv configuration, and training seed as ppo_hydrienv_v1. Only the
observation schema changes.

**Source basis:** M8.1R.4_sources_map.md — S2 (Andrychowicz et al. 2021). Differences in
training conditions confound the attribution to observation space. If v2 trains with
different hyperparameters, any performance delta is no longer interpretable as an
observation-space effect.

**Implementation requirement:** The training script for ppo_hydrienv_v2 must use:
- Same total_timesteps (500,000)
- Same seed (42)
- Same ShieldedEnv parameters (max_pressure_soft=0.85, max_pressure_hard=1.05,
  max_clog_soft=0.80, max_clog_hard=0.98, terminate_on_hard_violation=True)
- Same PPO algorithm hyperparameters as ppo_hydrienv_v1

### MVC-3: Calibration confound must be named explicitly

**Condition:** All M8 evaluation output, acceptance criteria notes, and the closeout
record must include an explicit statement that the comparison is confounded by
calibration-pending sensor noise parameters.

**Source basis:** M8.1R.4_sources_map.md — S9 (Dulac-Arnold et al. 2021). All confounds
in RL evaluation must be named. The calibration-pending nature of dp_drift_rate,
dp_fouling_gain, and flow_calibration_bias_std is a structural confound: the sensor
channels in obs8_deployment_v1 may behave differently under real hardware noise than
under the current placeholder model.

**Implementation requirement:** The evaluation report and closeout record must include
a section titled "Confounds and Limitations" that states:
> "M8 results are confounded by calibration-pending sensor noise. The sensor channel
> quality in obs8_deployment_v1 (indices 6–7 in the new schema, obs14_v1 indices 10–13)
> is governed by placeholder parameters (dp_drift_rate, dp_fouling_gain,
> flow_calibration_bias_std). The deployment gap measurement is internally valid under
> these parameters, but its predictive value for hardware deployment is bounded by
> the uncalibrated sensor model."

### MVC-4: Cross-schema deployment-gap framing required

**Condition:** The ppo_hydrienv_v2 vs ppo_hydrienv_v1 comparison must be labeled and
described as a "cross-schema deployment-gap measurement." It must not be described as
a "policy improvement" comparison or as evidence that one policy is generically better.

**Source basis:** M8.1R.4_sources_map.md — S5 (Pinto et al. 2018), S7 (Lee et al. 2020),
S8 (Rudin et al. 2022), S9 (Dulac-Arnold et al. 2021). The two policies operate on
different observation spaces. A return comparison between them measures the cost of the
observation gap, not a quality ordering.

**Implementation requirement:** All result summaries, metadata fields, and the closeout
record must use the phrase "cross-schema deployment-gap measurement" or equivalent, and
must not use phrases suggesting v2 improves on or competes with v1 as a general-purpose
policy.

**Execution note:** Training requires approximately 15–16 minutes at 500k steps.
Background Bash processes in this environment are killed at approximately 5 minutes.
Training must be executed as a manual foreground terminal command:
`python -m hydrion.train_ppo_hydrienv_v2`

### 7.2 Evaluation design

The M8 evaluation must run three agents in a single pass under the same environment
configuration:

| Agent | Observation schema | VecNorm stats | Type |
|-------|-------------------|---------------|------|
| ppo_hydrienv_v2 | obs8_deployment_v1 (8D) | ppo_hydrienv_v2_vecnorm.pkl | Trained policy |
| ppo_hydrienv_v1 | obs14_v1 (14D) | ppo_hydrienv_v1_vecnorm.pkl | Reference policy |
| random baseline | obs8_deployment_v1 (8D) | none | Random actions |

**Required evaluation outputs:**

1. Mean episode return ± std for all three agents
2. Shield violation count for v2 and v1 (hard violations only)
3. Channel activity diagnostics for v2:
   - obs8 indices 0–3 (actuator_feedback — obs14_v1 indices 6–9): mean and std
   - obs8 indices 4–7 (sensor_derived — obs14_v1 indices 10–13): mean and std
4. Maximum absolute difference between sensor channels and their corresponding
   truth-analog channels (obs8 index 6 / flow_sensor_norm vs obs14_v1 index 0 / flow;
   obs8 index 7 / dp_sensor_norm vs obs14_v1 index 1 / pressure) — diagnostic for
   sensor noise magnitude
5. Performance delta: `(mean_return_v2 - mean_return_v1) / abs(mean_return_v1)` as a
   signed percentage
6. Performance delta relative to random: `(mean_return_v2 - mean_return_random) / abs(mean_return_random)`

### 7.3 Cross-policy validity note

The ppo_hydrienv_v1 evaluation in the M8 eval pass re-evaluates v1 under the standard
HydrionEnv + ShieldedEnv configuration. The M7 canonical result (mean return 694.639)
should be reproduced closely; any material discrepancy must be investigated before M8
conclusions are drawn.

## 9. Four-Scenario Result Interpretation Table

This table is binding for all M8 result interpretation. It applies to the eval output,
the closeout record, and any future reference to M8 results. Do not apply interpretations
not listed here.

Source: M8.1R.4_sources_map.md Section 6; M8.1R.3 insight Section 5.

| Result | Correct interpretation | Must NOT conclude |
|--------|----------------------|-------------------|
| v2 return ≈ v1 return | obs8_deployment_v1 achieves near-equivalent performance at 500k steps under placeholder sensor noise. The 6 physics-truth channels do not appear to be strongly load-bearing at this training scale and noise regime. | "Channels 0–5 are unnecessary for deployment"; "ppo_hydrienv_v2 is deployment-ready"; "sensor control is proven sufficient" |
| v2 return < v1 return (small gap, < ~10%) | Moderate performance cost from removing physics truth channels at this training scale. The deployment-realistic policy is functional but not equivalent. | "Deployment is impractical"; "sensor channels are insufficient" |
| v2 return < v1 return (large gap, > ~20%) | Substantial performance cost. Contributing factors may include: (a) genuine dependence on physics truth channels, (b) insufficient training budget for v2 to exploit obs8 signals (S8 — Rudin et al.), (c) placeholder noise quality limiting sensor utility. All three require investigation before concluding information insufficiency. | "Deployment is impossible"; "the system cannot be controlled without physics truth"; "obs8 is fundamentally insufficient" |
| v2 return > random | Deployment-realistic policy learned above-chance control. obs8_deployment_v1 supports non-trivial policy learning. | "ppo_hydrienv_v2 is deployment-ready"; "the deployment gap is solved" |

**Mandatory gap-size investigation trigger:**
If the large-gap scenario is observed, M9 must evaluate, before concluding channel
insufficiency:
1. Training budget extension for ppo_hydrienv_v2 (S8: compute may close the gap)
2. Distillation from ppo_hydrienv_v1 as teacher (S7: Lee et al. 2020 DAgger path)
3. Whether calibration-pending noise is suppressing sensor signal quality

## 10. Prohibited Claims

The following claims must not appear in M8 evaluation output, closeout records, insights,
or any artifact produced under this execution document. They are listed explicitly so they
can be checked during AC review.

| Prohibited claim | Why prohibited |
|-----------------|----------------|
| "ppo_hydrienv_v2 is deployment-ready" | Deployment readiness requires calibrated sensor models and hardware validation. Neither is done in M8. |
| "Channels 0–5 are unnecessary" | Training ablation measures marginal contribution, not absolute necessity (S3 — Greydanus et al. 2018). A near-equivalent result proves substitutability at this training scale, not dispensability. |
| "obs8_deployment_v1 is sensor-only control" | obs8 includes 4 actuator-feedback channels (controller-issued commands). It is deployment-realistic, not sensor-only. |
| "Deployment is impossible" or "sensor control is insufficient" | A performance gap at 500k steps may reflect training budget or noise quality, not fundamental information insufficiency. Investigation required before this claim. |
| "ppo_hydrienv_v2 is better than ppo_hydrienv_v1" | The comparison is cross-schema. Return magnitude differences are not evidence of general quality ordering. |
| "M8 proves deployment viability" | M8 provides a lower bound on deployment-gap magnitude, not a viability proof. |
| "The calibration confound is minor" | Its magnitude is unknown. It must be named and cannot be characterized without calibration data. |

## 11. Risks and Confounds

### 11.1 Literature-thin areas (non-recoverable)

These are structural limitations of M8. They cannot be closed by additional research
within M8 scope. They must be acknowledged in all result claims.

**R1 — No flow-control RL ablation precedent:**
All ablation methodology cited (S2 Andrychowicz, S3 Greydanus) comes from robotics and
game-playing domains. The transfer assumption to HydrionEnv's continuous flow-control
domain is plausible but untested. M8 cannot cite literature validating its methodology
in the HydrOS domain.
*Implication:* M8 applies established methodology (matched conditions, training ablation,
named confounds) and acknowledges the transfer assumption. No stronger claim is possible.

**R2 — No empirical evidence that actuator history is sufficient for fouling-phase inference:**
Whether obs[6–9] (actuator command patterns: valve, pump, backflush, voltage) carries
sufficient information to track membrane fouling phase (loading, near-clog, post-backflush)
is theoretically plausible but not established in any cited source. M8 generates the
first data point.
*Implication:* M8 result is informative but not conclusive on this specific question.

**R3 — No literature support for ablation validity under placeholder noise:**
The calibration-pending nature of dp_drift_rate, dp_fouling_gain, and
flow_calibration_bias_std means sensor channel quality is not validated. RL ablation
under placeholder noise may produce results that do not predict hardware deployment
behavior. No published source addresses this confound.
*Implication:* M8 result is internally valid (consistent comparison under same noise
regime) but predictive deployment value is unknown and bounded by calibration status.

**R4 — No literature support for single-seed comparison:**
Multi-seed evaluation is the gold standard for RL ablation. M8 uses a single seed (42)
for both training runs. No source validates single-seed comparison as sufficient for
this type of ablation.
*Implication:* M8 provides a single data point, not a statistically robust estimate.
Results must not be over-generalized.

### 11.2 Execution risks

**R5 — Background execution kill limit:**
Training requires approximately 15–16 minutes. Background Bash processes in this
environment are killed at approximately 5 minutes. Training must be run as a manual
foreground terminal command. This is not a literature-thin issue; it is a documented
platform constraint from M7 execution.

**R6 — Schema selector implementation complexity:**
Changing HydrionEnv from obs14_v1 to obs8_deployment_v1 requires either a schema
parameter or a separate environment class. If implemented as a parameter, care must be
taken to ensure observation_space, VecNormalize, and eval paths all update consistently.
An assertion in the training script (AC2) prevents mismatches from silently propagating.

**R7 — Benchmark confusion risk:**
If the eval script accidentally evaluates v2 with v1's VecNorm stats (or vice versa),
the performance comparison is invalid. AC5 (eval script VecNorm path verification)
addresses this directly.
