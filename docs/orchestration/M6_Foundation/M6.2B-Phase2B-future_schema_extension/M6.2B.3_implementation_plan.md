## Summary

obs14_v1 is a narrow, add-only 12D → 14D extension of the HydrionEnv RL observation schema.
Two new sensor-derived indices are appended after the existing 12. No existing index changes
source, physical quantity, or normalization. CCE is untouched. ppo_cce_v2 is unaffected.

The total code change is 4 lines across 2 files (`sensor_fusion.py`, `env.py`) plus one new
test file. No configs change. No state initialization changes. No new YAML parameters.

| **13** | **dp_sensor_norm** | **sensor_state** | **`sensors["dp_sensor_kPa"]`** | **Noisy estimate of dp_total_pa (filter differential, kPa)** | **/ (P_max_Pa / 1000) = / 80.0** | **[0, ~0.79] nominal; clip [0, 1]** |

**Critical semantic distinction — indices 12 and 13 are new channels, not substitutes:**
- Index 12 measures q_processed_lmin (filter input, pre-diversion). Index 0 measures Q_out_Lmin
  (system output, post-diversion). These diverge up to 5× during active backflush.
- Index 13 measures dp_total (pure filter differential, in kPa). Index 1 measures P_in (inlet
  absolute pressure, includes valve and pipe resistance, encodes bypass threshold proximity).
  Replacing index 1 with a dp_total reading would remove the bypass-activation signal from
  the observation — this was ruled out in the M6.2 field audit.

## Section 3 — Normalization and Bounds Plan

### Index 12 — flow_sensor_norm

**Formula:**
```python
float(np.clip(sensors.get("flow_sensor_lmin", 0.0) / 20.0, 0.0, 1.0))
```

**Denominator justification:**
- `Q_max_Lmin = 20.0` is the existing YAML parameter for maximum system flow.
- It is already used to normalize index 0 (`truth["flow"]`).
- Reusing it for index 12 allows direct numerical comparison between truth flow and sensor
  flow during diagnostics. At steady state, index 0 ≈ index 12.
- No new YAML parameter is introduced.

**Why clipping is required:**
- The M6.1 flow sensor model includes a per-episode calibration bias (std = 0.2 L/min) and
  multiplicative AWGN (1% reading). At peak flow with positive calibration bias, `flow_sensor_lmin`
  can transiently exceed 20.0 L/min, producing a raw normalized value > 1.0.
- `spaces.Box(low=0.0, high=1.0)` is a contract. Clip enforces it.
- `np.clip(..., 0.0, 1.0)` is the correct mechanism — do not use `min()`/`max()` (not
  numpy-safe with arrays).

**Should flow_sensor_norm be added as a normalized key to sensor_state?**
- No. `sensor_state` stores raw physical-unit sensor readings (kPa, L/min). Normalization is
  the responsibility of `build_observation()` only. Mixing normalized values into `sensor_state`
  would blur the boundary between the physical sensor model and the RL observation contract.
- The normalized value exists only in the obs vector. It is not a separate sensor_state key.

---

### Index 13 — dp_sensor_norm

**Formula:**
```python
float(np.clip(sensors.get("dp_sensor_kPa", 0.0) / 80.0, 0.0, 1.0))
```

**Denominator justification:**
- `P_max_Pa = 80,000 Pa` is the existing YAML parameter.
- `P_max_Pa / 1000 = 80.0 kPa` converts to the same unit as `dp_sensor_kPa`.
- Physical interpretation: "what fraction of the system's maximum pressure budget is consumed
  by the filter differential?" This is policy-relevant for fouling management decisions.
- At maximum estimated fouling + nominal flow, dp_total ≈ 63 kPa < 80 kPa — the bound is
  conservative and physically appropriate.
- No new YAML parameter (`dp_max_kPa`) is introduced. If future calibration reveals dp_total
  routinely exceeds 80 kPa, this is the trigger to: (a) introduce `dp_max_kPa` and (b)
  bump obs14_v1 → obs14_v2 per the version bump rules in Section 4.

**Why clipping is required:**
- The M6.1 dp sensor model includes AWGN (σ = 0.25 kPa), random walk drift (capped ±2.0 kPa),
  and a fouling-induced bias offset. In the worst-case combination of high fouling + positive
  drift + high flow, `dp_sensor_kPa` can transiently exceed 80.0 kPa.
- Clip enforces the `spaces.Box` contract.

**Should dp_sensor_norm be added as a normalized key to sensor_state?**
- No. Same rationale as for index 12. Normalization belongs in `build_observation()`.

## Section 4 — Schema / Version Plan

### Version Label

**Label:** `obs14_v1`

**Applies to:** HydrionEnv only. CCE retains its unnamed 12D truth-derived schema.

**Naming convention (internal HydrOS architecture standard):**
- First number = dimensionality. `obs14` means 14 dimensions.
- `v1` = first schema at 14D.
- Dimensionality change always produces a new major label: obs12 → obs14.
- A minor bump (e.g., obs14_v1 → obs14_v2) occurs when an existing index changes its source
  key, physical quantity, or normalization reference within the same 14D shape.

### Where Version Labels Must Be Updated

| Location | Required update |
|---|---|
| `hydrion/sensors/sensor_fusion.py` docstring | Primary: change `obs12_v2` → `obs14_v1`; add change-from-previous note; add indices 12 and 13 to table |
| `hydrion/env.py` line 166 comment | Secondary: change `obs12_v1 — immutable` → `obs14_v1 — M6.2B — 2026-04-13` |
| `tests/test_obs14_v1.py` docstring | New file; label with `obs14_v1` at top |

**The `sensor_fusion.py` docstring is the canonical version label.** `env.py` is a secondary
reference. These two must be kept in sync. Any discrepancy between them constitutes schema
drift.

### Preventing Silent Schema Drift

Silent schema drift occurs when an index source or normalization changes without the version
label being updated. This is the primary failure mode to prevent.

**Prevention mechanisms for obs14_v1:**

1. **Version label in `sensor_fusion.py` docstring** — canonical. Updated in this
   implementation. Any future change to an index source, physical quantity, or normalization
   must update this label.

2. **Shape check in tests** — `test_obs14_v1.py` test T1 asserts `obs.shape == (14,)`.
   This will fail if `build_observation()` accidentally returns a different length.

3. **Semantic index verification in tests** — tests T3 and T4 (below) verify that indices 12
   and 13 trace back to their declared source keys. These fail if the source mapping silently
   changes.

4. **CCE shape guard** — test T7 (below) asserts CCE still returns 12D. This fails if CCE is
   accidentally modified.

5. **No normalization constants in sensor_state** — raw physical units in sensor_state prevent
   accidental semantic conflation between the sensor model and the obs contract.

### Distinguishing obs12_v2 and obs14_v1 in Docs and Artifact Lineage

| Distinction point | obs12_v2 | obs14_v1 |
|---|---|---|
| Shape | (12,) | (14,) |
| Indices 0–11 | Canonical | Identical — unchanged |
| Index 12 | Does not exist | flow_sensor_norm (sensor-derived) |
| Index 13 | Does not exist | dp_sensor_norm (sensor-derived) |
| Applies to | HydrionEnv + CCE (same shape, different semantics) | HydrionEnv only |
| Canonical version comment | `sensor_fusion.py` docstring | `sensor_fusion.py` docstring |
| HydrionEnv observation_space shape | `(12,)` | `(14,)` |

Any policy artifact trained under obs12_v2 (12D input) is architecturally incompatible with
obs14_v1 (14D input) at the policy input layer. There is no silent compatibility path.
The dimensionality mismatch is the hard schema boundary.

## Section 5 — Backward-Compatibility and Artifact Lineage Plan

### ppo_cce_v2 — FULLY VALID, UNAFFECTED

ppo_cce_v2 is trained on CCE. CCE is not modified. CCE `_obs()` returns a 12D truth-derived
vector and will continue to do so. Any rollout of ppo_cce_v2 on the post-implementation
codebase produces results identical to the pre-implementation benchmark artifact.

**Status:** ppo_cce_v2 remains the canonical M5 CCE physics benchmark.

### Any Prior HydrionEnv Policy Artifact — INCOMPATIBLE with obs14_v1

Any policy trained on obs12_v2 HydrionEnv observations has a 12D input layer. obs14_v1
provides a 14D observation. These are architecturally incompatible:
- Running an obs12_v2 policy on a obs14_v1 environment requires padding or truncating the
  observation. Neither is valid for performance measurement.
- Any such diagnostic run is labeled: **"obs12_v2 policy on obs14_v1 env — diagnostic only;
  not a comparable benchmark."** Performance metrics from such runs are not baseline-comparable.

**Note:** There is no existing canonical HydrionEnv training artifact to protect. HydrionEnv
has not had a published baseline training run before this phase. The incompatibility record
is stated here for completeness and future traceability, not because an artifact is at risk.

### What Becomes Diagnostic Only

Any test or evaluation that:
- Runs an obs12_v2-dimensioned policy on a obs14_v1 environment
- Compares obs12_v2 and obs14_v1 policy reward curves without explicit experimental controls

These runs are valid for characterizing the information value of the new sensor channels
but are not valid for benchmark comparison.

### New HydrionEnv Baseline Artifact Required

Phase 3 (M7 RL rebuild) must produce one canonical HydrionEnv policy artifact trained from
scratch on obs14_v1. This artifact becomes:
- The new HydrionEnv benchmark for all Phase 4+ comparisons
- The reference for evaluating whether obs14_v1 sensor channels improve policy performance
  vs a truth-derived obs12_v2 baseline

The ppo_cce_v2 CCE benchmark and the new HydrionEnv obs14_v1 artifact are maintained as
**independent benchmarks for independent environments.** They are not cross-comparable.

### What Must Be Stated in Metadata / Docs

Any doc, commit message, or artifact metadata that records a policy trained after M6.2B
implementation must:
1. State the observation schema version (`obs14_v1`).
2. State the environment (`HydrionEnv` or `CCE`).
3. If running a cross-schema diagnostic, label it explicitly as diagnostic.

The canonical location for this metadata is the training artifact's config file or a
companion `artifact_metadata.json` — to be defined in Phase 3.

### T6 — No Truth-State Contamination

```python
def test_no_truth_state_contamination():
    """
    Indices 12 and 13 must come from sensor_state, not truth_state.
    Verify by injecting a known sensor_state value and confirming it appears at index 12/13,
    while the corresponding truth value is different.
    """
    env = HydrionEnv()
    env.reset(seed=42)
    # After reset, sensor and truth may agree; run one step to create divergence
    env.step(env.action_space.sample())
    truth_flow_norm = env.truth_state.get("flow", None)
    sensor_flow_norm_expected = float(
        np.clip(env.sensor_state["flow_sensor_lmin"] / 20.0, 0.0, 1.0)
    )
    obs = env._observe()
    assert abs(obs[12] - sensor_flow_norm_expected) < 1e-5, (
        "Index 12 does not match sensor_state['flow_sensor_lmin']"
    )
    # truth_state["flow"] is Q_out, not Q_processed — verify index 12 != index 0 after divergence
    # (this may be equal at steady state, so this is a structural check not a numeric check)
    assert obs[12] is not obs[0]  # distinct array elements
    env.close()
```

**Purpose:** Structural guard against truth_state leakage into sensor-derived indices.

### T8 — Sensor Indices Diverge from Truth Indices Under Backflush

```python
def test_sensor_indices_diverge_from_truth_under_backflush(tolerance=0.01):
    """
    During active backflush (bf_cmd > 0), Q_out_Lmin < q_processed_lmin.
    Index 0 and index 12 must diverge. Demonstrates the dual-channel design is providing
    distinct information, not redundant signals.
    """
    env = HydrionEnv()
    env.reset(seed=42)
    # Force backflush by submitting max bf_cmd (action index 2)
    action = np.array([0.5, 0.5, 1.0, 0.5], dtype=np.float32)  # [valve, pump, bf, voltage]
    obs, _, _, _, _ = env.step(action)
    # During backflush, index 12 (q_processed) should exceed index 0 (Q_out) by design
    if obs[12] > tolerance:  # only test if flow sensor has a non-trivial reading
        assert obs[12] >= obs[0] - tolerance, (
            f"Under backflush, index 12 (q_processed={obs[12]:.4f}) "
            f"should be >= index 0 (Q_out={obs[0]:.4f})"
        )
    env.close()
```

**Purpose:** Functional validation that the dual-channel design carries distinct information.
Documents the intended semantic relationship for future readers.

## Section 7 — Risks and Failure Conditions

### Risk 1 — Accidental Modification of Indices 0–11

**Failure mode:** A code change to `sensor_fusion.py` accidentally inserts a new element
before index 12, shifting existing indices. Indices 0–11 silently change meaning.

**Detection:** Test T2 (bounds spot-check), test T3/T4 (source key verification).

**Prevention:**
- Append-only pattern: new lines go at the end of the array, after the `sensor_scatter` line.
- Review the full array after editing to confirm 12 original entries are unchanged.
- Do not reorder, insert between, or delete any existing array element.

---

### Risk 2 — Accidental CCE Observation Mutation

**Failure mode:** `build_observation()` is modified but `ConicalCascadeEnv._obs()` also
calls it, causing CCE to silently gain indices 12 and 13.
