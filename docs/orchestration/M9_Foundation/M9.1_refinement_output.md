## 2. Decision Framing

The core strategic question M9 must answer first:

> **Does the positive obs8 deployment gap persist under calibrated, hardware-realistic sensor noise — or was it an artifact of placeholder noise parameters that do not reflect how a real differential pressure sensor and flow meter behave?**

This question is load-bearing because:

1. All three unresolved M8 confounds impair the interpretability of the gap result.
2. The primary confound (placeholder noise) is the most tractable to address: it requires literature-supported sensor characterization, not new RL infrastructure.
3. Resolving it converts the M8 result from "internally valid simulation measurement" to "first hardware-relevant deployment gap estimate HydrOS has produced."
4. The answer determines the strategic direction of every milestone after M9 — if the gap survives calibration, domain randomization and reward alignment are well-motivated; if it does not, the scope of necessary observation infrastructure changes.

Choosing any direction other than sensor calibration as M9 risks building on a result whose calibration-validity is unknown.

## 3. Inherited Confounds

**Ranking (AROD Schema v1 — Section 12 confound-priority structure):**

**Primary — Placeholder sensor noise**
Parameters: `dp_drift_rate`, `dp_fouling_gain`, `flow_calibration_bias_std` in `configs/default.yaml` (and wherever sensor initialization reads them).
These govern the noise characteristics of obs8_deployment_v1 indices 4–7 (sensor_turbidity, sensor_scatter, flow_sensor_norm, dp_sensor_norm). The +21.2% result was produced under placeholder values, not measured sensor behavior. Until calibrated, the gap cannot be attributed to information content differences — it could reflect a favorable placeholder noise regime.

**Secondary — Step-budget asymmetry**
v2 ran for 360,448 steps; v1 ran for 500,000 steps. The gap comparison is not step-matched. A 500k-step v2 run would determine whether additional training increases, maintains, or narrows the gap and eliminate this source of comparison imprecision. This is operationally tractable (the full-budget run executes via `python -m hydrion.train_ppo_hydrienv_v2` with no arguments) but does not change the calibration picture.

**Tertiary — Single-seed training**
Both v1 and v2 used seed=42. No cross-seed variance estimate exists for either policy. The gap magnitude has no error bar attributable to training stochasticity. This confound is the least tractable to resolve without significant infrastructure (multi-seed training runs) and the least likely to reverse the qualitative result — policy collapse from seed sensitivity at 360k steps is implausible. It is worth noting but is not the priority.

**Which to reduce first:** The primary confound. M9 should produce calibrated sensor noise parameters as its central contribution. The secondary confound (step-matched benchmark) can be resolved as optional M9 infrastructure — the 500k rerun is a single command and does not require new architecture. The tertiary confound (multi-seed) is M10+ scope.

## 4. Candidate Branch Comparison

### Branch 1 — Sensor calibration / realism tightening

**Realism leverage:** Highest. Replaces placeholder physics with literature- or device-grounded sensor behavior. This is the most direct improvement to simulation fidelity available.

**Confound reduction:** Directly eliminates the primary confound. After M9, the gap result becomes the first hardware-realistic deployment gap HydrOS has produced.

**Benchmark safety:** High. The calibration pass does not require touching any trained model artifact. ppo_hydrienv_v1, ppo_hydrienv_v2, and the CCE lineage are all untouched. A fresh M9 retraining on calibrated noise produces a new artifact (ppo_hydrienv_v2_cal or equivalent); the M8 artifacts remain as the placeholder-noise reference.

**Interpretation leverage:** Highest. Calibrated noise converts the M8 result from "internally valid" to "hardware-relevant interpretation possible." Every downstream milestone benefits from this.

**Dependency readiness:** All infrastructure exists. The obs8 schema, v2 training script, v2 eval script, and ShieldedEnv config are in place. M9 adds only calibrated parameter values and a retraining pass.

**Implementation tractability:** Requires a research phase (literature or datasheet values for relevant sensor types — differential pressure transducer, flow meter). The parameter changes are localized to config or sensor initialization; no architecture changes required.

**Visual validity leverage:** High. Calibrated sensor channels mean the future console's sensor trace display is scientifically grounded, not arbitrary noise. The truth-vs-sensor comparison in the visual console gains meaning when the sensor noise reflects real behavior.

**Deployment relevance:** Highest. The path from current HydrOS to a hardware-relevant claim runs directly through sensor calibration.

**Risk of misleading result:** Low. Calibrated noise is more truthful than placeholder noise by construction. The risk is calibration imprecision (imperfect literature values), not overclaiming.

**Future work unlocked:** Domain randomization over calibrated ranges. Reward alignment unconfounded by noise regime. A visual console displaying hardware-interpretable telemetry. The obs8 result's meaning is clarified for all future uses.

---

### Branch 2 — Domain randomization over calibrated sensor ranges

**Realism leverage:** High — but only after calibration. Randomizing uncalibrated placeholder parameters would vary meaningless noise, not realistic operating envelope coverage.

**Confound reduction:** Does not directly reduce the primary confound. Extends calibrated results to a distribution, which is a different contribution.

**Dependency readiness:** Requires Branch 1 first. Cannot produce interpretable results without calibrated center-point values. This is a natural M10 direction after sensor calibration.

**Decision:** Deferred to M10 or combined with M9 calibration if scope allows after the research phase establishes parameter ranges.

---

### Branch 3 — Step-matched v2 benchmark completion

**Realism leverage:** None. A 500k-step rerun uses the same placeholder noise — it improves benchmark precision, not realism.

**Confound reduction:** Reduces the secondary confound only. Does not address calibration.

**Interpretation leverage:** Modest. Knowing whether +21.2% would be higher or lower at step-matched budgets is useful but not decisive. The qualitative conclusion (obs8 viable) is already established.

**Implementation tractability:** Highest of all branches — one command, no architecture changes.

**Decision:** Appropriate as optional M9 infrastructure, not as the defining M9 contribution. Execute as a preliminary step before calibration-based retraining if the 500k run can complete within the available execution window.

---

### Branch 4 — Distillation / teacher-student follow-on

**Realism leverage:** Indirect. Teacher-student distillation transfers privileged information from obs14 to obs8 policies but does not improve the physical realism of the sensor simulation.

**Confound reduction:** None against the three identified confounds.

**Dependency readiness:** Depends on a stable obs8 calibrated policy to serve as the deployment target. Premature without Branch 1.

**Risk:** Produces an interesting methodology result (a policy guided by privileged supervision) but does not advance the hardware-deployment pathway as directly as calibration. Could generate an impressive-looking result without improving the actual sim-to-real relevance.

**Decision:** Deferred. Low leverage now. Worth revisiting after M9 calibration if the obs8 result under calibrated noise weakens — distillation could then partially recover performance.

---

### Branch 5 — obs4_sensor_only ablation

**Realism leverage:** None. A 4D policy trained on placeholder noise is less informative than an 8D policy trained on calibrated noise.

**Confound reduction:** Does not reduce any of the three inherited confounds.

**Interpretation leverage:** Isolates the actuator-channel contribution, which is a genuine question (how much of the M8 result came from the 4 actuator command channels vs the 4 sensor channels?). But the answer under placeholder noise is confounded by the primary confound.

**Decision:** Deferred to after calibration. An obs4 ablation run after M9 calibration would be interpretable; before calibration it is not.

---

### Branch 6 — Frontend / telemetry milestone

**Visual validity leverage:** Highest of all branches for visual validity directly — a truth-vs-sensor-vs-policy visualization is exactly what the console milestone would produce.

**Realism leverage:** None. Frontend rendering does not change the underlying physics or sensor model.

**Confound reduction:** None. A visual console displaying placeholder noise is visually elaborate but scientifically misleading.

**Risk of misleading result:** Highest of all branches. A polished visualization of calibration-pending sensor traces would make the simulation look more scientifically grounded than it is. This is the exact problem the visual validity bridge is designed to prevent.

**Decision:** Deferred. Frontend work is strategically valuable and should be scheduled, but not before calibration. A milestone M10 or M11 frontend pass on calibrated sensor data would be high-value and honest. Before calibration, it would undermine visual validity rather than improve it.

**M9 direction: Sensor Realism & Calibration**

Replace the placeholder sensor-noise parameters (`dp_drift_rate`, `dp_fouling_gain`, `flow_calibration_bias_std`) with grounded values derived from literature or device datasheets for differential pressure transducers and flow meters appropriate to microplastic extraction devices. **Why this is highest leverage now:**

1. It directly eliminates the primary confound — the one that most impairs the interpretability of every M8 result downstream.
2. All RL infrastructure exists. The cost of M9 is a research phase (sensor characterization) plus a retraining run. No new architecture is required.
3. The calibrated result unlocks every subsequent HydrOS milestone in a way that placeholder noise cannot: domain randomization over real ranges, reward alignment without noise-regime confounds, and a frontend console displaying hardware-interpretable telemetry.
4. It is the shortest path from the M8 result to a deployment-relevant claim. The M8 result is currently valid internally; after M9 calibration it becomes valid in a hardware-interpretable sense.
5. Visual validity: calibrated sensor traces are the prerequisite for a scientifically grounded console. Every visual milestone after M9 benefits from this foundation.

No other branch produces this combination of confound reduction + infrastructure readiness + interpretive leverage + visual validity improvement.

## 7. What M9 Should and Should Not Attempt

### M9 should:

- Conduct a targeted research pass to identify grounded sensor noise parameters for:
  - differential pressure sensor: `dp_drift_rate`, `dp_fouling_gain` (and any other dp noise parameters)
  - flow meter: `flow_calibration_bias_std`
  - If applicable: turbidity and scatter sensor characterization for `sensor_turbidity` and `sensor_scatter` channels
- Update `configs/default.yaml` (or wherever sensor parameters are initialized) with the calibrated values
- Retrain `ppo_hydrienv_v2` under calibrated noise to produce a new canonical policy artifact (named to distinguish it from the placeholder-noise v2)
- Evaluate the calibrated policy against the v1 baseline using the existing eval script
- Report whether the gap direction and magnitude survive calibration
- Commit a fresh metadata artifact recording calibrated parameter provenance
- Document which parameter values came from literature vs internal estimates, and at what confidence level

### M9 should not:

- Randomize sensor parameters before establishing calibrated center values (domain randomization requires a calibrated center point)
- Modify the obs8 observation schema — the schema itself is sound; only its noise parameters change
- Attempt to retrain ppo_hydrienv_v1 or ppo_cce_v2 — these are protected reference artifacts
- Implement a frontend rendering milestone — premature before calibrated data exists
- Run an obs4 ablation — lower strategic priority and confounded without calibrated noise first
- Claim hardware deployment validity from the calibrated result alone — calibration improves interpretability but does not constitute hardware validation
- Weaken any M8 prohibited claims — the six prohibited claims from M8.3 Section 10 remain in force for M8 artifacts

### M9 scope boundary statement:

M9 is a sensor-realism pass, not a policy-redesign or frontend pass. The output is a calibrated noise configuration and a single retrained policy under that configuration. The question M9 answers is bounded and precise: does the M8 gap direction survive calibration?

The system model now has a tested, named observation selector. The assumption that the deployment gap would be negative was not confirmed — obs8 control is viable in-simulation.

- that assumption is validated approximately (calibrated values are close to placeholder behavior, gap is stable), or

- Future milestones must not claim the M8 placeholder-noise gap as a "hardware-relevant" baseline — the M9 calibrated-noise result supersedes it for hardware-relevant interpretation
- Future domain randomization passes must use M9 calibrated center values as the randomization reference, not the M8 placeholder values
- Future visual console work may claim that sensor trace rendering is hardware-interpretable for the calibrated channels, which was not claimable under placeholder noise
