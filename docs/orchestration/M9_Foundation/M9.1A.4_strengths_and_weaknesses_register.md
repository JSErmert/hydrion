## 1. Executive Snapshot

- **HydrOS is a top-heavy system** — a four-level maturity spread between L5 policy/benchmark/governance and L1 deployment / L2 sensor and visual layers (per M9.1A.2)
- **Strongest where farthest from hardware** — the RL control stack, benchmark discipline, and orchestration governance are unusually advanced for a research-grade twin
- **Weakest where it touches reality** — the sensor layer is placeholder-bound and the deployment relevance layer has no hardware contact
- **The gap is load-bearing** — every polished upper-layer result rests on an L2 sensor distribution; only textual discipline (prohibited claims, MVC, visual validity bridge) currently prevents false confidence
- **Correcting imbalance > adding features** — no new feature would improve the system as much as lifting the sensor layer by one or two levels

## 3. Weaknesses (Low Maturity / High Risk Areas)

### Weakness 1 — Measurement / Sensor Layer (PRIMARY)

| Field | Value |
|-------|-------|
| **Layer** | Measurement / sensor |
| **Maturity level** | **L2 (Emerging)** |
| **Why it is weak** | Sensors exist and are wired through `sensor_fusion.py` (dp_sensor, flow_sensor, turbidity, scatter) but all four channels are governed by placeholder noise parameters: `dp_drift_rate`, `dp_fouling_gain`, `flow_calibration_bias_std`. The numerical behavior of the sensor-derived observation channels does not reflect measured or literature-grounded sensor behavior. This is the definition of L2 — working implementation that is placeholder-bound. |
| **What depends on it** | `ppo_hydrienv_v2` training distribution. obs8_deployment_v1 sensor channels (indices 4–7). M8 +21.2% deployment gap result. Domain randomization (cannot center ranges without calibrated values). obs4 ablation (would be confounded identically). Frontend visual traces (would render placeholder data). Distillation scoping (cannot be determined until calibrated-noise gap is known). |
| **What risk it creates** | Every upper-layer result rests on an un-grounded distribution. The M8 result is internally valid but its hardware-predictive value is unknown. The stacked misalignment (L5 policy + L5 benchmark + L5 governance on top of L2 sensor) creates a credibility profile the sensor layer cannot substantiate. Without the M8.3 prohibited-claims list held forward, this produces implicit overclaim. |

### Weakness 2 — Deployment Relevance Layer

| Field | Value |
|-------|-------|
| **Layer** | Deployment relevance |
| **Maturity level** | **L1 (Conceptual)** |
| **Why it is weak** | No hardware has been connected. No real sensors. No device calibration. No domain randomization. No sim-to-real validation. The obs8_deployment_v1 schema is named "deployment-realistic" but the name describes **channel structure intent**, not calibration or hardware contact. |
| **What depends on it** | Any claim of hardware relevance. Any visual console rendering labeled as "what the deployed system would perceive." Any M10+ milestone that wants to produce cross-simulation-to-reality evidence. |
| **What risk it creates** | The largest single maturity gap in the system (4 levels below policy/benchmark/governance). A reader who sees the policy artifacts without seeing the deployment-relevance layer will reasonably but incorrectly infer HydrOS is "near-deployment." The M8.3 prohibited-claims list exists precisely because this risk is real. Propagation of prohibited claims through every downstream reference is the only current defense. |

### Weakness 3 — Visual Validity / Console Readiness Layer

| Field | Value |
|-------|-------|
| **Layer** | Visual validity / console |
| **Maturity level** | **L2 (Emerging)** |
| **Why it is weak** | Console infrastructure is present (mockup-k-o.html M4 baseline locked, machine-core-v4.html M5 reference, hydros-console Vite/React app). Rendering pipeline is real. **But** the sensor traces it would display are grounded only to the degree that the underlying data is grounded. No data-provenance labeling system exists. No truth-vs-sensor comparison view that explicitly marks calibration status. A rendered `dp_sensor_kPa` trace today would look scientifically grounded while displaying placeholder noise. |
| **What depends on it** | Any future operator-facing visualization. Any public or stakeholder-facing demo. Any debug or interpretation tool that renders policy behavior. Any "what the deployed system perceives" rendering. |
| **What risk it creates** | Polished visualization amplifies misleading effects. A textual report of placeholder data is plainly incomplete; a polished visual rendering of the same data implies scientific grounding it does not have. Any console milestone before M9 calibration is not just premature — it is actively misleading per the AROD visual validity bridge. |

## 5. Protected System Elements

The following elements MUST NOT be broken, weakened, overwritten, or silently modified:

### Policy lineage

- `models/ppo_cce_v2.zip`, `_vecnorm.pkl`, `_meta.json` — untouched since M6/M7; no CCE work in active milestones
- `models/ppo_hydrienv_v1.zip`, `_vecnorm.pkl`, `_meta.json` — canonical 500k obs14_v1 baseline; never load v1 vecnorm in any future training
- `models/ppo_hydrienv_v2.zip`, `_vecnorm.pkl`, `_meta.json` — M8 reference for the placeholder-noise comparison point; do not overwrite in M9; M9 calibrated artifact must be distinct (e.g. `ppo_hydrienv_v2_cal` or `ppo_hydrienv_v3`)
- `conical_cascade_env.py` — untouched; no CCE sensor work in M9

### Schema discipline

- `build_observation()` (obs14_v1) untouched — additive pattern preserves backward compatibility
- `build_observation_obs8()` — the M9 calibration pass changes **sensor initialization parameters only**, not the obs8 function itself
- `obs_schema` selector default remains `obs14_v1` — changing the default would alter every existing caller's behavior
- `_VALID_SCHEMAS` dict — new schemas added only via the additive extension pattern
- Named constant VecNorm path separation (`_V2_VECNORM_PATH`, `_V1_VECNORM_PATH`) — each constant used in exactly one `VecNormalize.load()` call; this structural property is what prevents schema swap

### Governance constraints

- Contract v3.1 — unchanged; no scope expansion before M9 closes
- AROD Schema v1 — reference-first; do not promote to mandatory-first; v2 refinement candidates are post-M9 evaluation targets
- CLAUDE.md co-orchestrator contract — binding on every milestone
- `docs/context/` — treated as required reading for every new milestone
- Realism roadmap ordering (hydraulics → clogging → backflush → electrostatics → particles → sensors → RL + reward) — no skipping; M9 sensor realism is the scheduled next step, not a detour

### Prohibited-claims discipline

- M8.3 Section 10 six prohibited claims — propagate forward in every artifact that references M8 results, without softening
- MVC-3 calibration confound statement — printed at runtime in any eval that uses obs8_deployment_v1 or any M8-derived schema
- Four-scenario interpretation table (M8.3 Section 9) — binding for any future cross-schema comparison
- "Do NOT conclude: deployment-ready or deployment gap solved" runtime banner — remains active; never conditionally suppressed
- M9 execution document (when written) must define **its own prohibited-claims list before results are known** — symmetric under positive and negative outcomes

### Test discipline

- 128/128 passing test count — new tests must be additive only; any regression must be resolved at root cause, not worked around
- Backward-compatibility gate T5 (obs14_v1 default behavior unchanged) — must continue to pass after any M9 sensor parameter update
- AC11 exact canonical match (694.639) — the environmental stability gate for all future cross-schema comparisons
