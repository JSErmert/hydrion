**Gate G1 is now CLOSED.**

HydrOS M9 formally commits to three real-world sensor device classes for the measurement layer. ### 3.4 Why this satisfies the research access rule

The standing HydrOS research access rule requires peer-reviewed source access or equivalent before training knowledge is used to set physics constants. The M9.1R.2 preliminary brief could not satisfy this because it lacked a named device class to search against. With the sensor class commitment:

- **The device class is now named** — not a generic "differential pressure sensor" but a specific class (piezoresistive transmitter) with a defined application (filtration monitoring) and range (~0–80 kPa differential).
- **Datasheets for this class are publicly available** from manufacturers (Honeywell, Omega, Rosemount, Endress+Hauser, etc.) and contain the drift, noise, and accuracy specifications that M9.1R.3 needs to ground each parameter.
- **ISO 7027 defines the optical measurement standard** — the turbidity sensor class is anchored to a published standard, not to an unnamed "optical sensor."
- **The electromagnetic flow meter class is well-characterized** in industrial metrology literature and manufacturer spec sheets.

M9.1R.3 can now search for specific manufacturer datasheets or peer-reviewed sensor characterization papers with concrete device-class keywords. The research access rule is satisfied because the search target is defined and the source type (datasheet or peer-reviewed characterization) is concrete.

## 4. Provenance Framework for M9.1R.3

### 4.1 Per-parameter provenance record structure

Every calibrated parameter value in M9.1R.3 (and carried into M9.3) must include:

```
parameter_name:
  value: <calibrated numeric value>
  units: <SI or HydrOS-specific>
  device_class: <committed class from §2>
  source_type: <datasheet | peer_reviewed | derived | estimated>
  source_ref: <specific document or publication identifier>
  confidence: <high | medium | low>
  assumptions: <any transformation or estimation applied>
  conversion_notes: <e.g., "converted from %/year to kPa/step using P_max=80 kPa and dt=0.1s">
```

### 4.2 Confidence level definitions (binding for M9)

| Level | Definition | M9.3 commit rule |
|-------|-----------|------------------|
| **High** | Value taken directly from a named manufacturer datasheet or peer-reviewed measurement paper for the committed device class, with no transformation beyond unit conversion | **Committable in M9.3.** This is the target level for all M9 calibration parameters. |
| **Medium** | Value derived from a related device class, from a peer-reviewed review paper covering the general sensor category, or from a datasheet with a transformation assumption (e.g., extrapolating from a different pressure range) | **Committable in M9.3 with explicit derivation note.** The derivation assumption must be documented in the `assumptions` field. |
| **Low** | Value estimated from engineering common knowledge without a specific source, or from a datasheet for a significantly different device class | **NOT committable in M9.3.** Must be flagged as calibration-pending-by-experiment. If no better source is available, the parameter retains its placeholder value with the provenance explicitly recording that grounding was attempted but not achieved. |

### 4.3 Classification taxonomy

Every parameter in M9 falls into exactly one of these categories:

| Category | Definition | Example in M9 |
|----------|-----------|---------------|
| **Source-grounded** | Value from a named source (datasheet, peer-reviewed paper) for the committed device class, at high or medium confidence | Target state for `dp_drift_rate`, `turbidity_noise_std` after M9.1R.3 |
| **Derived** | Value computed from source-grounded inputs with documented transformations | e.g., per-step drift from per-year drift spec, given dt and P_max |
| **Experimental** | Value that can only be grounded by direct experimental correlation on the HydrOS target system | All optical gain coefficients (NTU-to-mass mapping); camera noise |
| **Preserved placeholder** | Value retained from pre-M9 state because grounding was attempted but not achieved | Only if M9.1R.3 cannot find a suitable source for a given parameter; must be explicitly documented |

## 5. Boundary Conditions

### 5.1 No parameter values are committed by this document

This document commits **device classes**, not **numeric values**. Every parameter listed in §3.1 retains its current placeholder value until M9.1R.3 produces a verified replacement with provenance. The current placeholder values are:

- `dp_drift_rate_kPa_per_step = 0.0005` — **still placeholder** pending M9.1R.3
- `dp_drift_max_kPa = 2.0` — **still placeholder** pending M9.1R.3
- `dp_fouling_gain = 0.5` — **still placeholder** pending M9.1R.3 (semantics resolved as process B; magnitude pending)
- `turbidity_noise_std = 0.01` — **still placeholder** pending M9.1R.3
- `scatter_noise_std = 0.01` — **still placeholder** pending M9.1R.3

### 5.2 M9.1R.3 is still required

The verified-source pass (M9.1R.3) is the artifact that produces actual numeric values with provenance. This document enables M9.1R.3 by providing the device-class anchor. Without M9.1R.3, M9.3 cannot commit calibrated values — it would be committing the same M9.1R.2 preliminary ranges that the scoping-pass insight explicitly warns against.

### 5.3 This step enables grounding; it does not perform it

The distinction matters for the M9.1R.1 scoping-pass insight's integrity:

- **M9.1R.2** = structural scope (sensor class mapping, noise-model shape, fundamental limits)
- **M9.2A** = device-class commitment (names the specific real-world anchors)
- **M9.1R.3** = verified grounding (produces numeric values with provenance against the committed classes)
- **M9.3** = execution document (commits the values, specifies protections, authors prohibited claims)

Each step enables the next. Collapsing or skipping steps would propagate unverified values into the execution document.

### 5.4 Turbidity as proxy signal — fundamental interpretive limit preserved

The ISO 7027 nephelometric sensor class measures NTU (nephelometric turbidity units). NTU is a function of particle count, size distribution, shape, and refractive index. It is **not** a direct measurement of microplastic mass concentration.

M9 calibrates the **noise floor** of the turbidity measurement (how noisy is the NTU reading for a given true NTU). M9 does **not** calibrate the **NTU-to-mass mapping** (what does a given NTU value mean in terms of microplastic concentration). The NTU-to-mass mapping is experimentally-groundable-only and belongs to M10+ scope.

This boundary is enforced in M9.2 §3.3 (SP5 — optical gain coefficient boundary) and must be carried into M9.3 as an explicit scope constraint.

## 6. Visual Validity Implication

### 6.1 How device-class commitment enables future console rendering

**Before this commitment:** The HydrOS console would display sensor traces labeled as "dp_sensor_kPa" and "flow_sensor_lmin" with no reference to a real device. A viewer could not relate the displayed behavior to any real-world instrument. Any truth-vs-sensor comparison would be between grounded physics and unnamed noise.

**After this commitment:** Each sensor trace can be labeled with its device class and operating principle:
- "dp_sensor_kPa — piezoresistive DP transmitter, ~0–80 kPa, filtration monitoring"
- "flow_sensor_lmin — electromagnetic flow meter, 0–20 L/min, dirty-water service"
- "sensor_turbidity — ISO 7027 nephelometric, 90° / 860 nm, turbidity proxy"

A viewer can relate the displayed noise characteristics to a specific device class, evaluate whether the behavior looks realistic, and cross-check against published specs.

**After M9 calibration completes (M9.1R.3 + M9.3):** The displayed traces will reflect datasheet-grounded noise magnitudes for the committed device classes. The console becomes scientifically interpretable — each trace corresponds to how a real sensor of that class would behave under the simulated conditions.

### 6.2 Why this is required before any frontend milestone

Per the AROD visual validity bridge (Schema v1 Section 11) and M9.1A.5 §5 sequencing, a visual console milestone must not render sensor traces as "deployment-realistic" unless the underlying data is scientifically grounded. The device-class commitment is the first step in the grounding chain:

1. **Device class committed** (this document) — the console knows *what kind of sensor* each trace represents
2. **Parameters grounded** (M9.1R.3 + M9.3) — the console can verify that trace noise matches the device spec
3. **Post-calibration rendering safe** (M10+ visual milestone) — the console can label traces as "grounded to [device class], [source ref], [confidence level]"

Without step 1, steps 2 and 3 have no anchor. This document provides the anchor.

**Gate G2 — `ConicalCascadeEnv` config-block audit — is CLOSED.**

- The obs12_CCE schema is truth-derived (12D, no sensor channels)

**Consequence for M9:** The config isolation mechanism (SP1 — dedicated `configs/m9_calibrated.yaml`) protects HydrionEnv v1/v2 eval paths as specified in M9.2 §3.3. CCE does not need additional protection from the sensor-calibration path because it does not reference sensor parameters.

**One caveat preserved:** If M9 or a future milestone edits **non-sensor** shared config fields (e.g., hydraulics or clogging parameters), those edits would propagate into CCE. M9's scope does not touch non-sensor config fields, so this caveat is informational, not actionable.
