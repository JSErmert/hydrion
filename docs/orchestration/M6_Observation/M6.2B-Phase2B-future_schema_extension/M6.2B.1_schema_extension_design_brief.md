# M6.2b.0 — Schema Extension Design Brief

**Document type:** Schema design and research brief
**Phase:** M6.2b — Future schema extension for sensor-derived RL inputs
**Date:** 2026-04-13
**Status:** Final — ready for M6.2b.1 execution-document drafting
**Governed by:** HydrOS Co-Orchestrator Execution Contract

---

## 1. Executive Conclusion

The minimum architecturally honest path for introducing sensor-derived RL inputs is a
**12D → 14D add-only schema extension**, designated **obs14_v1**.

Key decisions:

- **Strategy:** Add-only extension. No existing obs12_v2 index changes source, label, or
  physical quantity. Indices 0–11 are immutably preserved.
- **New indices:** Index 12 (`flow_sensor_norm`) and index 13 (`dp_sensor_norm`), both sourced
  from `sensor_state`.
- **Rollout:** HydrionEnv only. CCE is deferred — it has no sensor infrastructure and carries
  the ppo_cce_v2 benchmark.
- **Version name:** obs14_v1 (HydrionEnv path only). CCE retains its current 12D truth-derived
  observation schema without a version bump.
- **ppo_cce_v2 status:** Unaffected. CCE observation is unchanged.
- **HydrionEnv obs12_v2 policy artifacts:** Incompatible with obs14_v1 at input level (12D vs
  14D). Phase 3 must produce a new HydrionEnv baseline under obs14_v1.
- **Broader redesign:** Not justified at this stage. Redesigning requires empirical evidence
  from M7 RL training, which has not started.

## 2. Current-State Constraints

### 2.1 Observation Schema — obs12_v2 (HydrionEnv)

Defined in `hydrion/sensors/sensor_fusion.py`, function `build_observation(truth, sensors)`.
12 dimensions, all normalized to [0, 1].

| Index | Key | Source | Physical quantity |
|---|---|---|---|
| 0 | flow | truth_state | Q_out_Lmin / Q_max_Lmin (20.0) |
| 1 | pressure | truth_state | P_in / P_max_Pa (80,000) |
| 2 | clog | truth_state | mesh_loading_avg |
| 3 | E_field_norm | truth_state | Normalized radial E-field (kV/m) |
| 4 | C_out | truth_state | Outlet particle concentration |
| 5 | particle_capture_eff | truth_state | Capture efficiency |
| 6 | valve_cmd | truth_state | Valve actuator command |
| 7 | pump_cmd | truth_state | Pump actuator command |
| 8 | bf_cmd | truth_state | Backflush command |
| 9 | node_voltage_cmd | truth_state | Node voltage command |
| 10 | sensor_turbidity | sensor_state | Optical turbidity (noisy) |
| 11 | sensor_scatter | sensor_state | Optical scatter (noisy) |

Indices 0–9 are truth-derived. Indices 10–11 are already sensor-derived. The schema is the
single source of truth for RL observation construction. Index ordering is immutable as long
as the version label is unchanged.

### 2.2 sensor_state Keys

Initialized in `hydrion/state/init.py` / `init_sensor_state()`:

```
sensor_turbidity   — optical (in obs12_v2)
sensor_scatter     — optical (in obs12_v2)
dp_sensor_kPa      — M6.1 differential pressure sensor (NOT in obs12_v2)
flow_sensor_lmin   — M6.1 flow rate sensor (NOT in obs12_v2)
```

`dp_sensor_kPa` measures `dp_total_pa` (filter differential). `flow_sensor_lmin` measures
`q_processed_lmin` (filter-stage input flow). Neither is a direct substitute for an existing
obs12_v2 index — this was established in the M6.2 field audit (see
`docs/orchestration/M6_Phase2/Phase2_M6_implementation_plan.md`).

### 2.3 HydrionEnv Observation Space

```python
self.observation_space = spaces.Box(low=0.0, high=1.0, shape=(12,), dtype=np.float32)
```

Defined at `hydrion/env.py` line 167. **Note:** the comment at line 166 reads
"obs12_v1 — immutable"; this is stale. The actual schema in use is obs12_v2.
Both the shape and the stale comment must be updated when obs14_v1 is implemented.

### 2.4 ConicalCascadeEnv Sensor State

```python
@property
def sensor_state(self) -> dict:
    return {}   # intentionally empty — CCE has no sensor noise layer
```

CCE has no sensor model infrastructure. `DifferentialPressureSensor` and `FlowRateSensor`
are not instantiated in CCE. Any extension that introduces sensor-derived indices into CCE
requires a separate architecture phase, not just a schema change.

### 2.5 Benchmark Artifact

`ppo_cce_v2` — canonical M5 benchmark. Trained on CCE truth-derived obs (12D), 500k steps,
seed=42, d_p=1.0 µm, obs12 CCE schema. Any change to CCE observation invalidates this
benchmark. CCE observation must remain frozen.

### 2.6 Normalization References

| Environment | Flow normalization | Pressure normalization |
|---|---|---|
| HydrionEnv | Q_max_Lmin = 20.0 L/min | P_max_Pa = 80,000 Pa |
| CCE | OBS_Q_MAX = 25.0 L/min | OBS_DP_MAX = 150.0 kPa |

The two environments use different normalization bounds reflecting different operating
envelopes. This is by design and must be preserved. Any new sensor-derived index in
HydrionEnv uses HydrionEnv normalization references only.

### 2.7 Known Test Gap

No existing test validates that observation values stay within [0, 1] at runtime, despite
`spaces.Box(low=0.0, high=1.0)` declaring this as a contract. The M6.2b.1 implementation
plan must include an obs bounds regression test.

## 3. Candidate Schema Extension Strategies

Three strategies were evaluated.

### Strategy A — Add-Only Extension (12D → 14D)

Append two new sensor-derived indices after all existing obs12_v2 indices. Indices 0–11
are untouched. The new indices introduce the two M6.1 sensor readings as independent
observation channels.

**Pros:**
- No semantic change to any existing index — obs12_v2 artifact comparisons remain interpretable
  for indices 0–11
- Minimum invasive: the only required changes are in `sensor_fusion.py` (2 new lines) and
  `env.py` (update `observation_space` shape and stale comment)
- Clear audit trail: any future policy trained on obs14_v1 is unambiguously distinguishable
  from any policy trained on obs12_v2 by input dimensionality
- Allows M7 to evaluate empirically whether the sensor channels add policy value

**Cons:**
- The policy sees both truth-derived (index 0: Q_out) and sensor-derived (index 12:
  q_processed) flow simultaneously; these are distinct but correlated during steady state.
  This is unusual but architecturally defensible (see Section 5).

**Verdict: RECOMMENDED** — see Section 4.

---

### Strategy B — Replacement with Version Bump

Replace indices 0 and/or 1 in obs12_v2 with sensor-derived values. Bump version to obs12_v3.

This was fully evaluated in the M6.2 field audit and ruled out:
- Index 0 (flow): Q_out_Lmin ≠ q_processed_lmin. Divergence is up to 5× during active
  backflush (Q_out = Q_processed × (1 - bf_diversion)).
- Index 1 (pressure): P_in ≠ dp_total_pa. Replacing P_in with dp_total removes the bypass-
  activation signal from the observation.
Both substitutions change the semantic content of the affected index. Any version bump using
substitution is dishonest unless the semantic change is accepted and the policy consequence
is characterized.

**Verdict: RULED OUT** — see M6.2 implementation plan for full derivation.

---

### Strategy C — Full Deferral

Do not extend the schema at all. Leave sensor readings in sensor_state for logging only.
M7 trains on obs12_v2 (truth-derived) with no sensor exposure.

**Pros:** Zero implementation risk; no schema change required.

**Cons:**
- M7 trains without any sensor noise exposure. If the goal is sim-to-real transfer, training
  on perfect-information truth observations and deploying on noisy sensor readings creates a
  distribution mismatch at deployment time.
- The M6.1 sensor models were built specifically to provide realistic sensor inputs for RL
  training. Not using them for training undermines the purpose of M6.1.

**Verdict: DEFERRED, not discarded.** Acceptable if M7 baseline training is scheduled first
and sensor exposure is added in a second training experiment. However, if the intent is to
begin sensor-realistic training immediately, Strategy A is preferred.

## 4. Recommended Default Path

**Strategy A — 12D → 14D add-only extension, designated obs14_v1.**

The extension is narrow, honest, and reversible in the sense that obs12_v2 semantics at
indices 0–11 are preserved unchanged. The new indices carry distinct physical information
(dp_total and q_processed) that does not replace existing channels. The policy trained on
obs14_v1 has simultaneous access to:
- Truth-derived system-level signals (Q_out, P_in) — indices 0, 1
- Sensor-derived filter-stage signals (q_processed noisy, dp_total noisy) — indices 12, 13

This dual-channel structure is appropriate for the current training phase, where the goal is
to establish a policy that is aware of sensor noise while retaining access to the truth
signals needed for bypass and backflush decisions.

**Architecture decision:** The dual-channel design (truth alongside sensor for the same
physical domain) is an internal HydrOS architecture decision. It is not derived from a
specific published standard. It is consistent with the general principle of "privileged
information" in simulation-based RL (discussed in Section 5), but that parallel is
descriptive rather than prescriptive.

## 5. Candidate New Observation Dimensions

### Index 12 — flow_sensor_norm

| Property | Value |
|---|---|
| Source dict | sensor_state |
| Source key | `sensors["flow_sensor_lmin"]` |
| Physical quantity | Noisy estimate of q_processed_lmin [L/min] |
| Noise model | Multiplicative AWGN (1% reading) + per-episode calibration bias (std = 0.2 L/min) |
| Normalization | `flow_sensor_lmin / Q_max_Lmin` = `flow_sensor_lmin / 20.0` |
| Expected range | [0, 0.75] nominal; [0, 1.0] at peak; clip to [0, 1] |
| Semantic relationship to index 0 | Index 0 = Q_out (post-backflush-diversion); index 12 = q_processed (pre-diversion, noisy). Different physical quantities; correlated at steady state. |

The normalization reference (20.0 L/min = Q_max_Lmin) is shared with index 0 (`truth["flow"]`)
for interpretability. The sensor can read slightly above 1.0 due to calibration bias; the
clip to [0, 1] is required to maintain the observation_space contract.

**Sensor behavior already verified:** M6.1 tests confirm flow_sensor_lmin diverges measurably
from truth (test 4 in `tests/test_sensors_m6.py`), does not contaminate truth_state (test 6),
and populates correctly after env.step() (test 9).

---

### Index 13 — dp_sensor_norm

| Property | Value |
|---|---|
| Source dict | sensor_state |
| Source key | `sensors["dp_sensor_kPa"]` |
| Physical quantity | Noisy estimate of dp_total_pa [converted to kPa] |
| Noise model | AWGN (σ = 0.25 kPa) + random walk drift (rate 0.0005 kPa/step, cap ±2.0 kPa) + fouling offset (gain 0.5 × mesh_loading_avg) + 1-step latency |
| Normalization | `dp_sensor_kPa / (P_max_Pa / 1000)` = `dp_sensor_kPa / 80.0` |
| Expected range | [0, ~0.8] at full fouling + nominal flow; [0, 1.0] at system limit; clip to [0, 1] |
| Semantic relationship to index 1 | Index 1 = P_in (inlet absolute pressure, includes valve + pipe); index 13 = dp_total (pure filter differential, noisy). Different physical quantities; correlated through filter resistance component. |

The normalization reference `P_max_Pa / 1000 = 80.0 kPa` reuses an existing YAML parameter
rather than introducing a new constant. The filter differential at maximum fouling and maximum
flow (estimated from hydraulics model: ~63 kPa) is within this range. The cap at 80 kPa is
conservative and physically appropriate.

**Why not add dp_max_kPa as a new YAML parameter:**
A separate `dp_max_kPa` would require a new YAML entry and a new normalization path in
sensor_fusion.py. Reusing `P_max_Pa / 1000` leverages an existing documented constant.
The physical interpretation is: "what fraction of the system's maximum pressure budget is
consumed by the filter differential?" This is policy-relevant for fouling management.

**Sensor behavior already verified:** M6.1 tests confirm dp_sensor_kPa diverges measurably
from truth (test 3 in `tests/test_sensors_m6.py`), 1-step latency is working (test 7), and
truth_state is not contaminated (test 5).

---

### Dual-Channel Design Context

Including both truth-derived and sensor-derived channels for the same physical domain
(flow, pressure) during RL training is a form of the "privileged information" training pattern
documented in the sim-to-real robotics literature. In this pattern, the training policy has
access to state information that will not be available at deployment, with the goal of
shaping the policy toward sensor-observable behavior.

**Relevant literature context (for document annotation):**
- The general principle of training with privileged information while deploying with partial
  observations is discussed in Loquercio et al. 2021, "Learning High-Speed Flight in the Wild"
  (Science Robotics), though the specific formulation differs from HydrOS.
- Chen et al. 2020, "Learning to Walk in Minutes Using Massively Parallel Deep RL" uses
  privileged state during training in a teacher-student framework.
- The original theoretical framing of POMDPs (Kaelbling et al. 1998, "Planning and Acting in
  Partially Observable Stochastic Domains") establishes the distinction between latent state
  (truth) and observations (sensor readings).

**Citation note:** These references describe the general class of approach. The specific
obs14_v1 dual-channel implementation is an **internal HydrOS architecture decision**, not
directly mandated by these sources. Where the document references them, label accordingly.

## 6. Normalization Strategy Options

Three normalization candidates were evaluated for each new index.

### For Index 12 (flow_sensor_norm)

**Option A (recommended):** `flow_sensor_lmin / Q_max_Lmin`
- Reuses existing normalization reference from YAML
- Consistent with index 0 normalization scale for direct comparison
- Conservative: at nominal max flow (15 L/min), value = 0.75; at system max (20 L/min), value = 1.0
- Risk: calibration bias can briefly push above 1.0 — clip required

**Option B:** `flow_sensor_lmin / Q_nominal_max_Lmin` (= flow_sensor_lmin / 15.0)
- Normalizes to nominal operating range; reaches 1.0 at nominal max
- More sensitive to changes within the operating envelope
- Values above 1.0 more frequent due to peak flow regime — less clean for [0,1] enforcement

**Option C:** Running min-max normalization
- Adaptive; not deterministic at training time — breaks reproducibility and violates
  observation space contract
- **Ruled out.**

**Recommendation: Option A.** Q_max_Lmin is already a YAML parameter. No new constant needed.

---

### For Index 13 (dp_sensor_norm)

**Option A (recommended):** `dp_sensor_kPa / (P_max_Pa / 1000)`
- Reuses P_max_Pa from YAML; no new parameter
- Physical interpretation: fraction of maximum system pressure budget consumed by filter
- At full fouling + nominal flow (~63 kPa estimate): value ≈ 0.79 — well-bounded
- Sensor noise + drift can briefly exceed 1.0 — clip required

**Option B:** `dp_sensor_kPa / dp_max_kPa` with new YAML parameter
- More explicit; dp_max_kPa could be tuned independently from P_max_Pa
- Adds one new YAML parameter and one new config path
- Advantage only if dp_total physically diverges significantly from P_max_Pa in future
  calibration — currently unconfirmed

**Option C:** `dp_sensor_kPa / OBS_DP_MAX` (= dp_sensor_kPa / 150.0, CCE constant)
- Uses CCE's normalization reference — semantically wrong for HydrionEnv operating envelope
- **Ruled out.** HydrionEnv and CCE have independent normalization references by design.

**Recommendation: Option A.** Reuse P_max_Pa / 1000. If calibration reveals dp_total
routinely approaches or exceeds this bound, Option B can be adopted with a version bump.

## 7. HydrionEnv vs CCE Rollout Recommendation

### HydrionEnv — Proceed with obs14_v1

HydrionEnv has the full sensor pipeline. DifferentialPressureSensor and FlowRateSensor are
already instantiated and running. sensor_state["dp_sensor_kPa"] and sensor_state["flow_sensor_lmin"]
are populated every step. The only changes required for obs14_v1 are:
1. Two new lines in `sensor_fusion.py` (append indices 12 and 13)
2. `observation_space` shape update in `env.py` (12 → 14)
3. Stale comment correction in `env.py` line 166
4. A new obs bounds regression test

The implementation is narrow and low-risk.

### CCE — Defer Indefinitely (Until Explicitly Scoped)

**Reason 1 — No sensor infrastructure:**
CCE.sensor_state = {}. Introducing sensor-derived obs into CCE requires:
- Instantiating DifferentialPressureSensor and FlowRateSensor in CCE.__init__()
- Adding a reset() and step() call for each sensor in the CCE physics pipeline
- Adding a sensor_state dict with initialization
This is a substantive architecture change to CCE, independent of the schema change.

**Reason 2 — Benchmark protection:**
ppo_cce_v2 is trained on CCE truth-derived obs. Any change to CCE._obs() — even at a
dimension that was previously a clean semantic match (delta_p at index 1) — would invalidate
ppo_cce_v2 as a performance baseline before a replacement exists.

**Reason 3 — No replacement baseline yet:**
Phase 3 (M7 RL rebuild) has not started. There is currently no CCE baseline under any schema
other than the ppo_cce_v2 obs. Invalidating ppo_cce_v2 before Phase 3 produces a successor
is architecturally unsafe.

**CCE sensor obs path — when to revisit:**
After Phase 3 produces a validated HydrionEnv baseline under obs14_v1, a separate "CCE
sensor extension" phase can be scoped. That phase would:
- Add sensor infrastructure to CCE
- Define a CCE-specific obs schema version (separate from HydrionEnv obs14_v1)
- Run CCE training from scratch under the new schema
- Produce a new canonical CCE benchmark artifact

Until that phase is explicitly started, CCE obs remains frozen at its current 12D truth-derived
state.

## 8. Benchmark / Artifact Lineage Implications

### ppo_cce_v2 — UNAFFECTED

ppo_cce_v2 is trained on CCE, which is not changed in M6.2b. Its validity as the canonical
M5 physics benchmark is fully preserved. Any rollout of ppo_cce_v2 produces results
comparable to the original benchmark artifact.

### Any HydrionEnv policy trained under obs12_v2 — INCOMPATIBLE with obs14_v1

obs12_v2 policies have a 12D input. obs14_v1 requires a 14D input. They are architecturally
incompatible at the policy input layer. There is no valid way to run an obs12_v2-trained
policy on obs14_v1 observations without zeroing or padding the new dimensions — and doing
so produces observations outside the training distribution, making any performance metric
non-comparable.

**Implication for Phase 3:** The M7 Phase 3 HydrionEnv training must be run from scratch
under obs14_v1. The resulting artifact is the new HydrionEnv baseline. There is no
ppo_hydrienv_v1 (or equivalent) from prior training to protect because HydrionEnv has not
had a canonical benchmark training run before this point.

### Diagnostic rollouts of obs12_v2 policies on obs14_v1 environments

If obs12_v2 policies are tested on a obs14_v1 environment with the new dimensions zeroed:
- Any performance recorded is diagnostic only
- It characterizes the sensitivity of the obs12_v2 policy to the new observation context
- It is NOT a valid benchmark comparison

Any such diagnostic run must be labeled explicitly as "obs12_v2 policy on obs14_v1 env —
diagnostic only; not a comparable benchmark."

### New HydrionEnv baseline artifact required

Phase 3 must produce: one canonical HydrionEnv policy artifact trained on obs14_v1.
This artifact becomes the new HydrionEnv benchmark for all Phase 4+ comparisons.
The ppo_cce_v2 CCE benchmark and the new HydrionEnv obs14_v1 artifact are maintained as
independent benchmarks for their respective environments.

## 9. Versioning Recommendation

### Version Name: obs14_v1

**Naming convention:**
- First number = observation dimensionality (12, 14, etc.)
- "v" + second number = version within that dimensionality (v1, v2, etc.)
- Version bumps within a dimensionality: when an existing index changes source, physical
  quantity, or normalization reference.
- Dimensionality changes always produce a new major name: obs12 → obs14.

**obs14_v1 applies to HydrionEnv only.** CCE retains its unnamed 12D truth-derived schema.
CCE is not versioned under the obs12/obs14 naming scheme until a deliberate CCE sensor
extension phase is started.

### Version Bump Rules (internal architecture standard)

These rules are not derived from a published RL-specific versioning standard (none exists).
They are analogous to semantic versioning (semver.org) applied to RL observation interfaces.
Label as: **internal HydrOS architecture decision.**

| Event | Version action |
|---|---|
| New index appended (dimensionality increases) | New major name: obs[N+k]_v1 |
| Existing index changes source dict (truth → sensor or vice versa) | Bump minor: obs14_v1 → obs14_v2 |
| Existing index changes physical quantity (same source, different key) | Bump minor |
| Existing index changes normalization reference | Bump minor |
| Schema is semantically equivalent but bug-fixed | Bump minor with "fix" annotation |
| No semantic change, documentation only | No bump |

### Anti-Patterns to Avoid

1. **Silent index drift:** Changing an index source without bumping the version label.
   This is the most dangerous failure mode — it silently invalidates any prior artifact
   comparison without any visible signal.
2. **Normalization aliasing:** Treating numerical closeness as semantic equivalence.
   `flow_sensor_lmin / 20.0 ≈ Q_out_Lmin / 20.0` at steady state, but they are
   different physical quantities. Shared normalization range does not imply shared semantics.
3. **CCE/HydrionEnv version conflation:** Applying the same version label to both environments
   when one changes and the other does not.

### Where the version label lives

`hydrion/sensors/sensor_fusion.py` docstring is the canonical version label for HydrionEnv
obs. It must include: version name, date, and a change-from-previous note for every bump.
The `env.py` observation_space comment is a secondary label that must be kept in sync.

## 10. Open Questions and Calibration-Pending Items

### 10.1 Calibration-Pending (Forwarded from M6.1)

These parameters affect the noise characteristics of indices 12 and 13.
obs14_v1 policy behavior will change quantitatively when calibration replaces placeholders:

| Parameter | Current value | Status | Implication for obs14_v1 |
|---|---|---|---|
| dp_drift_rate_kPa_per_step | 0.0005 | Calibration-pending | dp_sensor_norm will drift differently in real service |
| dp_drift_max_kPa | 2.0 | Calibration-pending | Clamp on drift accumulation; affects sensor noise envelope |
| dp_fouling_gain | 0.5 | Calibration-pending | Fouling-induced bias magnitude at index 13 |
| Turbidity-to-mass calibration | N/A | Calibration-pending | Not obs14 scope; affects existing optical indices |
| DEP Q_crit (7.7 L/min) | Physics-derived | Bench-test pending | Not obs scope; affects physics state |

When calibration values are updated in `configs/default.yaml`, the obs14_v1 schema does not
change — only the noise profile of the sensor-derived indices changes. No version bump is
required for calibration parameter updates.

### 10.2 Empirical Open Questions (Require M7 Training)

1. **Do indices 12 and 13 improve policy performance vs obs12_v2 truth-derived?**
   Cannot be answered before M7 training. This is the primary motivation for the extension.
   If training under obs14_v1 produces a materially better policy for backflush/bypass
   management, the extension is validated. If not, a later phase may drop the sensor indices.

2. **Should truth indices 0 and 1 be kept in obs14_v1, or replaced with duplicate sensor
   readings for deployment realism?**
   During training: keeping truth is appropriate (privileged information pattern). For
   deployment: index 0 and index 1 would need to be replaced with sensor-derived values or
   estimated from sensor readings. This is a deployment-phase question, not a training-phase
   question. Not resolved here.

3. **Optimal normalization for dp_sensor_norm:**
   P_max_Pa / 1000 = 80 kPa is conservative. If calibration shows dp_total at full fouling
   peaks well below 80 kPa, a tighter normalization reference (e.g., 50 kPa) would improve
   sensor resolution in the normalized obs. Revisit after M7 training data is available.

### 10.3 CCE Sensor Extension — Future Phase (Undefined Scope)

Conditions under which a CCE sensor extension phase should be scoped:
- Phase 3 has produced a validated HydrionEnv obs14_v1 baseline
- Empirical evidence exists that sensor-aware training improves policy quality
- A replacement CCE benchmark plan is ready (CCE training run from scratch)
- CCE architecture has been evaluated for sensor pipeline integration

Until all four conditions are met, CCE obs is frozen.
