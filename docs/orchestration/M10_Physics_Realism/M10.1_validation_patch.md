## 3. Ontology Mixing Check

### Can M4 and M5 capture affect the same variable at once?

**NO.** The toggle is an if/else branch at the call site (particles.py:286-291). Under `capture_mode == "m4"`, the three `_capture_eff_s*()` functions execute. Under `capture_mode == "m5_rt"`, the RT-based equivalents execute. Both paths produce `eta_s1`, `eta_s2`, `eta_s3` — local variables consumed by the same downstream assembly logic (`_eta_system`, capture_eff_dense, C_out). Only one branch runs per step.

**However, two distinct "capture efficiency" concepts coexist in truth_state:**

| Key | Source | Affected by toggle? |
|-----|--------|:-------------------:|
| `capture_eff` | CloggingModel — `clogging.py:299-308` | NO — function of `mesh_loading_avg` only |
| `particle_capture_eff` | ParticleModel — `particles.py:352` | YES — depends on per-stage capture |

These are different physical quantities: `capture_eff` is the clogging model's non-monotonic fouling-efficiency curve; `particle_capture_eff` is the particle transport model's dense-phase system efficiency. They happen to share a name fragment but are computed by different subsystems.

**Ontology mixing risk: NONE under the toggle.** Both paths write the same keys to truth_state. No state key is written by both M4 and M5 in the same step. No M4 function is called when M5 is active, or vice versa.

**Residual naming hazard:** The `capture_eff` (clogging) vs `particle_capture_eff` (particles) naming overlap could confuse future developers. The optical sensor reading `truth_state["capture_eff"]` is NOT reading the particle model's capture. This is not a mixing problem — it is a naming clarity problem. Recommend adding a comment at the optical sensor read site during implementation.

## 6. Hidden Coupling Risks

### 6.1 C_out → optical sensor → obs8 (CONFIRMED — not hidden, but missed by refinement)

**Path:** `particles.py:351` writes `C_out` → `optical.py:107` reads `C_out` → computes `sensor_turbidity`, `sensor_scatter` → `sensor_fusion.py:109-126` reads these into obs8 indices 4, 5.

**Severity:** MEDIUM. The reward is unaffected, but the policy's observation input changes. Training dynamics may differ between M4 and M5 modes. This is expected and acceptable (the whole point of physics integration is to change what the system computes), but must be documented — not claimed invisible.

### 6.2 PSD/shape enablement → C_fibers coupling (LATENT)

**Path:** `particles.py:344-346` — if `psd.enabled` AND `shape.enabled`, then `C_fibers = fiber_fraction × C_in` instead of 1.0. C_fibers feeds into `clogging.py:212` deposition rate → fouling → mesh_loading_avg → reward.

**Status:** Disabled in all canonical configs. Guard: M10 config must explicitly keep `psd.enabled: false, shape.enabled: false`.

**Severity:** HIGH if triggered (reward coupling). LOW probability (requires explicit config change).

### 6.3 Stokes settling double-count (CONFIRMED — requires implementation fix)

**Path:** M4 computes `capture_boost_settling = clip(v_s × t_res / h, 0, 0.05)` at `particles.py:307-309` and adds it to `capture_eff_dense` at line 313. M5 RT embeds gravity via the N_G dimensionless number: `0.00338 × A_s × N_G^1.2 × N_R^(-0.4)` — this IS the settling contribution.

Under `capture_mode == "m5_rt"`, if the separate Stokes settling boost is still added, gravity is counted twice: once in η₀ (RT gravity term) and once in the additive boost.

**Fix required:** Under `"m5_rt"` mode, set `capture_boost_settling = 0.0` (skip the Stokes computation). The gravity effect is already in the RT output.

**Severity:** MEDIUM. Double-counting overestimates dense-phase capture by up to 0.05 (the cap). This flows into C_out and then into optical sensors.

### 6.4 Particle size parameter mismatch

**M4 path:** Uses `eta_ref_d_um = 10.0` (µm) — a single reference diameter. No actual PSD.

**M5 RT path:** `rt_single_collector` takes `d_p_m` (meters). The conversion: `d_p_m = eta_ref_d_um × 1e-6`. Both paths evaluate at the same reference diameter.

**Risk:** If the toggle implementation converts units incorrectly (µm vs m), RT produces wildly wrong results. The safest approach: pass `d_p_m = p.eta_ref_d_um * 1e-6` explicitly.

**Severity:** LOW (caught by any sanity check on η₀ output range).

### 6.5 Density handling asymmetry

**M4:** Hard-codes `dense_fraction=0.70, neutral_fraction=0.15, buoyant_fraction=0.15`. Buoyant fraction passes through entirely. Dense gets Stokes + electrostatic boost. Neutral gets electrostatic only.

**M5 RT Phase 1 (as scoped):** RT is called at a single reference diameter with a single polymer density (`rho_dense_kgm3 = 1380.0`). It does NOT automatically compute per-polymer results for PP/PE/PET — that would require calling RT three times with three densities.

**Risk:** The refinement implies RT naturally handles the density split via N_G sign. This is true of the RT formula, but the Phase 1 toggle must decide: (a) call RT once at PET density (matching M4's dense-phase-only approach), or (b) call RT three times per polymer. Option (a) preserves M4's density-split structure; option (b) replaces it. Phase 1 should use option (a) for minimal scope.

**Severity:** MEDIUM — architectural decision that must be made before implementation, not during.

### 6.6 Fouling coupling in M4 vs M5

**M4:** `_capture_eff_s1/s2/s3` include fouling gain terms: `(1 + fouling_gain × fouling)`. Fouling improves capture in M4.

**M5 RT:** The RT formula is a clean-filter formula — it does not include fouling. Fouling coupling in M5 lives in `conical_stage.py:141` as `fouling_boost = clip(0.10 × fouling_frac × eta_RT, 0, 0.05)` — a [DESIGN_DEFAULT] coefficient.

**Risk:** Under `"m5_rt"` mode, the per-stage capture has NO fouling coupling unless explicitly added. This changes the fouling→capture feedback (which in M4 is a positive feedback: more fouling → higher capture → more particles removed → less clogging pressure). Under M5, this feedback disappears.

**Impact on reward:** NONE directly (capture→reward path is severed). Impact on C_out: YES — C_out will respond differently to fouling progression. Impact on obs8 (via optical sensor): YES — C_out changes propagate.

**Severity:** LOW-MEDIUM. The fouling coupling difference is part of the physics improvement. Document it; do not suppress it.

## 7. Failure Mode Enumeration

### FM1: Ontology mixing via conditional import leakage

**Cause:** If M5 modules are imported at module level in `particles.py` (e.g., `from .m5.capture_rt import ...` at the top of the file), they execute even under `capture_mode == "m4"`. If any M5 import has a side effect (unlikely but possible — e.g., modifying a global registry), it could contaminate M4 behavior.

**Detection signal:** Run M4-mode evaluation before and after the M10 code change. Compare `C_out`, `particle_capture_eff`, `eta_system` values step-by-step at fixed seed. Any difference = contamination. Must pass at machine precision.

**Prevention:** Defer M5 imports to inside the `"m5_rt"` branch, or verify M5 modules are side-effect-free (they are — all are pure functions with no global state mutation).

### FM2: Stokes double-count produces non-physical capture values

**Cause:** If `capture_boost_settling` is not zeroed under M5 mode, `capture_eff_dense` includes gravity twice: once in η₀ (via N_G) and once in the additive boost. This inflates dense-phase capture.

**Detection signal:** `capture_boost_settling > 0.0` when `capture_mode == "m5_rt"`. This should always be 0.0 under M5. Assert in test.

**Prevention:** Explicitly set `capture_boost_settling = 0.0` under `"m5_rt"` mode before the assembly step.

### FM3: Obs8 comparison invalidated by unacknowledged sensor perturbation

**Cause:** The M10.1 refinement claims capture is "telemetry only" and defines success criteria based on identical obs8 channels. But obs8 indices 4-5 change under M5 (via C_out → optical sensor). If the evaluation compares M9 and M10 training curves without acknowledging this, any training difference is misattributed to "hidden coupling" rather than the expected observation perturbation.

**Detection signal:** M9 vs M10 obs8 traces at fixed seed diverge at indices 4-5 while indices 0-3, 6-7 match. If the evaluator panics at this and calls it a failure, the comparison is invalidated by a false alarm.

**Prevention:** The evaluation plan must explicitly state: "obs8 indices 4 (sensor_turbidity) and 5 (sensor_scatter) are EXPECTED to differ between M4 and M5 modes due to the C_out → optical sensor pathway. Indices 0-3 and 6-7 must match. Reward must match."

### FM4: RT produces η₀ > 1.0 or η₀ < 0.0 at extreme parameters

**Cause:** The RT gravity term is signed. For very buoyant particles (large negative N_G) at large interception ratios, the total η₀ could go negative (gravity term overwhelms diffusion + interception). The `stage_capture_efficiency()` formula clips to [0, 1], but `rt_single_collector()` returns raw η₀ without clipping.

**Detection signal:** `eta_0 < 0` or `eta_0 > 1` in RT diagnostics at any step. Log and flag.

**Prevention:** The assembly logic in `particles.py` clips `capture_eff_dense` to `[capture_floor, capture_ceiling]` (lines 312-315). This catches extreme values. But the diagnostic output should still flag them for physics review.

### FM5: Unit mismatch between M4 (µm) and M5 (meters)

**Cause:** M4 functions take `d_p_um` (micrometers). M5 `rt_single_collector` takes `d_p_m` (meters). If the toggle passes µm to the RT function, all dimensionless numbers are wrong by 10^6.

**Detection signal:** `N_R = d_p_m / d_c_m` should be O(0.001–0.1) for 10 µm particles on 50–500 µm collectors. If N_R > 1.0 or N_R < 1e-6, unit conversion is wrong.

**Prevention:** Explicit conversion `d_p_m = p.eta_ref_d_um * 1e-6` at the toggle boundary. Unit test with known values.

## 9. Implementation Readiness Verdict

### **NOT READY** — two blocking issues, one required correction

---

### Blocking Issue 1: Stokes settling double-count (Section 6.3)

The M5 RT gravity term (N_G) and the M4 `capture_boost_settling` additive both model particle settling. Under `capture_mode == "m5_rt"`, both would be active, double-counting gravity.

**Required fix:** Add to the toggle logic: when `capture_mode == "m5_rt"`, set `capture_boost_settling = 0.0` and skip the `stokes_velocity_ms()` computation. Document in both code and config comments.

### Blocking Issue 2: Observation coupling not acknowledged (Section 5)

The M10.1 refinement output and M10.0 insight both claim capture is "telemetry only" and do not affect the observation schema. This is factually incorrect: `C_out` feeds into the optical sensor, which feeds into obs8 indices 4-5. The evaluation plan, success criteria, and risk analysis are built on the false premise of observation-decoupling.

**Required fix:** Update M10.1 refinement output Section 5 (Failure Risk Analysis), Section 6 (Evaluation Plan), and the M10.0 insight to acknowledge the `C_out → optical sensor → obs8` pathway. Reclassify obs8 from "UNAFFECTED" to "PARTIALLY AFFECTED (indices 4-5 via C_out)". Update success criteria per Section 8 correction above.

### Non-blocking: Density handling decision (Section 6.5)

Phase 1 must decide whether RT is called once (at PET density, matching M4 dense-phase approach) or three times (per polymer). This is an architectural decision, not a blocking defect. Recommend: call once at PET density for Phase 1 to minimize scope; document multi-polymer evaluation as Phase 1.5.
