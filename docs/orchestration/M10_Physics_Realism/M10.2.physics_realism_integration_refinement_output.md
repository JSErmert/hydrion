## 1. Integration Target Selection

### Selected: Particle capture — Rajagopalan-Tien (1976)

**Replace:** M4 power-law per-stage capture curves (`particles.py:_capture_eff_s1/s2/s3`)
**With:** M5 RT single-collector efficiency (`m5/capture_rt.py:rt_single_collector` + `stage_capture_efficiency`)

### Justification — highest leverage

1. **Foundational subsystem.** Particle capture defines the core function of the device — how efficiently particles are removed per stage. All three stages use capture curves. The M4 implementation is pure phenomenology: `(d_p / mesh_opening)^exponent` with arbitrary exponents 1.5/1.2/0.8. The M5 implementation is the standard liquid-phase filtration formula with diffusion, interception/van der Waals, and gravity terms — each with physical derivation and literature provenance (Rajagopalan & Tien 1976, AIChE J. 22(3):523).

2. **Density split emerges from physics.** M4 hard-codes `dense_fraction=0.70, neutral_fraction=0.15, buoyant_fraction=0.15` and treats buoyant particles as unconditional pass-through. M5 RT handles density via the signed gravity dimensionless number N_G: PP/PE (ρ < ρ_water) get N_G < 0, reducing capture naturally. PET (ρ > ρ_water) gets N_G > 0, enhancing capture. No arbitrary density split needed — the material properties determine the outcome.

3. **Self-contained integration path.** `capture_rt.py` has a clean API: mesh spec → `rt_single_collector()` → η₀ → `stage_capture_efficiency()` → per-stage E. The M4 functions being replaced are standalone module-level functions with no external dependencies. The swap is structurally clean.

4. **Mesh specs already defined.** `MESH_S1` (500 µm), `MESH_S2` (100 µm), `MESH_S3_MEMBRANE` (5 µm) are already specified in `capture_rt.py` with design-default wire diameters. The RT formula can run immediately with these specs.

### Why electrostatics (nDEP) is deferred

1. **Geometric dependency.** nDEP requires conical electrode geometry (tip radius, electrode gap) and `grad_E2` estimation. The current production code uses cylindrical radial geometry (`r_inner_m`, `r_outer_m`). The M5 nDEP module uses `DEPConfig` with hemisphere-on-post approximation flagged for FEM replacement. Integrating nDEP requires resolving the geometry mismatch — a design decision, not just a code swap.

2. **Field model incomplete.** `dep_ndep.py` documents its `grad_E2` approximation as "APPROXIMATE — replace with FEM simulation value for precision" in both code comments and `M5_PHYSICS_BASIS.md §7`. Integrating an approximation flagged by its own authors as insufficient is not a grounding improvement.

3. **Higher coupling complexity.** nDEP interacts with voltage dynamics, residence time, and field normalization — all currently computed in `electrostatics.py`. The integration surface is larger and touches the action→state mapping for the voltage command. RT capture only touches the state computation, not the action interface.

4. **nDEP is Phase 2 material.** Once RT capture is validated, the nDEP integration can build on the same toggle infrastructure and the RT-corrected baseline. Doing nDEP first would require building toggle infrastructure AND resolving geometry simultaneously.

## 2. Production vs Research Gap Definition

### What M4 currently does (production path)

**File:** `hydrion/physics/particles.py` (lines 75–108)

Per-stage capture efficiency at a reference particle diameter (`eta_ref_d_um = 10.0 µm`):

| Stage | Formula | Exponent | Reference |
|-------|---------|:--------:|-----------|
| S1 (500 µm) | `(d_p / 500)^1.5 × (1 + 0.15 × fouling)` | 1.5 | None — design curve |
| S2 (100 µm) | `(d_p / 100)^1.2 × (1 + 0.20 × fouling)` | 1.2 | None — design curve |
| S3 (5 µm) | `(d_p / 5)^0.8 × flow_penalty × (1 + 0.10 × fouling)` | 0.8 | None — design curve |

System efficiency: `η = 1 − (1−η₁)(1−η₂)(1−η₃)`

Density handling: hard-coded fractions (70/15/15). Buoyant fraction passes through entirely. Dense-phase gets Stokes settling boost (capped at 0.05). Electrostatic boost added as `alpha_E × E_capture_gain` (pDEP-flavored).

### What M5 provides (research path)

**File:** `hydrion/physics/m5/capture_rt.py`

RT single-collector contact efficiency:

```
η₀ = 4.0 A_s^(1/3) N_Pe^(-2/3)           [diffusion]
   + A_s N_Lo^(1/8) N_R^(15/8)            [interception + van der Waals]
   + 0.00338 A_s N_G^(1.2) N_R^(-0.4)     [gravity — SIGNED]
```

Stage efficiency from single-collector: `E = 1 − exp(−4 α η₀ L / (π d_c))`

Uses Happel sphere-in-cell flow parameter, Stokes-Einstein Brownian diffusivity, Hamaker constant for van der Waals. Mesh specs include solidity calculation from wire diameter and opening.

### Concrete inconsistencies

| Aspect | M4 Production | M5 Research | Inconsistency |
|--------|---------------|-------------|---------------|
| **Capture formula** | `(d_p / opening)^exp` power law | RT 1976: diffusion + interception + gravity | Different functional form; different size-dependence shape |
| **Size exponents** | 1.5 / 1.2 / 0.8 (arbitrary) | Derived from N_Pe^(-2/3), N_R^(15/8), N_G^(1.2) | M4 misses the interception peak — RT has non-monotonic size dependence |
| **Density handling** | Hard-coded 70/15/15 split; buoyant = unconditional pass-through | N_G sign determines gravity contribution per polymer | M4 treats PP/PE as completely uncapturable; RT computes their (reduced) capture rate |
| **Fouling coupling** | Linear gain: `1 + gain × fouling` | Not in RT (RT is a clean-filter formula) | M4 bakes fouling into capture; M5 separates capture physics from cake dynamics |
| **Stage parameters** | Mesh opening only | Wire diameter, collector diameter, solidity | M4 ignores mesh structure; RT needs it for Happel and bed-depth calculation |
| **Stokes settling** | Separate additive boost (capped at 0.05) | Embedded in N_G gravity term | M4 double-counts settling if both Stokes boost and capture curve applied |

## 3. Integration Architecture

### Toggle architecture — `physics_mode` config flag

```python
# In config YAML:
physics:
  capture_mode: "m4"     # "m4" = production power-law (M9 baseline)
                          # "m5_rt" = Rajagopalan-Tien 1976

# In ParticleModel.__init__():
self._capture_mode = cfg_raw.get("physics", {}).get("capture_mode", "m4")

# In ParticleModel.update():

    # Use RT single-collector per stage
    eta_s1 = _capture_rt_stage(d_p_m, MESH_S1, U_ms, polymer, ...)
    eta_s2 = _capture_rt_stage(d_p_m, MESH_S2, U_ms, polymer, ...)
    eta_s3 = _capture_rt_stage(d_p_m, MESH_S3, U_ms, polymer, ...)

    # Existing M4 power-law path (unchanged)
    eta_s1 = _capture_eff_s1(d_um, ff_s1, p.fouling_gain_s1)
    eta_s2 = _capture_eff_s2(d_um, ff_s2, p.fouling_gain_s2)
    eta_s3 = _capture_eff_s3(...)
```

## 5. Failure Risk Analysis

*[Revised by M10.1A correction patch — 2026-04-16]*

### Highest probability failure: RT capture values numerically different from M4

**Probability:** HIGH (near-certain — this is expected, not a failure per se)
**Impact:** LOW for reward (decoupled); MEDIUM for observations (partially coupled via optical sensor)

RT single-collector efficiency at 10 µm through a 500 µm mesh will produce a different numerical capture rate than M4's `(10/500)^1.5`. The M10 evaluation will show different `capture_eff`, `C_out`, `eta_system` values. This is expected and the entire point of the integration.

**Reward decoupling (confirmed):** With PSD and shape both disabled (current config), `particles.py:344-346` sets `C_fibers = 1.0` unconditionally. The particle model output does not feed back into clogging dynamics. The reward function (env.py:303-308) uses only `q_processed_lmin`, `pressure`, `mesh_loading_avg`, `q_bypass_lmin`, and `bf_active` — all from hydraulics and clogging. **The reward is capture-decoupled.**

**Observation partial coupling (corrected):** `C_out` (written by `particles.py:351`) is read by `optical.py:107` and used to compute `sensor_turbidity` (gain 0.6) and `sensor_scatter` (gain 0.6). These feed into obs8 indices 4 and 5. Therefore, **obs8 indices 4-5 will differ between M4 and M5 modes.** This is expected and correct — the optical sensor should observe different particle concentrations when the physics model changes. It is not a hidden coupling; it is a correct downstream consequence.

Note: `optical.py:106` also reads `truth_state["capture_eff"]`, but this key is written by **CloggingModel** (`clogging.py:379`), not by ParticleModel. The clogging-sourced capture_eff depends only on `mesh_loading_avg` and is unaffected by M10.

Consequence: changing the capture model changes the reward gradient zero, changes clogging dynamics zero, but changes obs8 indices 4-5 via C_out → optical sensor. Training may show minor policy differences due to changed observation inputs, even though the reward signal is identical.

### Stokes settling double-count (BLOCKING — guarded)

**Probability:** CERTAIN if unguarded
**Impact:** MEDIUM (inflates dense-phase capture by up to 0.05)

M5 RT embeds gravity in the N_G dimensionless number. M4 adds a separate `capture_boost_settling` (capped at 0.05). Under `capture_mode == "m5_rt"`, the Stokes settling additive MUST be set to 0.0 to prevent double-counting. **This is a mandatory implementation guard.**

### Highest impact failure: hidden coupling path not identified

**Probability:** LOW (one path exists — see below)
**Impact:** HIGH (if triggered, would silently change reward behavior)

The one coupling path that could activate: if a future config enables `psd.enabled: true` AND `shape.enabled: true`, then `C_fibers = fiber_fraction × C_in` instead of 1.0. This feeds into clogging deposition rate, creating a particle→clogging→reward feedback loop. The toggle must document that `capture_mode: "m5_rt"` has been validated ONLY with PSD/shape disabled.

### Risk matrix

| Risk | Probability | Impact | Mitigation |
|------|:-----------:|:------:|------------|
| **RT numerical divergence** | HIGH | LOW | Expected; reward-decoupled. Obs indices 4-5 differ (expected). |
| **C_out → optical → obs8 coupling** | HIGH | MEDIUM | Expected and documented. obs8 indices 4-5 change via C_out. Not a failure — a correct consequence. Evaluation plan distinguishes must-match vs expected-to-differ channels. |
| **Stokes settling double-count** | CERTAIN if unguarded | MEDIUM | **MANDATORY GUARD:** set `capture_boost_settling = 0.0` under `"m5_rt"`. Verification test: assert zero at every step. |
| **Mesh spec design defaults** | MEDIUM | MEDIUM | `MESH_S1/S2/S3` wire diameters are `[DESIGN_DEFAULT]`. RT results are physically derived but geometry-approximate. Document as design-default capture, not hardware-calibrated capture. |
| **Stage 3 membrane regime** | MEDIUM | LOW | `MESH_S3_MEMBRANE` (5 µm) is flagged "microporous membrane regime, not woven mesh; treat with caution." RT assumes sphere-in-cell (granular bed). Single-screen `bed_depth_m = d_c_m` mitigates but doesn't fully resolve. Flag for future work. |
| **PSD/shape enables coupling** | LOW | HIGH | Guard toggle with validation: M10 config must have `psd.enabled: false, shape.enabled: false`. Document in config comments. |
| **Training instability** | LOW | LOW-MEDIUM | Reward is decoupled. Obs8 indices 4-5 change (documented). Policy gradient sees same reward but slightly different inputs on 2 of 8 channels. Monitor for drift. |

### Success criteria

1. **M10 policy trains to comparable return as M9 canonical** (within M9 Config A/B range: 694-848). No higher return required.
2. **AC11 gate passes** at all checkpoints.
3. **Reward trajectory matches** between M4 and M5 at fixed seed within floating-point tolerance. Divergence is a FAILURE.
4. **Clogging trajectory matches** (mesh_loading_avg, fouling_frac_s*) between M4 and M5 at fixed seed. Divergence is a FAILURE.
5. **obs8 indices 0-3 and 6-7 match** between M4 and M5 at fixed seed. Divergence is a FAILURE.
6. **obs8 indices 4-5 differ** between M4 and M5 (expected — confirms C_out propagation through optical sensor). Identity is a FAILURE (RT not active).
7. **capture_eff trajectory differs** from M9 (expected — confirms RT is producing different values).
8. **`capture_boost_settling == 0.0`** at all steps under `"m5_rt"` mode. Nonzero is a FAILURE (Stokes double-count).
9. **RT diagnostic outputs are physically plausible**: N_G < 0 for PP/PE, N_G > 0 for PET; η₀ within [0, 1]; diffusion-dominant at small sizes, interception-dominant at large sizes.

### What success means

M10 success means: the production simulation runs source-grounded filtration physics, the policy trains stably against it, and the M9 canonical result remains reproducible via config toggle. The capture telemetry now reflects RT 1976 instead of arbitrary power-law curves. The optical sensor correctly observes the changed particle concentration (obs8 indices 4-5), while reward, clogging, and hydraulics remain identical. This does not mean capture is "correct" (mesh specs are design defaults) — it means capture is physics-derived rather than assumption-derived.
