## Corrections Applied

### Correction 1: Stokes Settling Guard

**Finding:** M5 RT embeds particle settling via the gravity dimensionless number N_G (`0.00338 × A_s × N_G^1.2 × N_R^(-0.4)`). M4 computes a separate additive `capture_boost_settling = clip(v_s × t_res / h, 0, 0.05)` at `particles.py:307-309`. Under `capture_mode == "m5_rt"`, both would be active, double-counting gravity.

**Correction:** Under `capture_mode == "m5_rt"`:
- `capture_boost_settling` MUST be set to `0.0`
- The `stokes_velocity_ms()` call and residence-time computation for settling MUST be skipped or their result discarded
- Reason: RT N_G already accounts for gravity-assisted and gravity-opposed capture. Adding the Stokes boost double-counts settling for dense particles and has no M5 counterpart for buoyant particles.

**Implementation guard:**
```python
if self._capture_mode == "m5_rt":
    capture_boost_settling = 0.0   # RT N_G embeds gravity; Stokes boost would double-count
else:
    # existing M4 Stokes settling computation (unchanged)
    ...
```

**Verification test:** Assert `capture_boost_settling == 0.0` at every step when `capture_mode == "m5_rt"`. Presence of a nonzero value under M5 mode is a test failure.

### Correction 2: Observation Coupling Acknowledgment

**Finding:** The M10.1 refinement and M10.0 insight claimed capture changes are "telemetry only" — invisible to both reward and observations. The reward-decoupling claim is correct. The observation-decoupling claim is incorrect.

**Coupling path identified by validation patch:**

```
particles.py:351  →  truth_state["C_out"]
                         ↓
optical.py:107    →  C_out read
                         ↓
optical.py:114-118 → sensor_turbidity = 0.8×mesh_avg + 0.6×C_out + 0.4×(1−capture_eff_clog)
optical.py:122-126 → sensor_scatter   = 0.4×flow_norm + 0.6×C_out
                         ↓
sensor_state["sensor_turbidity"], sensor_state["sensor_scatter"]
                         ↓
sensor_fusion.py:115-116 → obs8 index 4 (sensor_turbidity), obs8 index 5 (sensor_scatter)
```

**Key disambiguation:** `optical.py:106` reads `truth_state["capture_eff"]` — this is from **CloggingModel** (`clogging.py:379`), a function of `mesh_loading_avg` only, **NOT** from ParticleModel. The clogging-sourced `capture_eff` is unaffected by M10. The coupling is specifically through `C_out`, not through `capture_eff`.

**Corrected system statement:**

> M10 capture integration is **reward-decoupled** but **not observation-decoupled**. Changing the capture model changes `C_out`, which propagates through the optical sensor into obs8 indices 4 (sensor_turbidity) and 5 (sensor_scatter). This is expected and acceptable — the optical sensor SHOULD observe different particle concentrations when the physics model changes. It is not a hidden coupling; it is a correct downstream consequence of improved capture physics.
