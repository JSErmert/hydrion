## 1. Executive Objective

M7's first and only objective is to produce the initial canonical HydrionEnv policy artifact
trained under the `obs14_v1` observation schema. This is not a reward redesign, not a
robustness expansion, and not a deployment exercise. It is the minimum required training run
that establishes a valid HydrionEnv performance baseline — the reference point against which
all future HydrionEnv policy improvements will be measured. The baseline must be produced
with a reproducible seed, correct obs schema version-lock, and a complete metadata record.
Until this artifact exists, no HydrionEnv policy comparison is valid, and no subsequent M7+
work (reward shaping, curriculum training, sensor noise ablation) has a reference to compare
against.

## 3. Baseline Artifact Objective

### Required artifact set: `ppo_hydrienv_v1`

The M7 baseline training run must produce the following three files:

| Artifact | Path | Description |
|---|---|---|
| Policy | `models/ppo_hydrienv_v1.zip` | Trained PPO policy (stable-baselines3 format) |
| VecNormalize | `models/ppo_hydrienv_v1_vecnorm.pkl` | Observation/reward normalization statistics |
| Metadata | `models/ppo_hydrienv_v1_meta.json` | Schema version, seed, steps, reward version, env path |

The `v1` suffix denotes the first canonical HydrionEnv artifact. It does NOT imply
comparability with `ppo_cce_v2`. These are artifacts from different environments.

### Required metadata fields (`ppo_hydrienv_v1_meta.json`)

```json
{
  "obs_schema":        "obs14_v1",
  "action_schema":     "act4_v1",
  "env":               "HydrionEnv",
  "train_seed":        42,
  "total_timesteps":   500000,
  "reward_version":    "phase1_v1",
  "d_p_um":            1.0,
  "benchmark_regime":  "submicron",
  "note":              "First canonical HydrionEnv baseline under obs14_v1. Not comparable to ppo_cce_v2 (different environment)."
}
```

The `obs_schema` field must be set to `"obs14_v1"` and must be validated against
`HydrionEnv().observation_space.shape` at script startup — fail fast if shape != (14,).
This prevents silent schema drift from invalidating the artifact.

## 4. Environment and Schema Scope

### HydrionEnv — training target for M7 baseline

HydrionEnv is the only training environment for this phase. It provides:
- Full sensor pipeline (optical + hydraulic)
- obs14_v1: 14D observation with both truth-derived (indices 0–9) and sensor-derived (indices 10–13) channels
- ShieldedEnv pressure/clog safety wrapper (must be applied — see Section 5)
- Reproducible seeding via `env.reset(seed=...)`

### CCE — explicitly excluded from M7 baseline scope

CCE is not modified in M7. Its observation schema, reward function, and training
infrastructure remain unchanged. ppo_cce_v2 remains the canonical CCE artifact.

**Any training run on CCE in M7 must use the existing `train_ppo_cce.py` and is not the
M7 baseline.** If a parallel CCE experiment is conducted, it is logged separately and does
not constitute the M7 baseline artifact.

### obs14_v1 schema

`hydrion/sensors/sensor_fusion.py` is the canonical source. The training script must
import `HydrionEnv` and assert `env.observation_space.shape == (14,)` at startup.

## 6. Validation and Evaluation Expectations

### Minimum required outputs from the M7 baseline training run

1. **Training convergence record**
   - TensorBoard log directory under `runs/ppo_hydrienv_v1_<timestamp>/`
   - Episode return curve shows learning signal (monotonically improving or stabilizing trend by 300k–500k steps)
   - No sustained reward collapse after 100k steps

2. **Artifact integrity check**
   - All three artifact files (`ppo_hydrienv_v1.zip`, `ppo_hydrienv_v1_vecnorm.pkl`,
     `ppo_hydrienv_v1_meta.json`) exist and are non-empty after training
   - Metadata JSON contains correct `obs_schema = "obs14_v1"` field

3. **Post-training evaluation** via `eval_ppo_hydrienv_v1.py`
   - Minimum 5 deterministic evaluation episodes (seed=100, deterministic=True)
   - Record per-episode: return, episode length, safety violations, ShieldedEnv projections
   - Record mean and std of return across eval episodes

4. **Sensor channel diagnostics** (new requirement vs CCE baseline)
   - During eval, log mean values of obs[12] and obs[13] per episode
   - Confirm both channels are non-zero and vary across episodes (sensor pipeline is active)
   - This is not a performance criterion — it is a sensor integration sanity check

5. **Schema consistency check** (must be logged, not just asserted)
   - Before training: log `env.observation_space.shape` to stdout
   - Before eval: verify loaded model input dimension matches (14,)

### What counts as a valid first baseline

The M7 baseline is valid if:
- Training ran for ≥ 500k steps without crash
- All three artifacts were saved and are loadable
- Eval episodes complete without exception
- Mean eval return > mean eval return of a random-action baseline on the same env
- Sensor channels (obs[12], obs[13]) are demonstrably non-zero during eval
- Metadata correctly records obs_schema = "obs14_v1"

The M7 baseline does **not** require:
- Achieving a specific reward threshold
- Outperforming any prior policy (no prior HydrionEnv obs14_v1 policy exists to compare)
- Matching CCE reward levels (different environment)
- Demonstrating deployment readiness

## 9. Risks and Failure Conditions

### Risk 1 — Benchmark lineage confusion

**Failure mode:** The M7 training run uses `train_ppo.py` (the existing stale script), which
saves artifacts named `ppo_hydrion_final_12d`. These names suggest a 12D schema. Any future
developer sees these files and cannot determine which schema they correspond to.

**Prevention:** Do not use `train_ppo.py` or `train_ppo_v15.py` for M7. Create
`train_ppo_hydrienv_v1.py` with explicit naming and schema-lock assertion.

---

### Risk 2 — Schema confusion at load time

**Failure mode:** A model trained on obs14_v1 (14D) is loaded and evaluated against an
environment that has been inadvertently configured differently (e.g., by importing the wrong
env class, or if a future schema change is made without bumping the version label).

**Prevention:** The training script must assert `env.observation_space.shape == (14,)` at
startup and fail fast. The eval script must verify the loaded model's input dimension matches
the env's observation_space before running any episodes.

---

### Risk 3 — Improper CCE / HydrionEnv comparison

**Failure mode:** After M7 training, a report compares `ppo_hydrienv_v1` reward numbers
against `ppo_cce_v2` reward numbers and draws conclusions about relative policy quality.
The comparison is meaningless — different environments, different reward functions, different
obs schemas, different physics.

**Prevention:** Section 5 explicitly prohibits this comparison. Any report must state the
environment and obs schema for every metric. Cross-environment comparisons require explicit
experimental controls that do not exist at this phase.

---

### Risk 4 — Treating the first baseline as deployment-valid

**Failure mode:** M7 baseline training completes and the resulting policy is treated as
ready for hardware deployment or final sim-to-real transfer evaluation.

**Prevention:** The M7 baseline is a simulation-only artifact trained on a simulator with
calibration-pending sensor parameters. It is explicitly not deployment-valid. The metadata
note field must state this. Any deployment evaluation requires: completed hardware
calibration (replacing M6.1 placeholder parameters), domain randomization characterization,
and sim-to-real gap analysis — none of which are in M7 scope.

---

### Risk 5 — VecNormalize statistics mismatch

**Failure mode:** The eval script loads a VecNormalize statistics file from a prior
(obs12_v2-era) run. The normalization statistics were computed over a different observation
distribution (12D). Applied to a 14D obs, this either crashes or silently corrupts the
normalized input.

**Prevention:** The eval script must load only the VecNormalize file produced by the
corresponding `train_ppo_hydrienv_v1.py` run. Old VecNormalize files are incompatible and
must not be reused. Artifact naming (v1 suffix on all three files) enforces this.

---

### Risk 6 — No random-action baseline recorded

**Failure mode:** AC7 requires mean eval return > random-action baseline, but no
random-action baseline was ever recorded for HydrionEnv obs14_v1. The acceptance criterion
cannot be verified.

**Prevention:** The eval script (or a companion script) must run a random-action episode set
(n=5, same seed=100) and record its mean return alongside the policy eval. This is a
one-time measurement, not ongoing benchmarking.
