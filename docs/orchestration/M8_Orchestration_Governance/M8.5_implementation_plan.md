## 1. Executive Implementation Strategy

Add an `obs_schema` parameter to `HydrionEnv` (defaulting to `"obs14_v1"`) that selects
between the existing 14D observation and the new 8D `obs8_deployment_v1` observation; add
a new `build_observation_obs8()` function to `sensor_fusion.py` (additive — existing
`build_observation()` is unchanged); create `train_ppo_hydrienv_v2.py` and
`eval_ppo_hydrienv_v2.py` that use `obs_schema="obs8_deployment_v1"` and enforce schema
lock assertions; add `tests/test_obs8_schema.py` for AC12; and produce
`models/ppo_hydrienv_v2_meta.json` with the full channel taxonomy. Every change is
additive or parametric — no existing behavior, artifact, or test is modified.

```python
def build_observation_obs8(truth: dict, sensors: dict) -> np.ndarray:
    """
    obs8_deployment_v1 — deployment-realistic 8D observation contract.

    Retains only channels available at hardware deployment:
        - actuator_feedback channels (obs14_v1 indices 6–9, re-indexed 0–3)
        - sensor_derived channels    (obs14_v1 indices 10–13, re-indexed 4–7)

    Removed: obs14_v1 indices 0–5 (physics_truth — simulation-only quantities).

    Index mapping (obs8_deployment_v1):
        0  valve_cmd         truth_state    actuator command (obs14_v1 index 6)
        1  pump_cmd          truth_state    actuator command (obs14_v1 index 7)
        2  bf_cmd            truth_state    actuator command (obs14_v1 index 8)
        3  node_voltage_cmd  truth_state    actuator command (obs14_v1 index 9)
        4  sensor_turbidity  sensor_state   optical turbidity (obs14_v1 index 10)
        5  sensor_scatter    sensor_state   optical scatter   (obs14_v1 index 11)
        6  flow_sensor_norm  sensor_state   flow_sensor_lmin / 20.0, clip [0,1] (obs14_v1 index 12)
        7  dp_sensor_norm    sensor_state   dp_sensor_kPa / 80.0, clip [0,1]   (obs14_v1 index 13)

    Schema version: obs8_deployment_v1 (M8 — 2026-04-14)
    This schema is deployment-realistic, NOT sensor-only.
    Indices 0–3 are actuator command feedback (controller-generated, always self-known).
    Indices 4–7 are sensor-derived (hardware-measurable).

    Source traceability: M8.1R.4_sources_map.md S1 (POMDP self-knowledge),
    S4 (action history), S5 (asymmetric actor-critic deployment obs).

    DO NOT change index ordering without bumping the schema version label.
    obs14_v1 is preserved in build_observation() — these are parallel functions.
    """
    return np.array(
        [
            # actuator_feedback channels (controller-issued, always deployment-available)
            float(truth.get("valve_cmd",         0.0)),
            float(truth.get("pump_cmd",          0.0)),
            float(truth.get("bf_cmd",            0.0)),
            float(truth.get("node_voltage_cmd",  0.0)),

            # sensor_derived channels (hardware-measurable)
            float(sensors.get("sensor_turbidity", 0.0)),
            float(sensors.get("sensor_scatter",   0.0)),

            # hydraulic sensors — same normalization as obs14_v1 indices 12 and 13
            float(np.clip(sensors.get("flow_sensor_lmin", 0.0) / 20.0, 0.0, 1.0)),
            float(np.clip(sensors.get("dp_sensor_kPa",   0.0) / 80.0, 0.0, 1.0)),
        ],
        dtype=np.float32,
    )
```

```python
def _run_ppo_v2_episodes(vec_env, model, n_episodes: int) -> dict:
    """
    Run ppo_hydrienv_v2 for n_episodes.
    obs8_deployment_v1 channel diagnostics:
        obs8 indices 0–3: actuator_feedback (obs14_v1 indices 6–9)
        obs8 indices 4–7: sensor_derived    (obs14_v1 indices 10–13)
    """
    returns, ep_lengths, violations = [], [], []
    # Actuator channel accumulators (obs8 indices 0–3)
    act_means, act_stds = [], []
    # Sensor channel accumulators (obs8 indices 4–7)
    sensor_means, sensor_stds = [], []

    for ep in range(n_episodes):
        obs = vec_env.reset()
        done = False
        ep_ret, ep_steps, ep_viol = 0.0, 0, 0
        ep_act_vals    = []   # all obs8[0–3] values across steps
        ep_sensor_vals = []   # all obs8[4–7] values across steps

        while not done:
            action, _ = model.predict(obs, deterministic=True)
            obs, reward, done_arr, info_arr = vec_env.step(action)
            done = bool(done_arr[0])
            ep_ret   += float(reward[0])
            ep_steps += 1

            ep_act_vals.append(obs[0][0:4].tolist())
            ep_sensor_vals.append(obs[0][4:8].tolist())

            info   = info_arr[0]
            safety = info.get("safety", {})
            if any([
                safety.get("soft_pressure_violation"),
                safety.get("hard_pressure_violation"),
                safety.get("soft_clog_violation"),
                safety.get("hard_clog_violation"),
                safety.get("blockage_violation"),
            ]):
                ep_viol += 1

        returns.append(ep_ret)
        ep_lengths.append(ep_steps)
        violations.append(ep_viol)

        act_arr    = np.array(ep_act_vals)     # shape (steps, 4)
        sensor_arr = np.array(ep_sensor_vals)  # shape (steps, 4)

        act_means.append(float(np.mean(act_arr)))
        act_stds.append(float(np.std(act_arr)))
        sensor_means.append(float(np.mean(sensor_arr)))
        sensor_stds.append(float(np.std(sensor_arr)))

        print(
            f"  [V2  ] ep {ep:02d}: "
            f"return={ep_ret:8.3f}  steps={ep_steps}  violations={ep_viol}  "
            f"act_mean={act_means[-1]:.4f}  sensor_mean={sensor_means[-1]:.4f}"
        )

    return {
        "mean_return":          float(np.mean(returns)),
        "std_return":           float(np.std(returns)),
        "mean_ep_length":       float(np.mean(ep_lengths)),
        "mean_violations":      float(np.mean(violations)),
        "act_global_mean":      float(np.mean(act_means)),
        "act_global_std":       float(np.mean(act_stds)),
        "sensor_global_mean":   float(np.mean(sensor_means)),
        "sensor_global_std":    float(np.mean(sensor_stds)),
    }
```

## 3. Schema Implementation Plan

### 3.1 Implementation approach: additive function + parametric env

The narrowest correct approach:

1. **New function in sensor_fusion.py** — `build_observation_obs8()`. The existing
   `build_observation()` is not touched. The two functions are parallel, independently
   testable, and independently versioned.

2. **`obs_schema` parameter in HydrionEnv** — Default `"obs14_v1"` means zero behavior
   change for any existing code, script, or test. The parameter is validated at init
   time against a `_VALID_SCHEMAS` dict that maps schema name → dimension.

3. **`observation_space` shape** — Derived from `_VALID_SCHEMAS[obs_schema]` at init time.
   Both the space and the actual observation vector are determined by the same constant.

4. **`_observe()` dispatch** — Single `if self.obs_schema == "obs8_deployment_v1"` branch.
   All other schemas fall through to `build_observation()`. No schema parameter is passed
   to the observation functions — the dispatch is in the env, not in sensor_fusion.

### 3.2 How obs14_v1 is preserved

- `build_observation()` — unmodified. No arguments added, no logic changed.
- `HydrionEnv()` without `obs_schema` — still obs14_v1 (default parameter).
- `test_obs14_v1.py` — all 15+ tests call `HydrionEnv()` without `obs_schema`;
  they continue to pass without modification.
- No existing training or eval script references `obs_schema` — they instantiate
  `HydrionEnv()` without it.

### 3.3 How silent schema drift is prevented

- Schema lock assertion in `train_ppo_hydrienv_v2.py`: `assert shape == (8,)` at startup.
  Training fails immediately if the env returns the wrong shape.
- Schema lock assertion in `eval_ppo_hydrienv_v2.py`: checks both v2 `(8,)` and v1 `(14,)`
  after loading.
- `_VALID_SCHEMAS` dict in `HydrionEnv.__init__`: unknown schema names raise `ValueError`
  immediately, before any physics initialization.
- Test T7 (`test_unknown_schema_raises`) verifies the validation.

## 9. Risks, Confounds, and Operational Safeguards

### S1 — VecNorm confusion safeguard

**Risk:** Loading `ppo_hydrienv_v1_vecnorm.pkl` for v2 evaluation produces invalid
normalization for the 8D observation space (dimension mismatch or silent misscaling).

**Safeguard:**
- Named constants `_V2_VECNORM_PATH` and `_V1_VECNORM_PATH` in eval script.
- Each constant is used in exactly one `VecNormalize.load()` call bound to its own vec env.
- Schema lock assertions after loading verify dimension alignment.
- AC5 verification: code review of load paths.

### S2 — Schema-mismatch safeguard

**Risk:** env.py `obs_schema` parameter could be omitted or misnamed in the training
script, silently defaulting to obs14_v1.

**Safeguard:**
- Schema lock assertion in train script: `assert observation_space.shape == (8,)` before
  any training infrastructure. Training fails immediately with clear message.
- `_OBS_SCHEMA = "obs8_deployment_v1"` is a named constant — passed explicitly to
  `HydrionEnv(obs_schema=_OBS_SCHEMA)`.
- `_VALID_SCHEMAS` validation in env.py raises `ValueError` for unknown strings.
- Test T7 in test_obs8_schema.py validates the ValueError path.

### S3 — Manual training runtime note

**Risk:** Running `train_ppo_hydrienv_v2.py` as a background Bash process causes the
process to be killed at ~5 minutes (documented M7 platform constraint). Training requires
~15–16 minutes.

**Safeguard:** Training must be run as a manual foreground command in a terminal:
```
python -m hydrion.train_ppo_hydrienv_v2
```
This is documented in the training script's module docstring and main() output.
The M8 execution prompt must explicitly remind the user of this constraint.

### S4 — Single-seed interpretation caution

**Risk:** M8 produces a single training data point per schema. The performance delta may
not be representative of average behavior across initialization seeds.

**Safeguard:** The calibration confound statement (MVC-3) and result interpretation
(Section 9 four-scenario table) both mention single seed as an acknowledged limitation.
The eval script prints this. The metadata `confounds` field names it explicitly.

### S5 — Placeholder-noise confound propagation

**Risk:** The sensor noise in obs8_deployment_v1 is governed by calibration-pending
parameters. If the obs8 policy underperforms, the degradation may reflect noise quality
rather than information loss.

**Safeguard:** MVC-3 (calibration confound statement) is printed verbatim in:
- Training script final output
- Evaluation script final output
- `ppo_hydrienv_v2_meta.json` `calibration_status` field
- `confounds` array in metadata

This ensures the confound appears in every artifact that records M8 results.
