# Calibration Integration Notes

This document defines how calibration information is integrated into HydrOS.

Its purpose is to ensure that:
- research findings are converted into usable simulator parameters
- evidence quality remains visible
- literature, proxy assumptions, and bench data are not conflated
- calibration changes remain validation-aware

---

# 1. Integration Philosophy

HydrOS calibration does not begin with “final truth.”

It begins with:
- literature-supported anchors
- proxy assumptions where necessary
- explicit placeholders where evidence is weak
- planned bench replacement

Calibration must therefore be staged.

---

# 2. Calibration Layers

HydrOS should treat parameter integration as three layers:

## Layer 1 — Literature Initialization
These are first-pass values or ranges drawn from:
- direct data
- strong engineering references
- adjacent-domain literature

Primary file:

```text
configs/calibration/literature_v1.yaml
``` id="wg6wpj"

---

## Layer 2 — Proxy-Constrained Modeling
These are values that are not directly measured for the exact HydrOS device, but are still useful as bounded engineering assumptions.

These must remain labeled as:
- proxy
- medium/low confidence
- pending replacement

---

## Layer 3 — Bench Replacement
These are values that eventually replace proxy or placeholder parameters once HydrOS-specific experimental data exists.

Future example:

```text
configs/calibration/bench_v1.yaml
``` id="4fkv0t"

---

# 3. Parameter Integration Rules

## 3.1 Direct values
May be used as strong initial anchors when:
- the operating context is highly similar
- the units and measurement meaning are clear
- the source is trustworthy

Examples:
- pressure sensor range
- flow sensor range
- washer pump capacity bounds

---

## 3.2 Proxy values
May be used only if:
- they are clearly marked as proxy
- they have a validation hook
- they are bounded conservatively
- they are expected to be refined later

Examples:
- pleated filter loading threshold from adjacent filter literature
- pulse-recovery coefficients from non-laundry backpulse studies

---

## 3.3 Placeholder values
May exist only when:
- the parameter is required structurally
- no strong source exists yet
- the value is explicitly temporary

Placeholder values must always include:
- a note
- a confidence class
- a replacement condition

---

# 4. Safe Integration Order

HydrOS should integrate calibration values in this order:

1. Hydraulics
2. Clogging
3. Backflush
4. Particles
5. Sensors
6. Electrostatics

This order must align with the realism roadmap.

---

# 5. Validation Coupling Rule

No realism parameter should be treated as “integrated” unless it is tied to at least one of:

- a validation test
- a scenario check
- a sensor truth-vs-measurement check
- a bench experiment plan

If no such hook exists, the parameter is not yet operationally trustworthy.

---

# 6. What can be loaded now

The following categories are safe to initialize now using literature_v1:

## Safe now
- washer-side flow bounds
- nominal / peak device operating envelope
- stage areas
- maintenance threshold doctrine
- pulse morphology doctrine
- pressure sensor range / error
- flow sensor range / error
- laundry microfiber release ranges

These are strong enough to improve realism immediately.

---

# 7. What should remain provisional

The following should remain explicitly provisional:

- stage-specific fouling decomposition weights
- irreversible fouling rate
- recirculated-effluent cleaning efficiency
- microfiber-specific optical mapping
- electrostatic capture gain
- conductivity attenuation
- zeta-potential priors for actual laundry chemistry

These should not be framed as calibrated truth yet.

---

# 8. Update Process

When integrating or changing calibration values:

## Step 1
Consult:
- `CALIBRATION_PARAMETER_REGISTER.md`
- `literature_v1.yaml`

## Step 2
Identify:
- direct / proxy / placeholder
- confidence
- intended milestone

## Step 3
Implement only the smallest correct change

## Step 4
Re-run the relevant validation

## Step 5
Update the register if the parameter meaning, range, or source status changes

---

# 9. Bench Data Policy

Once HydrOS-specific bench data exists:

- bench-backed values should supersede proxy values
- previous proxy values should remain documented for traceability
- the register should explicitly record that replacement occurred

Do not silently replace proxy assumptions with bench values.

---

# 10. Documentation Coupling

Calibration integration must remain connected to:

- the calibration register
- YAML configuration
- validation files
- future research report appendices

HydrOS should always be able to answer:

> “Where did this number come from, how strong is the evidence, and when should it be replaced?”

---

# 11. Final Directive

Calibration integration is not parameter dumping.

It is the controlled translation of reality into simulation structure.

HydrOS must remain:
- source-aware
- evidence-aware
- milestone-aware
- validation-aware
